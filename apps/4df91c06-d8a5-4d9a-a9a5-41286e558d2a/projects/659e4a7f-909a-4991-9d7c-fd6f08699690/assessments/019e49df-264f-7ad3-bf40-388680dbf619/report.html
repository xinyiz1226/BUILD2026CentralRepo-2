Data Storage & External Services

The application uses a SQL Server database (ZavaBankDB) as its primary data store, accessed directly via ADO.NET without an ORM. Key tables include Transactions, Accounts, AlertRules, AccountAlerts, and AlertHistory. External messaging is handled through RabbitMQ's HTTP Management API (not the native AMQP client), publishing alert events to the zava.notifications exchange. Both database and RabbitMQ credentials are injected via environment variables (DB_HOST, DB_PORT, RABBITMQ_HOST, etc.) with fallback to web.config connection strings.

Key Architectural Decisions

  • WCF over BasicHttpBinding: Exposes all service operations as SOAP endpoints, making the service consumable by any SOAP-compatible client over plain HTTP without transport security.
  • RabbitMQ via HTTP Management API: Alert events are published through the RabbitMQ HTTP Management API rather than the AMQP protocol, avoiding a native client library dependency on Mono.
  • Environment-first configuration: DbConfig reads environment variables first and falls back to web.config connection strings, enabling container-friendly deployment.

Communication Patterns

Synchronous (Inbound): Clients communicate with the service via SOAP over HTTP using BasicHttpBinding. No transport security (TLS/HTTPS) is configured — all SOAP traffic is plain HTTP. No authentication or authorization is enforced at the WCF binding or service behavior level; all four operations are publicly accessible with no credential checks.

Asynchronous (Outbound): When a rule is triggered during EvaluateTransaction, RabbitMqPublisher posts an AlertEvent JSON payload to the RabbitMQ zava.notifications exchange using the HTTP Management API (POST /api/exchanges/{vhost}/zava.notifications/publish). This approach uses fire-and-forget semantics with a simple try/catch; there are no retry policies, delivery confirmations, publisher confirms, or circuit-breaker patterns. If the HTTP call fails for any reason, the alert is silently dropped (return false) and the failure is recorded only in the AlertHistory.NotificationSent column.

Service Discovery: No service discovery mechanism is present. The RabbitMQ host, port, and credentials are resolved from environment variables at runtime (RABBITMQ_HOST, RABBITMQ_MANAGEMENT_PORT, RABBITMQ_USER, RABBITMQ_PASSWORD, RABBITMQ_VHOST) with hard-coded defaults (rabbitmq, 15672, zava_app, zava_pass, /zavabank). The SQL Server connection is likewise resolved from environment variables or the web.config connection string.

Resilience: No retry, circuit-breaker, timeout, or bulkhead patterns are implemented for either SQL Server access or RabbitMQ publishing. All outbound HTTP calls to RabbitMQ use the default HttpWebRequest timeout (approximately 100 seconds).

Security Posture: No TLS, no authentication, and no authorization are configured. All WCF endpoints are publicly accessible over plain HTTP. The web.config contains a hard-coded SQL Server password (Zava123!) in the default connection string fallback.

The application has no built-in startup dependency management. It starts immediately on container launch and will fail at runtime (not startup) if SQL Server or RabbitMQ are unavailable:

Core Business Workflows

ZavaAlertService is a banking alert engine that evaluates financial transactions against customer-configured alert rules and dispatches notifications to a downstream messaging queue when those rules are triggered.

Primary Workflows

Workflow 1: Transaction Alert Evaluation

The core business workflow — triggered by a SOAP client (typically the core banking platform after posting a transaction):

  1. Receive transaction ID via EvaluateTransaction(transactionId) SOAP operation.
  2. Validate input: transactionId must be greater than 0; otherwise a SOAP fault is returned.
  3. Load transaction data: Query SQL Server for the transaction record, joining to the Accounts table to retrieve accountId, amount, balanceAfter, customerId, and channel.
  4. Load active alert rules: Retrieve all enabled AccountAlerts joined to their AlertRules for the account, applying per-account threshold overrides where set.
  5. Evaluate each rule using the rule code as a dispatch key (see Business Rules section).
  6. For each triggered rule:

a. Construct an AlertEvent payload.

b. Publish to RabbitMQ via HTTP Management API (fire-and-forget).

c. Insert an AlertHistory record with the trigger value, notification sent status, and a details string.

  1. Return AlertEvaluationResponse listing all triggered rule codes and whether any alert was published to the queue.

Workflow 2: Create Alert Rule

Triggered by a SOAP client managing account alert configuration:

  1. Validate input: RuleName, RuleCode, AccountId, and CustomerId are all required.
  2. Insert AlertRules record: Creates the rule definition with the provided name, code, category, threshold, and active status within a SQL transaction.
  3. Insert AccountAlerts record: Links the new rule to the specified account and customer, with the provided notification channel preference.
  4. Commit transaction: Both inserts succeed atomically or both are rolled back.
  5. Return AlertRuleMutationResponse with the new RuleId and AccountAlertId.

Workflow 3: Update Alert Rule

Triggered by a SOAP client modifying an existing alert rule:

  1. Validate input: ruleId must be greater than 0; RuleName and RuleCode are required.
  2. Update AlertRules record: Modifies rule metadata within a SQL transaction.
  3. Upsert AccountAlerts record (if AccountId > 0): Updates an existing account-rule assignment or inserts a new one if none exists.
  4. Commit transaction: Both operations succeed atomically.
  5. Return AlertRuleMutationResponse with the updated RuleId and AccountAlertId.

Workflow 4: Retrieve Alert Rules for an Account

A read-only workflow returning all configured rules for an account:

  1. Validate input: accountId must be greater than 0.
  2. Query AccountAlerts joined to AlertRules, applying per-account threshold overrides, ordered by rule name.
  3. Return AlertRulesResponse with the full list of AlertRuleInfo items.

Cross-Service Data Flows

ZavaAlertService is a single-service application with no service-to-service REST calls. Its cross-context data flows are:

Inbound (read): The service reads Transactions and Accounts data directly from the shared ZavaBankDB database — data owned by the core banking platform. There is no API call to a banking service; ownership boundaries are logical, not physical.

Outbound (async): When a rule fires, an AlertEvent JSON payload is published to the RabbitMQ zava.notifications exchange via the HTTP Management API. The notification service (a separate consumer, not part of this codebase) is responsible for reading from this exchange and dispatching the actual notification (email, SMS, push, etc.) to the customer. If the RabbitMQ publish call fails, the event is silently dropped — there is no retry, dead-letter handling, or compensating action. The AlertHistory.NotificationSent flag records whether the publish succeeded, but no re-delivery mechanism exists.

Business Rules & Decision Logic

Alert Rule Evaluation Codes

Five built-in rule codes are recognized. Unknown rule codes silently evaluate to false (not triggered):

Rule CodeTrigger ConditionNotes
LOW_BALbalanceAfter < thresholdFires when account balance falls below the configured threshold after the transaction
LARGE_TXNabs(amount) >= thresholdFires when the absolute transaction amount meets or exceeds the threshold
OVERDRAFTbalanceAfter < 0Fires when the account goes negative; threshold is ignored
INTL_WIREchannel contains "WIRE" (case-insensitive) AND abs(amount) >= thresholdFires on international wire transfers above a threshold
DAILY_SPENDSum of today's debit amounts for the account >= thresholdRequires an additional SQL query to compute the daily spend total

Threshold Resolution

When loading account-level rules, the threshold is resolved as: use the AccountAlerts.Threshold override if it is non-empty; otherwise fall back to AlertRules.DefaultThreshold. If neither parses as a valid decimal, 0 is used.

Transaction Boundaries

CreateAlertRule and UpdateAlertRule use explicit SqlTransaction objects to wrap their multi-statement DDL sequences. The EvaluateTransaction workflow does not use an explicit transaction — each INSERT AlertHistory statement is auto-committed independently, meaning a failure mid-loop could leave partial history records without rolling back already-published alert events.

Error Handling

All SOAP operations surface errors as WCF FaultException with a plain-text message. There are no typed fault contracts (FaultException<T>). The serviceDebug includeExceptionDetailInFaults="true" setting in web.config causes full .NET stack traces to be returned in SOAP fault responses, which is a security risk in production.

Audit Trail

Every triggered rule produces an AlertHistory record containing: the trigger value (amount), notification delivery status, and a human-readable Details string including rule code, account ID, balance after, and channel. This provides an immutable audit trail of all rule firings, but there is no separate structured event log or change-tracking mechanism.

Version & Compatibility Risks

All dependencies are built-in .NET Framework 4.8 BCL assemblies. .NET Framework 4.8 remains in a supported (maintenance) state but will not receive new features. System.ServiceModel (WCF) and System.Web (classic ASP.NET/ASMX) are not available in .NET Core or .NET 5+; migrating to modern .NET would require replacing WCF endpoints with gRPC or ASP.NET Core minimal APIs, and replacing System.Web HTTP handlers with ASP.NET Core middleware. The System.Data.SqlClient BCL package is superseded by the cross-platform Microsoft.Data.SqlClient NuGet package and would need to be replaced during any .NET modernization effort.

Notable Observations

  • Zero third-party NuGet packages: The project has an empty packages.config and no PackageReference items — all functionality is built from .NET Framework BCL assemblies only, which simplifies dependency management but limits access to modern libraries.
  • RabbitMQ via HTTP instead of AMQP client: Rather than using the official RabbitMQ .NET client NuGet package, the project uses raw HttpWebRequest calls to the RabbitMQ Management HTTP API. This is unconventional and loses reliability guarantees (delivery acknowledgements, publisher confirms) provided by the AMQP protocol.
  • WCF on Mono runtime: The Dockerfile uses the Mono 6.12 runtime with xsp4 to host this .NET Framework application on Linux. Mono's WCF support is partial and may cause runtime compatibility issues that would not appear on full .NET Framework on Windows.
  • ADO.NET without ORM: All database interactions use hand-crafted inline SQL strings with SqlCommand/SqlConnection. There is no Entity Framework or other ORM, making the data access layer more brittle and harder to migrate to modern persistence patterns.

Caching Strategy

No caching layer is present. All data is read directly from SQL Server on every request with no in-process or distributed cache. The DAILY_SPEND calculation in particular re-queries all of today's transactions on every evaluation, which could become a performance bottleneck under high transaction volume.

Data Ownership Boundaries

ZavaAlertService operates against a single shared ZavaBankDB SQL Server database. It has write ownership over AlertRules, AccountAlerts, and AlertHistory, while Transactions and Accounts are read-only from this service's perspective — they are expected to be populated by an upstream core banking service. There is no database-per-service isolation. All SQL is executed via direct connection from within the service process; there is no data API boundary between this service and the shared banking database.
