Data Storage & External Services

The service persists alert rules, account alert mappings, transaction lookups, and alert history in SQL Server (ZavaBankDB) using direct SQL commands. For downstream notification fan-out, it posts alert payloads to RabbitMQ's management publish endpoint.

Key Architectural Decisions

  • Uses a single service class (AlertService) as the orchestration point for read, evaluate, create, and update alert workflows.
  • Uses direct SQL (no ORM) to keep rule evaluation and alert history writes explicit and deterministic.
  • Uses RabbitMQ management API publishing instead of AMQP client libraries, reducing dependencies but coupling to management endpoint availability.

DTOs & Contracts

Contract classes are defined with WCF DataContract/DataMember and include request and response payloads used by service operations: AlertRuleDefinition (request contract for create/update), AlertRuleInfo and AlertRulesResponse (read contracts), AlertEvaluationResponse (evaluation result), and AlertRuleMutationResponse (mutation result). Internal event composition uses AlertEvent before publishing to RabbitMQ. DTOs are mutable C# classes rather than immutable record types. Serialization is handled through WCF contract serialization for service traffic and JavaScriptSerializer for outbound RabbitMQ publish payloads.

Communication Patterns

Communication is primarily synchronous. Client calls arrive over WCF basicHttpBinding, and AlertService executes synchronous SQL operations via SqlConnection/SqlCommand. When a rule triggers, the service performs a synchronous HTTP POST to RabbitMQ management API (/api/exchanges/.../publish) using Basic authentication. No explicit circuit breaker, retry, or timeout policy framework is configured; publisher failures are caught and converted to false while business flow continues. Service discovery and API gateway patterns are not present; dependencies are resolved through static host/environment configuration. Startup dependency behavior is implicit: SQL Server and RabbitMQ should be reachable for full functionality. API-level security posture is minimal: no application authentication or authorization checks are defined for service operations, and transport security is not enforced in code/config beyond deployment environment defaults.

Primary Workflows

Workflow 1: Evaluate Transaction and Emit Alerts

  1. Caller submits EvaluateTransaction(transactionId) through WCF endpoint.
  2. Service loads transaction/account context and active alert subscriptions for the account.
  3. For each rule, decision logic evaluates supported rule codes (LOW_BAL, LARGE_TXN, OVERDRAFT, INTL_WIRE, DAILY_SPEND).
  4. When a rule is triggered, service composes alert event payload and attempts notification publish to RabbitMQ.
  5. Service records each triggered rule in alert history, including whether notification publish succeeded.
  6. Service returns aggregated evaluation response with triggered rule codes and publish summary.

Workflow 2: Create Alert Rule for Account

  1. Caller submits CreateAlertRule with rule definition payload.
  2. Service validates required rule and account/customer context.
  3. Inside a SQL transaction, service creates a new reusable rule definition.
  4. Service then creates account-specific alert subscription referencing that rule.
  5. Service commits and returns success result including generated identifiers.

Workflow 3: Update Alert Rule and Account Binding

  1. Caller submits UpdateAlertRule(ruleId, rule).
  2. Service validates input and verifies target rule exists.
  3. Service updates base rule definition attributes.
  4. If account context is provided, it updates existing account subscription or inserts a new one.
  5. Service commits changes and returns mutation response.

Cross-Service Data Flows

The workflow is primarily single-service orchestration with two external integrations: SQL Server for stateful reads/writes and RabbitMQ for asynchronous notification fan-out. Transaction evaluation composes transaction, account, and alert-rule data in one service boundary, then emits outbound notification events when triggers fire. If RabbitMQ publish fails, the workflow degrades by returning a successful evaluation response while flagging publish status and persisting alert history for traceability.

Business Rules & Decision Logic

  • Transaction evaluation requires a positive transaction identifier and an existing transaction record.
  • Rule definitions require non-empty RuleName and RuleCode; create operations additionally require valid account and customer identifiers.
  • Rule-code decision logic:
    • LOW_BAL: triggers when post-transaction balance falls below configured threshold.
    • LARGE_TXN: triggers when absolute transaction amount meets/exceeds threshold.
    • OVERDRAFT: triggers when post-transaction balance is negative.
    • INTL_WIRE: triggers when channel indicates wire transfer and amount meets threshold.
    • DAILY_SPEND: triggers when same-day debit aggregate meets/exceeds threshold.
  • Rule create/update operations apply SQL transaction boundaries to keep rule and account-alert linkage consistent.
  • Notification failure does not abort evaluation; outcome is captured in response and persisted alert history.

Caching Strategy

No explicit caching provider, cache abstraction, or cache-aside/read-through strategy is configured in the project. All reads are served directly from SQL Server, and message publish responses are not cached.

Data Ownership Boundaries

The repository implements a single-service ownership model. Alert domain data is maintained directly by ZavaAlertService in shared SQL Server tables, with no separate service-local datastore boundaries or CQRS split. Cross-service data exchange is not implemented through service APIs; instead, notifications are emitted as outbound messages to RabbitMQ while authoritative alert state remains in SQL Server.
