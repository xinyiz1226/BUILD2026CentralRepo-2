Data Storage & External Services

ZavaNotifyWorker does not own a database. It relies on two external services: RabbitMQ acts as the message broker, providing the q.notify.email durable queue from which the worker polls for notification messages; undeliverable messages are routed to the zava.dlx dead-letter exchange. SMTP (defaulting to MailHog on port 1025 in development) is the email delivery backend — the worker creates a plain-text email per notification and submits it directly over SMTP.

Key Architectural Decisions

  • Polling over push: Uses synchronous BasicGet in a sleep-based loop rather than a BasicConsume push model, keeping the design simple at the cost of some latency.
  • Dual message format support: Accepts both JSON and XML message bodies, auto-detected at runtime by inspecting the first character of the payload.
  • Environment-variable-first configuration: Every setting is resolved first from an environment variable, then from app.config, then from a hardcoded default, enabling 12-factor-style deployment without changing the binary.

DTOs & Contracts

A single internal DTO is used to carry notification data between parsing and email dispatch:

ClassRoleImmutabilityNotes
NotificationMessageInternal message contract (parsed from queue payload)Mutable (public fields)Not serialized out; used only within the process

NotificationMessage carries: AlertType, TransactionId, Amount, Currency, CustomerId, CustomerName, CustomerEmail, Details. No OpenAPI, Swagger, or protobuf schemas are defined. Inbound messages are deserialized from either JSON (via JavaScriptSerializer) or XML (via XDocument), auto-detected by inspecting the first character of the raw payload.

Communication Patterns

Asynchronous (inbound): The worker polls RabbitMQ using BasicGet (synchronous pull, not push-based consume) on the q.notify.email durable queue at a configurable interval (default 5 000 ms). On successful processing the message is acknowledged (BasicAck); on failure it is negatively acknowledged (BasicNack with requeue=false), routing it to the zava.dlx dead-letter exchange.

Synchronous (outbound): Email delivery uses System.Net.Mail.SmtpClient with a blocking Send call. There is no retry policy, circuit breaker, timeout configuration, or resilience library (no Polly or similar). A failed SMTP send throws an exception that is caught by the polling loop, logs the error, and nacks the message to the DLX.

Service discovery: None. Both RabbitMQ and SMTP endpoints are resolved from environment variables or app.config key-value pairs at startup; no service registry or DNS-based discovery is used.

Security posture: No authentication or TLS is configured for either the outbound SMTP connection (UseDefaultCredentials = true, no SSL flag) or the RabbitMQ connection (plain username/password in configuration). No API-level authentication, authorization, or RBAC is present (as there is no HTTP surface).

Primary Workflows

Workflow 1: Customer Notification Delivery

ZavaNotifyWorker continuously polls the q.notify.email RabbitMQ queue. When a message arrives, it is parsed into a NotificationMessage, validated for a customer email address, formatted as a plain-text email, and sent over SMTP. Success results in an acknowledgment (removing the message from the queue); any failure results in a dead-letter routing.

Steps:

  1. Poll queue — BasicGet retrieves one message from q.notify.email per poll cycle.
  2. Detect format — Inspect the first character of the raw body: < indicates XML; anything else is treated as JSON.
  3. Parse message — Deserialize the body into a NotificationMessage. Multiple field name variants are tried for each property (e.g., alertType / AlertType / type) to accommodate upstream schema variations.
  4. Validate customer email — If CustomerEmail is absent or empty, throw InvalidOperationException; the message is nacked to the DLX.
  5. Compose email — Build subject ("Zava Bank {AlertType}: Transaction {TransactionId}") and plain-text body containing all notification fields.
  6. Send email — Dispatch via SmtpClient.Send to the configured SMTP server.
  7. Acknowledge or nack — On success, BasicAck; on any exception, BasicNack with requeue=false (routes to dead-letter exchange).

Cross-Service Data Flows

ZavaNotifyWorker receives notification payloads from upstream banking services via RabbitMQ — there is no direct service-to-service HTTP call. The entire integration is mediated by the message broker:

  • Upstream → Queue: A banking service (e.g., transaction processor) publishes a JSON or XML notification event to q.notify.email. The publishing service is the source of truth for all customer and transaction data embedded in the payload.
  • Queue → Worker: ZavaNotifyWorker pulls one message at a time. No aggregation across multiple messages or multiple upstream services occurs within a single processing cycle.
  • Worker → SMTP: The worker acts as a relay — it reformats the queue payload into an SMTP email without enriching it from any external source.

Fallback on failure: If email dispatch fails (SMTP unreachable, malformed address), the message is moved to zava.dlx. No retry at the worker level is attempted; the dead-letter consumer (outside this repository) is responsible for any reprocessing or alerting.

Business Rules & Decision Logic

Validation Rules:

  • CustomerEmail must be non-null and non-whitespace. If absent, the message is rejected to the dead-letter exchange with an InvalidOperationException.
  • The message body must be non-null and non-whitespace; an empty body is rejected as invalid.

Decision Logic:

  • Format detection: First character < → XML parser; otherwise → JSON parser. This allows upstream services to publish in either format without worker reconfiguration.
  • Field aliasing: Each NotificationMessage field accepts multiple source key names (e.g., alertType, AlertType, type; customerId, CustomerId, userId), providing backward-compatible deserialization across upstream schema versions.
  • Email subject composition: AlertType defaults to "Notification" if absent; TransactionId defaults to "N/A" if absent — the email is always sendable even with partial data.
  • Field defaults in body: Any missing field in the email body is rendered as "N/A" via ValueOrDefault().

State Transitions:

  • Message lifecycle: Delivered (in queue) → Processing → Acknowledged (success) or Dead-lettered (parse failure, missing email, SMTP failure).

Error Handling:

  • All exceptions within PollQueue() are caught at the top-level polling loop, logged to stdout, and do not crash the worker — the next poll cycle begins after Worker.PollIntervalMs sleep.
  • Failed messages are nacked with requeue=false, ensuring they are not re-delivered to the same queue and preventing infinite retry loops.

Cross-Cutting Concerns:

  • Transactions: No database transactions; delivery is effectively at-most-once (BasicGet + BasicAck; no two-phase commit with SMTP).
  • Audit/logging: All significant events are logged to stdout (Console.WriteLine) including delivery success with customer ID, email, alert type, transaction ID, and amount.
  • Authorization: None — the worker trusts all messages delivered by the queue; no message-level authentication or authorization check is performed.
  • Graceful shutdown: Console.CancelKeyPress and AppDomain.ProcessExit signal _keepRunning = false, allowing the current poll cycle to finish before exit.

Version & Compatibility Risks

RabbitMQ.Client 5.2.0 (released 2018) is significantly behind the current 6.x / 7.x line, which introduced async-first APIs, support for .NET Standard 2.0 / .NET 5+, and important security and stability fixes. Migrating to .NET (Core/5+) will require upgrading to RabbitMQ.Client 6.x or later, as 5.x targets .NET Framework only. System.Web.Script.Serialization (JavaScriptSerializer) is a legacy JSON API that ships only with the full .NET Framework and has no equivalent in .NET Core/5+ — it must be replaced (e.g., with System.Text.Json or Newtonsoft.Json) during any cross-platform migration.

Notable Observations

  • Minimal dependency footprint: Only one NuGet package is declared. All serialization, email, and configuration functionality is handled by BCL types, which keeps the project lean but ties it to the .NET Framework BCL surface.
  • Legacy JSON serializer: JavaScriptSerializer from System.Web.Extensions is not available on .NET Core/5+ and is considered obsolete even on .NET Framework; modern alternatives (System.Text.Json, Newtonsoft.Json) offer better performance and cross-platform support.
  • No logging framework: The project uses Console.WriteLine for all diagnostic output. There is no structured logging library (e.g., Serilog, NLog, Microsoft.Extensions.Logging), which limits observability in container/cloud environments.
  • No observability libraries: No OpenTelemetry, Application Insights, or metrics libraries are present, making it difficult to monitor queue-processing health in production.

Caching Strategy

No caching layer is present. Each message is fully re-parsed from the raw queue payload on every delivery. Configuration values (RabbitMQ and SMTP settings) are loaded once at process startup from environment variables or app.config and held in static fields for the lifetime of the process.

Data Ownership Boundaries

ZavaNotifyWorker owns no data. Notification payloads are produced and owned by upstream services (e.g., a banking transaction service) that publish to the q.notify.email queue. The worker is purely a consumer: it reads, transforms to an email, and discards the payload after acknowledgment. There is no shared database, no cross-service direct DB access, and no read/write separation to reason about.

Failed messages (parse errors, SMTP failures) are routed by RabbitMQ to the zava.dlx dead-letter exchange, where a separate consumer (not included in this repository) would be responsible for managing the dead-lettered payload.
