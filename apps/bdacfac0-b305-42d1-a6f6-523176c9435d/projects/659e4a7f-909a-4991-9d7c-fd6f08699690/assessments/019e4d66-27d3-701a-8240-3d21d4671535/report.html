Data Storage & External Services

The worker is stateless and does not persist to a database. It reads messages from RabbitMQ, keeps parsed payload data in memory for the processing cycle, sends email via SMTP, and relies on RabbitMQ dead-lettering for failed message handling.

Key Architectural Decisions

  • Pull-based queue processing is implemented with BasicGet in a timed loop instead of a push consumer callback.
  • Message compatibility supports both JSON and XML payload shapes to tolerate upstream producer variance.
  • Reliability behavior is explicit: successful processing leads to BasicAck, while failures lead to BasicNack without requeue to route to dead-letter exchange.

Core Business Workflows

This worker supports the notification domain by consuming transaction alert events and delivering customer-facing emails. Its business behavior centers on message normalization, delivery attempt handling, and dead-letter routing for failed notifications.

Primary Workflows

Workflow 1: Process Queue Notification and Send Email

  1. Worker polls RabbitMQ queue for one message.
  2. Worker validates payload presence and chooses parser based on format (JSON/XML).
  3. Worker maps source fields to NotificationMessage and validates customer email presence.
  4. Worker composes notification subject/body and sends email through SMTP.
  5. On success, worker acknowledges the queue message.

Business rule involved: notification delivery requires CustomerEmail; missing email causes processing failure.

Workflow 2: Failed Notification Handling

  1. Parsing or SMTP delivery throws an exception.
  2. Worker logs failure context.
  3. Worker rejects the message without requeue.
  4. Message is routed to dead-letter exchange for downstream triage.

Business rule involved: failed notifications are not retried inline; they are routed for later handling.

Cross-Service Data Flows

Cross-system flow is linear: RabbitMQ provides notification payload data, the worker transforms it, and SMTP receives finalized message content. There is no data join with other internal services in this repository. Fallback behavior for failures is dead-letter routing instead of downstream composition.

Business Workflow Sequence

sequenceDiagram participant Broker as "RabbitMQ" participant Worker as "Notify Worker" participant Parser as "Payload Parser" participant Mail as "SMTP Service" participant DLX as "Dead Letter Exchange" Broker-->>Worker: Notification event available Worker->>Parser: Normalize message to business notification Parser-->>Worker: NotificationMessage ready Worker->>Worker: Validate required email recipient Worker->>Mail: Deliver customer notification alt Delivery succeeded Worker-->>Broker: Acknowledge processed message else Validation or delivery failed Worker-->>DLX: Route message for later handling Note over Worker: Business fallback avoids inline retries end

Business Rules & Decision Logic

  • Validation rule: empty message body is invalid and fails processing.
  • Validation rule: CustomerEmail must be present before attempting notification delivery.
  • Decision logic: message format branch chooses XML parser for markup payloads and JSON parser otherwise.
  • Delivery decision: success path acknowledges message; failure path dead-letters message with no requeue.
  • Cross-cutting: exceptions are logged per poll cycle, and worker continues running for subsequent messages.
