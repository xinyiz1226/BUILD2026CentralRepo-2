Data Storage & External Services

The application does not use a persistent database. File system directories serve as transient storage: the drop directory receives input files (CSV, TXT, DAT), a processed sub-directory stores successfully published files, and an error sub-directory stores files that failed processing. All parsed record payloads are published to a RabbitMQ broker — a Topic Exchange (filedrop.events) for normal records and a Fanout Dead Letter Exchange (zava.dlx) for files that could not be processed. Connection details (host, port, virtual host, credentials, exchange names) are read from app.config and can be overridden via environment variables at runtime.

Key Architectural Decisions

  • Dual detection strategy: Combines FileSystemWatcher events with a periodic poll loop so files that arrive between event callbacks are never missed; a HashSet<string> with a lock de-duplicates paths before processing.
  • Per-batch RabbitMQ connection: A single IConnection and IModel are opened per poll cycle rather than kept open continuously, simplifying error recovery at the cost of connection overhead.
  • Extension-based routing: The AMQP routing key is derived from the file extension (filedrop.csv, filedrop.txt, filedrop.dat), allowing downstream consumers to subscribe selectively by file type.

DTOs & Contracts

The application produces two JSON message shapes, constructed inline using System.Web.Script.Serialization.JavaScriptSerializer (no formal DTO classes are declared):

File Record Message — published to the filedrop.events Topic Exchange:


{
  "sourceFile": "<filename>",
  "processedAt": "<yyyy-MM-ddTHH:mm:ss>",
  "record": { /* key-value pairs from CSV headers or fixed-width field names */ }
}

The record payload schema varies by file extension:

  • CSV files: keys are taken from the first header row; values are the corresponding column values.
  • Fixed-width files (.txt / .dat): keys are RecordType, CustomerId, AccountId, Amount, Details (fixed positions in an 80-character line layout).

Dead Letter Message — published to the zava.dlx Fanout Exchange on processing failure:


{
  "sourceFile": "<filename>",
  "error": "<exception message>",
  "failedAt": "<yyyy-MM-ddTHH:mm:ss>"
}

No OpenAPI/Swagger specifications, protobuf schemas, GraphQL schemas, or formal DTO class definitions exist in the codebase. All message structures are implicitly defined by dictionary construction in BuildPayload and PublishDeadLetter.

Communication Patterns

Asynchronous (outbound only): ZavaQueueBridge publishes messages to RabbitMQ using the AMQP 0-9-1 protocol via RabbitMQ.Client 5.2.0. Two exchanges are used:

  • filedrop.events — Topic Exchange; routing key is derived from file extension (e.g. filedrop.csv, filedrop.txt, filedrop.dat). Downstream consumers subscribe to specific routing key patterns to receive only the file types they care about.
  • zava.dlx — Fanout Exchange; receives error payloads for any file that could not be processed.

All messages are marked Persistent = true and ContentType = application/json.

No synchronous inter-service communication: The application makes no HTTP calls, gRPC calls, or other synchronous remote requests.

Resilience policies: No circuit breaker, retry, or timeout policy is implemented for the RabbitMQ connection (AutomaticRecoveryEnabled = false). A transient connection failure during a poll cycle will cause that cycle to fail entirely; the files remain in the pending set and will be retried on the next poll interval. A file-level retry is implemented only for file readiness (up to 5 attempts with 250 ms sleep to wait for the file to be fully written).

Service discovery: None. The RabbitMQ host, port, and virtual host are read directly from configuration (environment variables or app.config). No dynamic service registry is used.

Security posture: No TLS/SSL is configured for the AMQP connection (plain TCP on port 5672). Credentials are passed via UserName/Password on the ConnectionFactory. No token-based authentication, certificate authentication, or encrypted transport is implemented. If the application is deployed in a network where the RabbitMQ broker is not isolated, credentials and message payloads are transmitted in clear text.

Primary Workflows

Workflow 1: File Detection

Entry points: FileSystemWatcher events (Created, Changed, Renamed) and the periodic poll loop (every 5 000 ms by default).

  1. FileSystemWatcher raises an event when a file is created, modified, or renamed in the drop directory.
  2. The event handler calls QueuePendingFile, which checks the file extension — only .csv, .txt, and .dat files are accepted; directories named processed or error are explicitly excluded.
  3. The accepted file path is added to an in-memory HashSet<string> (thread-safe via a lock) to await the next poll cycle.
  4. In parallel, the poll loop runs Directory.GetFiles on the drop directory at each interval, adding any files not already in the pending set. This ensures files that arrive between watcher events are never missed.

Business rule: Only files with extensions .csv, .txt, or .dat are eligible for processing.

Workflow 2: File Processing & Message Publishing

Entry point: Poll cycle drains the pending file set and opens a RabbitMQ connection.

  1. For each pending file, WaitForFileReady attempts to open the file up to 5 times (250 ms apart) to confirm the upstream writer has finished.
  2. ParseRecords dispatches to the appropriate parser based on file extension:
    • CSV: The first line is treated as a header row; subsequent non-empty lines are split into key-value dictionaries by SplitCsvLine (quote-aware tokeniser). Empty lines are skipped.
    • Fixed-width (.txt / .dat): Each non-empty line is padded to 80 characters and sliced into five named fields: RecordType (0–9), CustomerId (10–29), AccountId (30–49), Amount (50–64), Details (65+).
  1. For every parsed record, BuildPayload constructs a JSON envelope with sourceFile, processedAt (ISO-8601 timestamp), and the record dictionary.
  2. BuildRoutingKey derives the AMQP routing key from the file extension (e.g., filedrop.csv), allowing downstream consumers to subscribe selectively.
  3. The message is published to the filedrop.events Topic Exchange as a persistent, application/json message.
  4. After all records in the file are published, the file is moved to the /processed archive directory (with a timestamp-prefixed name if a collision occurs).

Workflow 3: Error Handling & Dead-Lettering

Trigger: Any exception thrown during ProcessSingleFile (file lock failure, parse error, RabbitMQ publish error).

  1. The exception is caught in the poll loop.
  2. PublishDeadLetter builds an error envelope (sourceFile, error, failedAt) and publishes it to the zava.dlx Fanout Exchange so all bound dead-letter queues receive the failure notification.
  3. The failed file is moved to the /error archive directory.
  4. Processing continues with the remaining files in the current batch.

Cross-Service Data Flows

ZavaQueueBridge is a single-service pipeline with no aggregation of data from multiple backend services. The data flow is strictly unidirectional:


Upstream producer → File System (drop directory) → ZavaQueueBridge → RabbitMQ (filedrop.events)
                                                                     ↘ RabbitMQ (zava.dlx) on error

Downstream consumers of filedrop.events are fully decoupled — they receive messages asynchronously and are not part of this service. There is no request/response pattern, no circuit breaker between ZavaQueueBridge and its consumers, and no fallback that merges data from multiple sources.

The only cross-context dependency is the implicit contract between the upstream file producer and this service: the producer must write files with the correct extension and a schema compatible with the CSV or fixed-width parsers.

Business Constraints

  • Exchange durability: Both filedrop.events (Topic) and zava.dlx (Fanout) are declared durable (durable=true); messages are marked Persistent=true. This ensures messages survive a RabbitMQ broker restart.
  • No deduplication guarantee: If ZavaQueueBridge restarts between publishing a record and moving the file, the same file may be re-processed and its records published again (at-least-once delivery semantics).
  • No transactional publishing: There is no AMQP transaction or publisher confirm; a network failure after BasicPublish but before the broker acknowledges can result in lost messages (fire-and-forget within the connection lifetime).

Version & Compatibility Risks

.NET Framework 4.8 is Microsoft's final .NET Framework release and is in long-term servicing mode with no new feature development; it cannot run cross-platform. RabbitMQ.Client 5.2.0 targets the legacy net451 TFM and is several major versions behind the current 6.x/7.x line, which targets .NET Standard 2.0+ and modern .NET. The 5.x series no longer receives security patches from the maintainers. System.Web.Extensions.JavaScriptSerializer is a legacy, feature-limited JSON serializer that has been superseded by System.Text.Json (built into modern .NET) and is not available on .NET 5+, making it a direct migration blocker.

Notable Observations

  • Single external dependency: The minimal dependency footprint reduces migration complexity, but the one external library (RabbitMQ.Client 5.2.0) requires a major version upgrade (to 6.x or 7.x) when targeting modern .NET.
  • JavaScriptSerializer is a migration blocker: System.Web.Extensions (which contains JavaScriptSerializer) does not exist on .NET 5+; all usages must be replaced with System.Text.Json.JsonSerializer or Newtonsoft.Json before the project can target modern .NET.
  • No logging framework: The application uses raw Console.WriteLine throughout. Adopting a structured logging library (e.g., Serilog, Microsoft.Extensions.Logging) would be required for production-grade cloud deployments.
  • No dependency injection or configuration abstractions: Configuration is read directly via ConfigurationManager; migrating to Microsoft.Extensions.Configuration and IOptions<T> would align the project with modern .NET conventions and improve testability.

Caching Strategy

No caching layer is implemented. The only in-memory state is a HashSet<string> (PendingFiles) that de-duplicates file paths between the FileSystemWatcher event callback and the poll cycle. This set is drained on every poll cycle and provides no persistence across process restarts.

Data Ownership Boundaries

ZavaQueueBridge is a single-process, single-service application with no shared data store and no inter-service data access. All data it owns is confined to the local file system mount:

  • Drop directory — input: files written here by an external upstream producer (not part of this codebase)
  • Processed directory — output: successfully published files archived here
  • Error directory — output: files that could not be processed archived here

The application is a unidirectional pipeline: it reads from the file system and writes to RabbitMQ. It does not read from RabbitMQ or write back to any upstream system. Downstream consumers of the filedrop.events exchange are entirely separate services not included in this repository.
