Data Storage & External Services

The application does not use a relational database or cache. Its primary external integration is RabbitMQ, reached via AMQP on a configurable host and port (default: rabbitmq:5672). It publishes successfully parsed file records to a Topic Exchange (filedrop.events) using a routing key derived from the file extension (e.g., filedrop.csv). Failed records are forwarded to a Fanout Dead-Letter Exchange (zava.dlx) and the source files are moved to an error directory. Processed files are moved to a separate processed/ subdirectory for auditability.

Key Architectural Decisions

  • Hybrid event + polling model: A FileSystemWatcher reacts to file system events while a configurable polling loop (Worker.PollIntervalMs, default 5 s) catches any files missed by the watcher, ensuring no data loss under high load or race conditions.
  • File extension–based routing: The AMQP routing key is built from the file extension (filedrop.<ext>), allowing downstream consumers to subscribe selectively to specific data formats via RabbitMQ topic bindings.
  • Configuration layering: Every setting can be overridden by an environment variable (e.g., RABBITMQ_HOST) before falling back to app.config and finally to a hard-coded default, enabling twelve-factor app configuration without code changes.

DTOs & Contracts

Published message envelope (JSON, constructed in BuildPayload via JavaScriptSerializer):


{
  "sourceFile": "<filename>",
  "processedAt": "yyyy-MM-ddTHH:mm:ss",
  "record": { "<field>": "<value>", ... }
}

The record object is a flat string-to-string dictionary whose keys are driven by the file format:

  • CSV files: Keys are the header row column names (dynamic, file-dependent).
  • TXT/DAT files (fixed-width): Keys are always RecordType, CustomerId, AccountId, Amount, Details.

Dead-letter message envelope (published to zava.dlx on processing failure):


{
  "sourceFile": "<filename>",
  "error": "<exception message>",
  "failedAt": "yyyy-MM-ddTHH:mm:ss"
}

Neither envelope uses a formal schema definition (no OpenAPI, Protobuf, or Avro). Both are plain JSON objects serialized by System.Web.Script.Serialization.JavaScriptSerializer. Message properties on both envelopes: persistent=true, content-type=application/json.

Communication Patterns

Asynchronous (outbound — AMQP):

  • ZavaQueueBridge connects to RabbitMQ over AMQP 0-9-1 using RabbitMQ.Client 5.2.0.
  • A new IConnection + IModel pair is opened per poll cycle and closed on completion (no connection pooling).
  • Messages are published to filedrop.events (Topic Exchange, durable) with a routing key of filedrop.<ext> (e.g., filedrop.csv, filedrop.txt, filedrop.dat).
  • Failed records are published to zava.dlx (Fanout Exchange, durable) with an empty routing key.
  • Both exchanges are declared idempotently at publish time (channel.ExchangeDeclare).

Inbound (file system):

  • A FileSystemWatcher reacts to file create/change/rename events in the drop path.
  • A polling loop (default 5-second interval) supplements the watcher by scanning the drop directory directly, guarding against missed events.
  • File readiness is verified by attempting to open the file for shared read access, retrying up to 5 times with 250 ms delays.

Resilience:

  • AutomaticRecoveryEnabled = false — no automatic AMQP connection recovery; a poll-cycle failure is caught, logged to stdout, and the loop continues after the configured sleep interval.
  • No circuit breaker, retry policy (Polly or equivalent), or timeout is configured for the AMQP publish path.
  • File-level errors are caught individually; a failed file is dead-lettered and moved to error/ without stopping processing of remaining files in the same batch.

Service discovery: None. RabbitMQ is addressed by a single configurable hostname (RABBITMQ_HOST / RabbitMQ.Host). There is no Consul, Eureka, or Kubernetes DNS-based discovery.

Security posture: No TLS/AMQPS is configured — the connection uses plain AMQP on port 5672 with username/password credentials (zava_app / zava_pass defaults). Credentials are stored in app.config and can be overridden by environment variables. No certificate validation, no mutual TLS, and no token-based authentication are present. All communication between ZavaQueueBridge and RabbitMQ is unencrypted in the default configuration.

Startup Dependency Chain

ZavaQueueBridge has no formal startup dependency declaration (no Docker Compose depends_on, no Kubernetes readiness probe, no dockerize wait-for-TCP wrapper).

Effective startup order:

  1. File-drop directories created — EnsureDirectories() runs first; creates _dropPath, _processedPath, and _errorPath if they do not exist.
  2. FileSystemWatcher started — begins reacting to file events immediately after directory creation.
  3. RabbitMQ connection — attempted on the first poll cycle (after Worker.PollIntervalMs ms); if the broker is unavailable, the poll-cycle exception is caught and logged but the process continues — meaning messages can be lost silently until the broker becomes reachable.

Risk: There is no startup health gate for RabbitMQ availability. If the broker is not ready when the first poll cycle fires and files are already present in the drop directory, those files will be dead-lettered or moved to error/ before the broker is reachable.

Primary Workflows

Workflow 1: File Detection

The bridge detects new files via two complementary mechanisms:

  1. A FileSystemWatcher raises an event immediately when a file is created, modified, or renamed in the drop directory.
  2. A background polling loop scans the drop directory every Worker.PollIntervalMs milliseconds (default 5 s), catching any files that were missed by the watcher (e.g., files already present at startup, or events dropped under high load).
  3. Each detected file path is checked by the candidate filter (IsCandidateFile): only .csv, .txt, and .dat extensions are accepted; paths inside processed/ or error/ subdirectories are rejected.
  4. Accepted paths are added (de-duplicated) to the in-memory pending set protected by a mutex lock.

Workflow 2: File Processing and Record Publishing

For each file in the pending batch:

  1. File readiness check — the bridge attempts to acquire a shared read lock on the file, retrying up to 5 times with 250 ms delays. This guards against files that are still being written by the upstream producer.
  2. Format detection — the file extension determines the parsing strategy: .csv → header-driven CSV parser; .txt/.dat → fixed-width field parser.
  3. Record parsing — each data row is parsed into a flat string dictionary. CSV records use column headers as keys; fixed-width records use fixed field names (RecordType, CustomerId, AccountId, Amount, Details).
  4. Message construction — each record is wrapped in a JSON envelope with the originating filename and a processing timestamp.
  5. Routing key assignment — the AMQP routing key is derived from the file extension (e.g., filedrop.csv), enabling downstream consumers to subscribe selectively.
  6. Durable publish — each record is published to the filedrop.events topic exchange with persistent=true, ensuring broker-side durability across RabbitMQ restarts.
  7. File archival — on success, the source file is moved atomically to the processed/ subdirectory.

Workflow 3: Error Handling and Dead-Lettering

If any step in Workflow 2 raises an exception for a given file:

  1. An error envelope (source filename, exception message, failure timestamp) is published to the zava.dlx fanout exchange, notifying all dead-letter consumers.
  2. The source file is moved to the error/ subdirectory for manual investigation.
  3. Processing continues with the remaining files in the current batch — a single file failure does not abort the batch.
  4. Poll-cycle–level exceptions (e.g., broker connection failure) are caught and logged to stdout; the bridge sleeps for the configured interval and retries on the next cycle.

Cross-Service Data Flows

ZavaQueueBridge is a single-service application; there are no peer services to aggregate data from. Data flows are strictly linear:


Upstream Producer → File System → ZavaQueueBridge → filedrop.events → Downstream Consumers
                                                  ↘ zava.dlx       → Error Consumers

Upstream boundary: Upstream producers (core banking batch jobs or SFTP landing zones) write files to the shared volume. ZavaQueueBridge has no knowledge of the upstream system's identity or schedule.

Downstream boundary: Downstream consumers bind queues to the filedrop.events exchange using topic routing keys (e.g., filedrop.csv, filedrop.#). The bridge has no knowledge of downstream consumer identities, queue names, or message-acknowledgement outcomes. Message delivery guarantees are delegated entirely to RabbitMQ.

Fallback behavior: If the RabbitMQ broker is unavailable during a poll cycle, the entire cycle fails silently (exception logged to stdout). Files that were scanned in that cycle are not re-queued to the pending set for the next cycle — they will be rediscovered by the next poll-cycle directory scan. This creates a window of up to Worker.PollIntervalMs during which files may be delayed but not lost.

Business Rules & Decision Logic

File Acceptance Rules

  • Extension filter: Only files with extensions .csv, .txt, or .dat are processed. All other file types are silently ignored.
  • Directory exclusion: Files located under the processed/ or error/ subdirectories are never re-processed, even if the FileSystemWatcher raises events for them.
  • File readiness gate: A file must be openable for shared reading within 5 attempts (250 ms apart). Files that cannot be locked are skipped for the current cycle and reconsidered on the next poll — preventing partial-write ingestion.

Parsing Rules

  • CSV: First line is treated as the header; subsequent non-blank lines are data rows. Values are comma-delimited with basic double-quote handling (quotes are stripped; commas inside quotes are preserved). Values are trimmed of leading/trailing whitespace.
  • Fixed-width (TXT/DAT): Lines are padded to 80 characters. Fields are extracted by fixed character offsets: RecordType (0–9), CustomerId (10–29), AccountId (30–49), Amount (50–64), Details (65–79). Empty files raise an error.
  • Format selection: Determined exclusively by file extension (.csv → CSV parser; all others → fixed-width parser).

Routing Rules

  • The AMQP routing key is filedrop.<ext> where <ext> is the lowercase file extension without the leading dot (e.g., filedrop.csv, filedrop.txt). If the extension is absent or blank, the routing key defaults to filedrop.unknown.

Error Handling & Dead-Lettering

  • Per-file isolation: Errors on one file do not abort processing of other files in the same batch.
  • Dead-letter contract: Every failed file produces exactly one dead-letter message containing sourceFile, error (exception message), and failedAt timestamp. No retry logic is applied to failed files — re-processing requires manual intervention (moving the file back to the drop directory).
  • Poll-cycle failures: Broker connection failures cause the entire poll cycle to fail; all pending files in that batch are skipped but remain discoverable by the next scan.

Audit Trail

  • File system: processed/ and error/ directories serve as an implicit audit trail. Files moved there are renamed with a timestamp prefix if a name collision exists (e.g., 20260521100508_data.csv).
  • Stdout logging: Key events (startup, file published, file moved, errors) are logged to stdout with a [QueueBridge] prefix. No structured logging or audit database is used.

Authorization

No authentication or authorization controls exist. Any process or user with write access to the file-drop directory can introduce files that will be parsed and published to RabbitMQ without any validation of origin or content integrity (e.g., no checksum or signature verification).

Version & Compatibility Risks

The sole external dependency, RabbitMQ.Client 5.2.0 (released 2018), is significantly outdated. The current stable series is 7.x, which targets .NET 6+ and introduces async-first APIs, connection lifecycle improvements, and breaking changes to the IModel/IChannel surface. Migrating to the modern client requires updating connection-management code and adopting IChannel in place of the deprecated IModel. The application itself targets .NET Framework 4.8, which is in long-term maintenance mode with no new feature development; hosting on Azure services (App Service, ACA, AKS) strongly favors a migration to .NET 8 or later. Additionally, System.Web.Extensions (used solely for JavaScriptSerializer) is a legacy .NET Framework assembly not available on .NET Core/5+; it should be replaced with System.Text.Json or Newtonsoft.Json as part of any framework upgrade.

Notable Observations

  • Single external dependency: The project has an exceptionally lean dependency footprint — only RabbitMQ.Client comes from NuGet. This minimises upgrade complexity but means all JSON serialization, HTTP, and configuration utilities are drawn from the aging .NET Framework BCL.
  • System.Web.Extensions coupling: Using JavaScriptSerializer (from the ASP.NET Web Forms era) for JSON payload construction creates a hard tie to .NET Framework. It must be replaced before any cross-platform migration.
  • No logging framework: The application uses Console.WriteLine exclusively. There is no structured logging library (e.g., Serilog, NLog, Microsoft.Extensions.Logging), making distributed tracing and log aggregation on Azure difficult.
  • No resilience library: RabbitMQ connections are created per-poll-cycle with AutomaticRecoveryEnabled = false and no retry policy. A resilience library such as Polly would reduce data-loss risk under transient broker failures.

ZavaQueueBridge has no relational database or ORM layer; its persistence contract consists of the local file system (input and audit trail) and the AMQP message envelopes it publishes to RabbitMQ.

Caching Strategy

No caching layer is implemented. There is no Redis, EhCache, MemoryCache, or equivalent. The in-memory HashSet<string> PendingFiles acts as a deduplication buffer across a single poll cycle but is not a persistent or distributed cache — it is cleared after each batch.

Data Ownership Boundaries

ZavaQueueBridge is a single-service application with no peer services sharing its data stores.

File system: The drop directory (/shared/filedrop) is the sole inbound data boundary. The application has read/write ownership of the drop path and exclusive ownership of the processed/ and error/ subdirectories. In a production deployment, the drop path would typically be a shared volume mounted from an upstream file producer (e.g., an SFTP server, a batch process). There is no coordination mechanism (advisory locks, distributed semaphore) to prevent concurrent instances from processing the same file.

Message broker: ZavaQueueBridge is a producer-only participant on the RabbitMQ broker. It declares and publishes to filedrop.events and zava.dlx but does not consume from any queue. Downstream consumers (external to this repository) own the queue-binding configuration and message-acknowledgement lifecycle.

Read/write pattern: The application is write-only with respect to the message broker and read-then-move with respect to the file system. There is no CQRS separation or event-sourcing pattern.

Assessment: Because the bridge is positioned in a banking context (ZavaBank namespace, /zavabank vhost), the files it processes are likely to contain PII or PCI data. No encryption-at-rest is configured for the file-drop directories, no TLS is configured for the AMQP connection, and credentials are stored in plaintext configuration. These represent significant security gaps that must be addressed before any production or cloud deployment.
