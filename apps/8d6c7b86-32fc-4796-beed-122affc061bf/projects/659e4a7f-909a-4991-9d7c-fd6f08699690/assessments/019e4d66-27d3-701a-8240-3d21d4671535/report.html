Data Storage & External Services

The application connects to a Microsoft SQL Server instance (ZavaBankCore database) over JDBC to persist ingested records into three staging tables: StagingCheckImage, StagingWireConfirmation, and StagingRegulatoryFeed. After each file is processed, an event is published to a RabbitMQ topic exchange (zava.events) using the routing key file.ingested, enabling downstream consumers to react to ingestion outcomes. The shared file system is accessed directly via the Java NIO API from a mounted volume path (/shared/file-ingestion).

Key Architectural Decisions

  • Poll-based ingestion: A blocking while-loop polls a watch directory at a configurable interval (default 5 s), providing a simple and dependency-light scheduling mechanism without a task framework.
  • File-name-driven routing: File type classification is driven entirely by filename patterns (prefix keywords and extensions), avoiding the need for a manifest or metadata sidecar file.
  • Environment-variable override pattern: All connection properties can be overridden at runtime via environment variables (e.g., SQLSERVER_URL, RABBITMQ_HOST), making the service container-friendly.

DTOs & Contracts

There are no DTO or request/response model classes. The application's "data contracts" are implicit:

  • Inbound file contracts (undocumented, convention-based):
    • Check image files (check.csv): single-line CSV — checkNumber, accountNumber, imagePath
    • Wire confirmation files (wire.txt): fixed-width records — positions 0–15 wireReference, 16–27 amount, 28–67 beneficiary
    • Regulatory feed files (reg.csv / feed.csv): multi-line CSV — feedCode, recordDate, payload
  • Outbound event payload (ad-hoc JSON string, not backed by a typed class):

```json

{ "eventType": "file.ingestion", "fileName": "...", "fileType": "...", "status": "processed|failed", "ingestedAt": "<epoch-ms>" }

```

  • No OpenAPI/Swagger specification, no protobuf schemas, and no GraphQL schema exist.
  • Serialization is handled by manual string concatenation for the JSON event payload.

Communication Patterns

Synchronous — JDBC (SQL Server):

Each file record is persisted via a direct JDBC DriverManager.getConnection(...) call inside executeInsert. A new connection is opened and closed for every individual INSERT statement. There is no connection pool, no transaction manager, and no retry policy. If the INSERT fails, the error is logged and the file is moved to the error directory.

Asynchronous — AMQP (RabbitMQ):

After each file is processed (success or failure), publishIngestionEvent opens a new RabbitMQ Connection and Channel, declares the zava.events topic exchange as durable, publishes the JSON payload with routing key file.ingested, and immediately closes both resources. There is no connection pool, no retry policy, and no dead-letter queue. A publish failure is caught, logged to stdout, and silently swallowed — the file move proceeds regardless.

Resilience patterns: None configured. There are no circuit breakers, no retry loops, no timeout settings for either the JDBC or AMQP calls, and no bulkhead patterns.

Service discovery: None. All connection parameters (SQL Server URL, RabbitMQ host/port) are resolved from static properties or environment variables at startup. There is no dynamic registration or lookup.

API gateway: Not applicable — no HTTP surface is exposed.

Security posture: No authentication, TLS, or authorization is configured at any layer. Database credentials and RabbitMQ passwords are stored in plain text in fileingestion.properties. Environment-variable overrides are available but no secrets management integration (e.g., Azure Key Vault) is present.

Core Business Workflows

ZavaFileIngestion automates the ingestion of bank operational files — check image metadata, wire confirmations, and regulatory feeds — from a shared filesystem into a SQL Server staging layer, publishing status events after each file is handled.

This is a single-service application. All bounded contexts are implemented within the one Java process — there are no separate microservices, modules, or downstream internal services.

Primary Workflows

Workflow 1: Scheduled File Polling and Dispatch

The application continuously polls the watch directory on a configurable interval (default 5 seconds). On each cycle it lists all files present in the directory, skips any subdirectories and files whose names contain the words "processed" or "error" (to avoid reprocessing moved files), identifies the business file type for each remaining file by name, and dispatches each file to the appropriate processor. Files with names that do not match any known type keyword are silently skipped.

Steps:

  1. List all entries in the watch directory (/shared/file-ingestion).
  2. Skip directories and non-file entries.
  3. For each file, invoke the file-type classifier using the filename.
  4. If the file type is not recognised, skip the file (no error, no move).
  5. Dispatch the file to the matching processor (check images → CSV parser; wire confirmations → fixed-width parser; regulatory feeds → CSV parser).
  6. On success: publish a processed event and move the file to the processed directory.
  7. On failure: publish a failed event and move the file to the error directory.
  8. Sleep for the configured interval and repeat.

Workflow 2: Check Image File Ingestion

Processes a CSV file containing check image metadata. The file is expected to have a single data line; only the first line is read. The three fields (check number, account number, image path) are split by comma. A single INSERT is executed for the record. Empty or missing fields default to empty strings.

Steps:

  1. Open the file and read the first line.
  2. Split by comma to extract checkNumber, accountNumber, imagePath.
  3. Insert one row into StagingCheckImage with IngestedAt = GETDATE().
  4. Return true (success) or false (insert failure or parse exception).

Workflow 3: Wire Confirmation File Ingestion

Processes a fixed-width text file containing wire transfer records. All non-blank lines are processed. Each line is sliced at fixed character positions: positions 0–15 for wire reference, 16–27 for amount, 28–67 for beneficiary. A separate INSERT is executed for each line.

Steps:

  1. Open the file and iterate over all non-blank lines.
  2. Extract wireReference (chars 0–15), amount (chars 16–27), beneficiary (chars 28–67) via fixed-width slicing.
  3. Insert one row per line into StagingWireConfirmation.
  4. Return true if at least one row was inserted successfully.

Workflow 4: Regulatory Feed File Ingestion

Processes a CSV file containing regulatory data records. All non-blank lines are processed. Each line is split by comma to yield feedCode, recordDate, and payload. A separate INSERT is executed per line.

Steps:

  1. Open the file and iterate over all non-blank lines.
  2. Split each line by comma to extract feedCode, recordDate, payload.
  3. Insert one row per line into StagingRegulatoryFeed.
  4. Return true if at least one row was inserted successfully.

Workflow 5: Ingestion Event Publishing

After every file processing attempt — regardless of outcome — a JSON event is published to the RabbitMQ topic exchange. The event carries the file name, file type, processing status (processed or failed), and the current epoch timestamp. A failure to publish the event is caught and logged but does not alter the file's disposition (the file is still moved to the processed or error directory).

Cross-Service Data Flows

ZavaFileIngestion is the sole service; there are no other internal services and no API gateway. Cross-service data flow is limited to:

  • Inbound: files are placed onto the shared file system by an upstream system (not modelled in this codebase).
  • Outbound to SQL Server: staging records are inserted synchronously; downstream reporting or processing services are expected to query the staging tables independently.
  • Outbound to RabbitMQ: ingestion events are published asynchronously; downstream consumers (unknown from this codebase) subscribe via the zava.events topic exchange with the routing key file.ingested.

No API gateway aggregation, circuit breaker fallback, or cross-service read pattern exists within this codebase.

Business Rules & Decision Logic

File Type Classification Rules

RuleLogic
Skip already-moved filesFilenames containing the substrings processed or error are excluded from processing to prevent double-ingestion
Check image identificationFilename ends with .csv AND contains check
Wire confirmation identificationFilename ends with .txt AND contains wire
Regulatory feed identificationFilename ends with .csv or .txt AND contains reg or feed
Fallback keyword matchIf none of the above patterns match, the classifier tests for the keywords check, wire, or reg in the filename (in that priority order) and maps to the associated type
Unrecognised fileIf no keyword matches, the file is silently skipped — no error is logged and the file is left in the watch directory

Validation Rules

  • No schema validation: file contents are not validated against a schema. Malformed lines result in empty string values being inserted (e.g., a missing CSV field yields "").
  • Empty file handling: a check image file with no first line returns false immediately (no insert). Wire confirmation and regulatory feed processors skip blank lines silently.
  • Fixed-width boundary safety: the wire confirmation parser uses safe substring slicing — if a line is shorter than the expected width, the short segment is returned rather than throwing an exception.

State Transitions

Files follow a simple two-state lifecycle:


[Pending in watch dir] → [Processing] → [Processed dir] (success)
                                      → [Error dir]     (failure)

There is no retry state: a file that moves to the error directory is not automatically reprocessed.

Error Handling & Business Constraints

  • Silent event-publish failures: a RabbitMQ publish error is logged to stdout and swallowed. Downstream consumers may miss the event without any compensating action.
  • No connection pooling: every INSERT and every event publish opens and closes a new connection. Under high file volume this can exhaust SQL Server or RabbitMQ connection limits.
  • No deduplication: if a file with the same name re-appears in the watch directory (e.g., after a manual reset), it will be processed and inserted again, creating duplicate staging records.
  • No transactional boundary: the INSERT and the file move are not wrapped in a transaction. A crash between INSERT success and file move could result in the file being reprocessed on the next poll cycle.

Audit & Logging

  • Every processing outcome is logged to stdout via System.out.println with a [ZavaFileIngestion] prefix.
  • IngestedAt is set to GETDATE() at insert time, providing a server-side audit timestamp on every staging record.
  • No structured logging, correlation IDs, or distributed tracing are implemented.

Version & Compatibility Risks

The application targets Java 8, which reached Oracle end-of-life for public updates in March 2019 (OpenJDK builds continue via Adoptium/Temurin, but Java 8 is significantly behind current LTS releases Java 17 and 21). The mssql-jdbc 12.6.1.jre8 driver is modern and compatible with SQL Server 2016+, but the .jre8 classifier must be swapped for .jre11 or .jre17 if the runtime is upgraded. The amqp-client 5.21.0 is a current release and does not carry immediate version risk, but upgrading the Java runtime to 17 or 21 would require reverification of the library's class compatibility. The Gradle Shadow Plugin 7.1.2 used for fat-JAR packaging is compatible with Gradle 7.6 but would need to be updated alongside any Gradle upgrade.

Notable Observations

  • Minimal dependency footprint: With only two external libraries, the project carries a very small transitive dependency tree, which reduces supply-chain risk and simplifies future upgrades.
  • No logging framework declared: The application uses System.out.println directly. There is no SLF4J, Logback, or Log4j dependency, meaning log routing, levels, and structured output are not configurable at runtime.
  • Java 8 end-of-life risk: The build targets sourceCompatibility = JavaVersion.VERSION_1_8. Upgrading to Java 17 (or 21 LTS) would be advisable before containerising for Azure, as the base image (eclipse-temurin:8-jre) is an older image tier.
  • No security or observability libraries: The project has no authentication, tracing, or metrics library. Deploying to Azure would benefit from adding Azure Monitor / Application Insights or OpenTelemetry for observability.

No caching layer is configured. The application does not use Spring Cache, Redis, EhCache, Caffeine, or any other cache provider. Data Ownership Boundaries

ZavaFileIngestion owns all three staging tables exclusively. There is no evidence of other services sharing the ZavaBankCore database, but the connection string points to a shared SQL Server instance, meaning logical isolation is the only boundary — no schema-per-service or database-per-service pattern is enforced.

Cross-service data access is not applicable: the service performs no SELECT queries and exposes no read API. Downstream consumers must query the staging tables directly (via their own JDBC/ORM access) or consume the RabbitMQ events published after each ingestion.

The service is entirely write-path (INSERT-only); there is no CQRS separation, no read model, and no event-sourcing pattern.
