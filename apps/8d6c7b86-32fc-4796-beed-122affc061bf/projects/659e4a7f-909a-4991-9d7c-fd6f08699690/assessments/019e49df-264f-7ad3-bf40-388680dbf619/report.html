Data Storage & External Services

The application persists parsed file data into three staging tables (StagingCheckImage, StagingWireConfirmation, StagingRegulatoryFeed) in a Microsoft SQL Server database (ZavaBankCore) via raw JDBC connections (no ORM). After each file is processed, an event is published to a RabbitMQ topic exchange (zava.events) on the routing key file.ingested, enabling downstream consumers to react to ingestion outcomes. Connection parameters for both SQL Server and RabbitMQ can be supplied through environment variables, overriding the defaults in fileingestion.properties.

Key Architectural Decisions

  • Poll-based ingestion loop: The application scans a shared filesystem directory on a configurable interval (default 5 s) rather than using inotify or a message-driven trigger, keeping the design simple but requiring careful interval tuning under high volume.
  • JDBC without an ORM: Data is written using raw PreparedStatement calls, reducing dependencies but requiring manual resource management (connection/statement lifecycle handled via try/finally blocks).
  • File-type detection by filename convention: File routing is determined by matching keywords (check, wire, reg/feed) in the filename, making the routing logic tightly coupled to naming conventions of upstream file producers.

DTOs & Contracts

There are no formal DTO classes. All data is extracted inline within the processing methods using string splitting (splitCsv()) or character-position slicing (safeSlice()). Parsed fields are passed directly as String[] arrays to the JDBC insertion methods.

The outbound event contract is an unversioned hand-crafted JSON string with four fields:


{
  "eventType": "file.ingestion",
  "fileName": "<file name>",
  "fileType": "<check_images|wire_confirmations|regulatory_feeds>",
  "status": "<processed|failed>",
  "ingestedAt": "<epoch millis>"
}

There are no OpenAPI / Swagger specifications, protobuf schemas, GraphQL schemas, or Jackson configuration classes. No serialization library is used; JSON is constructed via string concatenation with a custom escape() helper.

Communication Patterns

Inbound (file-based): The application polls a local filesystem path on a configurable interval (default 5 000 ms). There is no push notification, inotify, or queue-based trigger; the service loops indefinitely calling File.listFiles().

Synchronous outbound — JDBC: For each parsed record the service opens a new JDBC Connection, executes a PreparedStatement INSERT, and immediately closes the connection. There is no connection pool, no retry policy, no timeout configuration, and no circuit breaker. A failure logs a warning and causes the file to be moved to the error directory.

Asynchronous outbound — AMQP: After processing each file (success or failure), the service creates a new RabbitMQ Connection and Channel, publishes a JSON event to the zava.events topic exchange with routing key file.ingested, and closes the connection immediately. There is no persistent channel, no publisher confirm, no retry, and no dead-letter configuration. A publish failure is logged and silently swallowed.

Resilience patterns: None. There is no Resilience4j, Spring Retry, Polly, or equivalent library. All failures result in a single attempt; no retry, circuit breaker, bulkhead, or timeout is applied to either the database or the messaging channel.

Service discovery: None. Connection endpoints for SQL Server and RabbitMQ are hardcoded in fileingestion.properties and may be overridden via environment variables (SQLSERVER_URL, RABBITMQ_HOST, etc.). There is no Eureka, Consul, or Kubernetes DNS integration.

Security posture: No authentication or TLS is configured on any outbound connection by default. The JDBC URL uses encrypt=false;trustServerCertificate=true. RabbitMQ credentials are guest/guest in the default configuration. All connections are plaintext. There is no inbound HTTP server, so no API-level authentication, authorization, or HTTPS applies.

Primary Workflows

Workflow 1: Continuous File Ingestion Loop

The service runs an infinite polling loop that periodically scans the watch directory. On each tick it discovers all files present, routes each one to the appropriate file-type handler, and then waits for the next interval before repeating. The loop runs until the JVM receives a shutdown signal.

Steps:

  1. Wait for the configured poll interval (default: every 5 seconds).
  2. List all files present in the watch directory (/shared/file-ingestion).
  3. For each file, determine its type by matching filename keywords (see Business Rules).
  4. Dispatch the file to the matching handler (Check Image, Wire Confirmation, or Regulatory Feed workflow).
  5. Move the file to /processed on success, or /error on failure.
  6. Publish an ingestion outcome event to RabbitMQ.
  7. Return to step 1.

Workflow 2: Check Image File Ingestion

Triggered when a CSV file whose name contains the keyword check (and ends with .csv) lands in the watch directory.

Steps:

  1. Open the file and read the first line as a comma-separated record.
  2. Extract three fields: CheckNumber (col 0), AccountNumber (col 1), ImagePath (col 2).
  3. Insert a row into the StagingCheckImage staging table with the extracted fields and the current timestamp.
  4. If the insert succeeds, publish a processed event; move the file to /processed.
  5. If any step fails (parse error, DB failure), publish a failed event; move the file to /error.

Workflow 3: Wire Confirmation File Ingestion

Triggered when a fixed-width text file whose name contains the keyword wire (and ends with .txt) lands in the watch directory.

Steps:

  1. Open the file and read all non-blank lines.
  2. For each line, extract three fields by character position: WireReference (chars 0–16), Amount (chars 16–28), Beneficiary (chars 28–68).
  3. Insert a row into StagingWireConfirmation for each parsed line.
  4. If at least one insert succeeded, publish a processed event; move the file to /processed.
  5. If all inserts fail or no non-blank lines were found, publish a failed event; move the file to /error.

Workflow 4: Regulatory Feed File Ingestion

Triggered when a file whose name contains reg or feed (with .csv or .txt extension) lands in the watch directory.

Steps:

  1. Open the file and read all non-blank lines as CSV records.
  2. For each line, extract three fields: FeedCode (col 0), RecordDate (col 1), Payload (col 2).
  3. Insert a row into StagingRegulatoryFeed for each parsed line.
  4. If at least one insert succeeded, publish a processed event; move the file to /processed.
  5. If all inserts fail or the file is empty, publish a failed event; move the file to /error.

Cross-Service Data Flows

ZavaFileIngestion is a single service with no inter-service API calls. Its cross-boundary data flows are:

  • Inbound: Upstream file-producing systems (imaging platforms, payment networks, regulatory systems) drop files into the shared filesystem directory. The ingestion service has no way to signal backpressure; it simply processes whatever is present on each poll cycle.
  • Outbound (persistence): Parsed records are written synchronously to SQL Server staging tables. Downstream consumers (not part of this codebase) are expected to poll or subscribe to these tables to continue processing.
  • Outbound (events): After each file is processed, a lightweight JSON event is published to the zava.events RabbitMQ topic exchange with routing key file.ingested. Downstream event-driven consumers can subscribe to this exchange to trigger post-ingestion workflows (e.g., further validation, enrichment, notification).

There is no aggregation logic, no circuit breaker fallback, and no cross-service data composition within this application.

Business Rules & Decision Logic

File Type Identification

A file is routed to a handler based on keyword and extension matching applied to its lowercase filename:

  • Check Image: filename contains check AND ends with .csv
  • Wire Confirmation: filename contains wire AND ends with .txt
  • Regulatory Feed: filename contains reg or feed AND ends with .csv or .txt
  • Fallback keyword matching: if none of the above match, a secondary scan checks for the presence of keywords check, wire, or reg anywhere in the filename (without extension restriction) and routes accordingly
  • Unmatched files: files that match none of the above patterns are silently skipped with no event published and no file move

Files whose names contain processed or error are explicitly excluded to prevent reprocessing of already-handled files.

Ingestion Success / Failure Logic

  • Check image files: success is determined by a single INSERT succeeding. Only the first CSV line is read; subsequent lines (if any) are ignored.
  • Wire confirmation / regulatory feed files: success is determined by at least one INSERT succeeding across all non-blank lines. Partial failures (some lines failing, some succeeding) are treated as overall success.
  • File disposition: once processing completes (success or failure), the source file is unconditionally moved. A file is never left in the watch directory after an ingestion attempt.

Validation Rules

  • No schema validation is applied to incoming files before parsing
  • Missing CSV fields default to empty strings (no hard rejection)
  • Fixed-width fields are trimmed of whitespace; positions beyond the line length return empty strings
  • No format checks are applied to dates, amounts, or reference numbers — all values are stored as raw strings

Error Handling

  • Parse errors: caught per-file; the file is moved to /error and a failed event is published
  • Database errors: caught per INSERT; if all inserts fail the file is moved to /error; no compensation or retry is performed
  • Messaging errors: caught per publish; a failed publish is logged and silently swallowed — the file disposition (move to /processed or /error) has already occurred at this point
  • No dead-letter queue: files in /error are not automatically retried; manual intervention is required

Transactions & Audit

  • No @Transactional or equivalent transaction boundaries are declared; each INSERT is auto-committed independently
  • The IngestedAt timestamp is set by SQL Server's GETDATE() function at INSERT time
  • No application-level audit log, change tracking, or structured event log is maintained beyond the RabbitMQ event and System.out.println console messages
  • No authorization or role-based access control is applied within the application

Version & Compatibility Risks

The two runtime dependencies are both reasonably current. Java 8 (sourceCompatibility = JavaVersion.VERSION_1_8) is the most significant compatibility concern: it reached its end of free public updates from Oracle and many container base images are moving to Java 17/21 LTS. The Shadow plugin (com.github.johnrengelman.shadow 7.1.2) and the Gradle wrapper (7.6) are also dated; Gradle 8.x and the newer shadow plugin major version have since been released. The JDBC driver version 12.6.1.jre8 is the last Java 8-compatible release, so a Java upgrade would require selecting the jre11/jre17 variant. The RabbitMQ AMQP client 5.21.0 is compatible with Java 8 through 21, so it poses no immediate migration risk.

Notable Observations

  • No logging framework declared: The application uses bare System.out.println for all logging. There is no SLF4J, Logback, or Log4j2 dependency, making structured logging, log-level control, and centralised log aggregation impossible without a dependency addition.
  • No ORM or connection pool: Data access relies on opening a new raw JDBC Connection per SQL statement and closing it immediately after. There is no HikariCP, DBCP2, or equivalent connection pool, which can lead to performance degradation under high file throughput.
  • Minimal dependency surface: With only 2 runtime dependencies the transitive dependency tree is small, which is an advantage for security scanning and containerisation. However it also means common cross-cutting concerns (retry, metrics, configuration binding) are implemented ad-hoc in application code.
  • No observability library: There is no Micrometer, OpenTelemetry, or Application Insights dependency. Migrating to a managed Azure service (AKS / Container Apps) without instrumentation would leave the application unmonitored.

Data Architecture & Persistence Layer

ZavaFileIngestion writes parsed financial data to three staging tables in a Microsoft SQL Server database using raw JDBC with no ORM, no schema migration tool, and no caching layer.

Caching Strategy

No caching layer is implemented. Each parsed record results in a direct synchronous database write with a freshly opened JDBC connection. There is no in-memory cache, no second-level cache, no query result cache, and no distributed cache (Redis, EhCache, Caffeine, etc.).

Data Ownership Boundaries

ZavaFileIngestion is a single-service application. All three staging tables reside in one shared SQL Server database (ZavaBankCore). There are no other services accessing these tables through this application; downstream consumers are expected to read directly from the staging tables or be triggered by the RabbitMQ events published after each successful ingestion.

Cross-service data access: Not applicable within this service. External consumers (not part of this codebase) own the downstream processing of staging records.

Read/write pattern: Write-only. The application never reads back from the database after inserting. There is no CQRS separation, no read replica, and no event sourcing.

Connection isolation: A new java.sql.Connection is opened and closed for every INSERT statement. Under high file throughput this will create significant connection pressure on the SQL Server instance. No connection pool (HikariCP, DBCP2, etc.) is configured.
