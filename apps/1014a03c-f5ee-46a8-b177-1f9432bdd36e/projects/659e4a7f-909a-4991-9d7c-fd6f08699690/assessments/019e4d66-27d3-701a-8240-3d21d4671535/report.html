Data Storage & External Services

The application writes a single audit record to a SQL Server database (ZavaBankCore) via bare JDBC after every batch run. The BatchRunLog table captures run type, start/end timestamps, overall status (SUCCESS/FAILED), and per-service boolean outcomes. No ORM or connection pool is used; a new DriverManager.getConnection is opened and closed per batch run. Three external HTTP microservices are invoked synchronously: the interest accrual service, the statement generation service, and the daily reconciliation service. All three are called over plain HTTP with a configurable connect/read timeout (default 15 s). A RabbitMQ amqp-client dependency is declared in build.gradle but is not referenced in any source code.

Key Architectural Decisions

  • Single-class, framework-free design: The entire orchestration logic resides in one Main.java file with no DI container, no Spring, and no Jakarta EE — keeping the deployment artifact minimal (Shadow JAR).
  • Configuration externalised via properties + env vars: batchscheduler.properties provides defaults; ten well-named environment variables (BATCH_SCHEDULER_INTERVAL_MS, SQLSERVER_, _SERVICE_URL, etc.) allow runtime override without rebuilding the image.
  • Fixed-delay scheduling with graceful shutdown: ScheduledExecutorService.scheduleWithFixedDelay ensures runs do not overlap; a JVM shutdown hook signals the scheduler to stop, preventing mid-run termination.

Synchronous REST (outbound only): All three downstream calls use java.net.HttpURLConnection with a POST method and Content-Type: application/json. Service discovery: None. Downstream base URLs are hard-coded in batchscheduler.properties (e.g., http://zavainterestcalculator:8080) and resolved via Docker/Kubernetes internal DNS. No Eureka, Consul, or cloud service discovery client is used.

API gateway: None. ZavaBatchScheduler calls each downstream service directly; there is no gateway or BFF layer.

Security posture: No authentication, no TLS, and no authorization are configured. All outbound calls are plain HTTP with no Authorization header, no client certificates, and no token exchange. The downstream services are assumed to be reachable only within a private container network and not exposed publicly; however, this is not enforced at the application level.

Primary Workflows

Workflow 1: Nightly Batch Run Orchestration

Trigger: Fixed-delay scheduler fires every 5 minutes (configurable; typically set to 24 hours in production via BATCH_SCHEDULER_INTERVAL_MS).

Steps:

  1. Scheduler fires — ScheduledExecutorService invokes executeBatchRun(). A running flag is checked first; if shutdown is in progress, the run is skipped.
  2. Record run start time — current timestamp captured as runStart.
  3. Call Interest Accrual — HTTP POST sent to zavainterestcalculator. Within 15 s, either a 2xx response sets interestOk = true, or a non-2xx or timeout sets interestOk = false. The scheduler does not abort; it always proceeds to the next service.
  4. Call Statement Generation — HTTP POST sent to zavastatementservice. Same timeout and outcome logic; sets statementOk.
  5. Call Daily Reconciliation — HTTP POST sent to zavaledger. Sets reconOk. This is always the last call, ensuring that interest and statement activity are already committed before ledger reconciliation begins.
  6. Determine batch status — If interestOk AND statementOk AND reconOk → SUCCESS; otherwise → FAILED.
  7. Write audit record — BatchRunLog row inserted with run type, start/end timestamps, composite status, and per-service detail string (e.g., interest=true, statements=true, reconciliation=false).
  8. Log to stdout — Run outcome emitted to standard output with ISO-8601 timestamp.

Business significance: A FAILED status indicates at least one downstream banking operation did not complete, which may require manual intervention or re-run. The Details field identifies which service(s) failed without requiring log analysis.

Workflow 2: Graceful Shutdown

Trigger: JVM receives SIGTERM or SIGINT (e.g., container stop, kubectl delete pod).

Steps:

  1. JVM shutdown hook sets running = false.
  2. scheduler.shutdownNow() is called — cancels pending future runs.
  3. Any in-progress executeBatchRun() invocation completes naturally (it checks running only at the top of the method).
  4. Shutdown is logged to stdout.

Business significance: A batch run that starts before shutdown will complete and write its BatchRunLog record. Shutdown does not cancel an in-flight run.

Cross-Service Data Flows

ZavaBatchScheduler is a fan-out orchestrator with no response aggregation. It sends three independent POST requests, each carrying a minimal trigger payload ({"job":"<job_name>"}). It does not consume or merge the response bodies from downstream services — only the HTTP status code is used to determine success.

Flow pattern:

  • ZavaBatchScheduler → zavainterestcalculator: trigger-only (no data returned to orchestrator)
  • ZavaBatchScheduler → zavastatementservice: trigger-only (no data returned to orchestrator)
  • ZavaBatchScheduler → zavaledger: trigger-only (no data returned to orchestrator)

Fallback behavior: When a downstream service is unavailable or returns an error, ZavaBatchScheduler records false for that service in BatchRunLog.Details and continues calling the remaining services. There is no retry, no compensating action, and no notification mechanism (e.g., email, alert). The only signal of failure is the FAILED status in BatchRunLog and the stdout log entry. Operators must monitor the BatchRunLog table or application logs to detect and respond to failures.

Call ordering: The three downstream calls are always executed sequentially in this fixed order: interest accrual → statement generation → reconciliation. If interest accrual fails, statement generation and reconciliation are still triggered. This means a partial run (e.g., interest accrued but statements not generated) is possible and logged as FAILED.

Business Rules & Decision Logic

Business Rules

Batch success rule: A batch run is SUCCESS only when all three downstream service calls return HTTP 2xx within the configured timeout. Any single failure causes the overall status to be FAILED. There is no concept of partial success at the batch level.

Sequential ordering constraint: Interest accrual must be triggered before statement generation, and statement generation before reconciliation. This ordering ensures that interest postings are reflected in statements, and that all daily transactions (including interest and new statements) are visible to the reconciliation job. This ordering is hard-coded and not configurable.

Non-interrupting failure rule: A failed service call does not abort the batch run. All three services are always called, regardless of prior outcomes. This prevents a failure in interest accrual from blocking the reconciliation job from running.

Configurable interval: The scheduling interval is an operational parameter — the default of 300 s (5 min) is intended for development/testing; production deployments are expected to set BATCH_SCHEDULER_INTERVAL_MS to a 24-hour value aligned with the close-of-business processing window.

Audit completeness rule: Every batch run — successful or failed — must produce a BatchRunLog record. If the database write itself fails, the failure is logged to stdout but does not affect the scheduler's continued operation.

Cross-Cutting Concerns

Transactions: No distributed transaction coordination exists. Each downstream service manages its own transaction internally. ZavaBatchScheduler has no rollback capability; a partial run (e.g., interest accrued but reconciliation failed) cannot be automatically reversed.

Error handling: All exceptions in callService() and insertBatchRunLog() are caught, logged to stdout, and suppressed. The scheduler continues running indefinitely. There is no dead-letter queue, no alerting integration, and no escalation path beyond the BatchRunLog table.

Audit/logging: Every batch run produces two audit artifacts: a stdout log line and a BatchRunLog database row. The database row is the durable record; stdout logs are ephemeral unless captured by a log aggregation system.

Authorization: No business-level authorization is implemented. ZavaBatchScheduler calls all downstream services without any authentication token or credential. The downstream services are assumed to accept unauthenticated requests from within the container network.

Version & Compatibility Risks

mssql-jdbc 12.6.1.jre8 is a recent, actively maintained release and is compatible with Java 8 through 21, posing no immediate migration risk. The jre8 classifier bundles JDBC 4.2 support and is the correct variant for the Java 8 runtime; upgrading the JRE to 11+ would allow switching to the jre11 or plain classifier. amqp-client 5.21.0 is a current RabbitMQ release; however, because it is not exercised in the codebase it adds unnecessary transitive dependencies (notably slf4j-api and the RabbitMQ binary protocol codec) to the Shadow JAR without any functional benefit.

Notable Observations

  • Unused messaging dependency: com.rabbitmq:amqp-client:5.21.0 is declared as a compile-scope implementation dependency but is never imported or used in Main.java. It inflates the Shadow JAR and may signal an unfinished migration away from RabbitMQ or a copy-paste from another project.
  • No logging framework declared: The application writes to stdout via System.out.println. No SLF4J, Logback, or Log4j2 is declared; structured/configurable logging is absent.
  • No connection pooling: Database connections are opened and closed per batch run using raw DriverManager. There is no HikariCP or c3p0 dependency, which is acceptable for a low-frequency scheduler but worth noting for migration to a managed platform.
  • Minimal dependency footprint: Only 2 runtime dependencies make the project easy to containerise and migrate, but also indicate that capabilities expected from a production service (retry libraries, metrics, health endpoints) are currently missing.

Data Architecture & Persistence Layer

ZavaBatchScheduler interacts with a single SQL Server database through bare JDBC — no ORM, no repository abstraction, and no entity model. The entire persistence layer consists of one INSERT statement that writes an audit record to the BatchRunLog table after each batch run.

Caching Strategy

No caching layer of any kind is implemented. The application does not declare any cache provider (EhCache, Redis, Caffeine, etc.), does not use Spring Cache annotations, and does not maintain in-memory state between batch runs. Each run starts fresh.

Data Ownership Boundaries

ZavaBatchScheduler is the sole writer to the BatchRunLog table in the shared ZavaBankCore SQL Server database. No other service in this repository reads from or writes to BatchRunLog; however, because the database is the central ZavaBankCore instance, other banking services may share the same SQL Server instance and have access to the same schema unless row-level or schema-level isolation is enforced externally.

The application performs no reads from SQL Server — data flows in one direction only (INSERT after each batch run). There is no CQRS, no event sourcing, and no cross-service data aggregation at the database level. The three downstream services (interest, statements, reconciliation) own their own data stores, which are not accessed by ZavaBatchScheduler directly.
