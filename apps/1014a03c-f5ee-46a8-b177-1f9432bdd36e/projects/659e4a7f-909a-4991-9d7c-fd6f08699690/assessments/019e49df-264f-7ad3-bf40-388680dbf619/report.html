Data Storage & External Services

The scheduler writes one row per batch execution to a SQL Server BatchRunLog table and coordinates three external HTTP services: interest accrual, statement generation, and daily reconciliation. It acts as an orchestration component and does not expose inbound APIs.

Key Architectural Decisions

  • Uses a single-process scheduled executor to run periodic batch orchestration without a separate job server.
  • Implements outbound service integration with direct HTTP requests and captures per-service success status in run details.
  • Externalizes all key endpoints and database connection settings through properties with environment-variable overrides.

Primary Workflows

Workflow 1: Scheduled Nightly Batch Orchestration

  1. Scheduler timer triggers a run based on scheduler.interval.ms.
  2. Scheduler submits batch command to interest service.
  3. Scheduler submits batch command to statement service.
  4. Scheduler submits batch command to reconciliation service.
  5. Scheduler evaluates overall success only when all three calls return success status.
  6. Scheduler persists final run status and details to BatchRunLog.

Workflow 2: Operational Failure Logging

  1. Any downstream non-2xx response or request exception is interpreted as failed job execution.
  2. Scheduler continues processing remaining jobs in the same cycle.
  3. End-of-run status is marked FAILED when one or more jobs fail.
  4. Failure details are written to BatchRunLog for operational troubleshooting.

Cross-Service Data Flows

The scheduler composes a run-level outcome by collecting status signals from three downstream services and reducing them into a single SUCCESS or FAILED state. Data exchange is command-and-status oriented: each service returns an HTTP status/body that is aggregated into the final log detail string. When one service is unavailable, the batch cycle degrades by recording a failed run while still attempting all configured calls.

Business Rules & Decision Logic

  • A batch run is successful only when all downstream jobs return successful HTTP statuses.
  • Missing URL or endpoint configuration for any job is treated as a failed job result.
  • Scheduler prioritizes continuity over fail-fast behavior by attempting all job calls in each cycle.
  • Persisting run metadata is mandatory for auditability; insert failures are logged but do not stop the scheduler loop.
