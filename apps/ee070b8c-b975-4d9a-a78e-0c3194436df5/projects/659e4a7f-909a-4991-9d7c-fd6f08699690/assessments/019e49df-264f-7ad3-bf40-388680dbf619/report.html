Data Storage & External Services

The application uses a single external data store: a Microsoft SQL Server instance hosting the ZavaBankDB database. Connection details (host, port, database name, credentials) are resolved at runtime from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with fallback to ledger.properties. There are no other external services, message brokers, caches, or third-party APIs.

Key Architectural Decisions

  • Direct JDBC without ORM: All database interactions use DriverManager.getConnection() and raw PreparedStatement calls; no JPA/Hibernate or connection pool is configured.
  • XML as the API wire format: All endpoints produce and consume application/xml; responses are hand-assembled string XML, with a dedicated LedgerXml utility for escaping.
  • Dual configuration source: Runtime configuration is read first from environment variables, then from ledger.properties, allowing container-friendly deployment without code changes.

DTOs & Contracts

Inline / anonymous contracts (no formal DTO classes):

All request and response types are assembled inline in each servlet using raw StringBuilder XML. There are no external DTO classes, OpenAPI/Swagger specifications, protobuf schemas, or GraphQL schemas.

Two private static inner classes exist only within TransactionServlet:

  • TransactionRequest — captures parsed XML input fields (debit/credit account IDs, amount, description, reference number). Not exposed externally; not serialized.
  • TransactionResult — carries DB-generated transaction IDs and updated balances for building the XML response. Not exposed externally; not serialized.

There is no Jackson, JAXB, or other serialization framework. All XML is produced by string concatenation and sanitized via LedgerXml.escape(). No OpenAPI specification file exists.

Communication Patterns

Synchronous only. All communication is synchronous HTTP. There is no asynchronous messaging, event-driven pattern, message broker, or pub/sub mechanism.

Single service — no inter-service communication. ZavaLedger is a monolithic WAR with no upstream or downstream service calls. The only external dependency is the SQL Server database, accessed via JDBC on every request.

No connection pooling or resilience patterns. A new JDBC connection is opened per request via DriverManager.getConnection(). There is no circuit breaker, retry policy, timeout configuration, or connection pool (HikariCP, DBCP, etc.).

No service discovery. Database coordinates are resolved from environment variables or ledger.properties at startup. There is no service registry (Eureka, Consul, Kubernetes DNS resolution) in use.

No API gateway. The servlet container (e.g., Tomcat) is the sole entry point. There is no reverse proxy, API gateway, or load balancer declared in the project.

Security posture: No authentication, no authorization, no TLS configured. All three endpoints (/health, /api/accounts/*, /api/transactions) are publicly accessible with no authentication or authorization checks. No security filter, JAAS, Spring Security, or JASPIC implementation is present. The JDBC URL sets encrypt=false, disabling TLS between the application and SQL Server.

ZavaLedger is a banking ledger system that manages account balances and records double-entry financial transactions — debiting one account and crediting another atomically in a single operation.

Primary Workflows

Workflow 1: Query Account Balance

An API client requests the current balance and available balance for a specific account by account ID. The servlet validates the URL path segments, queries the Accounts table, and returns a structured XML response.

Steps:

  1. Client sends GET /api/accounts/{id}/balance
  2. Path segments are parsed and the account ID is validated as numeric
  3. A JDBC query retrieves Balance and AvailableBalance from Accounts where AccountID = {id}
  4. If the account exists, a success XML response is returned with both balance fields
  5. If the account does not exist, a NOT_FOUND XML response is returned with HTTP 404

Workflow 2: List Account Transactions

An API client requests the full transaction history for a specific account, ordered by most recent first.

Steps:

  1. Client sends GET /api/accounts/{id}/transactions
  2. Path segments are parsed and the account ID is validated as numeric
  3. A JDBC query retrieves all transaction fields from Transactions where AccountID = {id}, ordered by TransactionDate DESC, TransactionID DESC
  4. Each transaction row is serialized to XML with XML-escaped string fields
  5. The full transaction list is returned wrapped in a transactionsResponse envelope

Workflow 3: Post a Double-Entry Transaction

This is the core business workflow. A client submits a transfer between two accounts. The application debits one account and credits another atomically within a single SQL transaction.

Steps:

  1. Client sends POST /api/transactions with an XML body
  2. XML request body is parsed via a secure DocumentBuilder (XXE protections enabled)
  3. Input fields are extracted: debitAccountId, creditAccountId, amount, description, referenceNumber
  4. Business rules are validated (see Business Rules section)
  5. A JDBC transaction is opened (autoCommit=false)
  6. Current balances are read for both the debit and credit accounts
  7. Active TransactionType IDs are resolved for debit and credit legs
  8. New balances are computed: debit account balance decreases by amount; credit account balance increases
  9. Both account balances are updated in Accounts
  10. Two transaction ledger entries are inserted into Transactions (one debit leg, one credit leg)
  11. The JDBC transaction is committed
  12. The generated transaction IDs and resulting balances are returned in the XML response
  13. On any error (validation or SQL), the JDBC transaction is rolled back

Workflow 4: Health Check

A lightweight liveness check used by load balancers or orchestration platforms.

Steps:

  1. Client sends GET /health
  2. The servlet returns a static HTML page confirming the application is online

Cross-Service Data Flows

ZavaLedger is a single-service application with no cross-service data flows. All data originates from and is persisted to a single SQL Server database. There is no gateway aggregation, event-driven composition, or inter-service communication.

Security Rules (XXE Prevention)

The XML parser in TransactionServlet is explicitly hardened against XML External Entity (XXE) injection:

  • disallow-doctype-decl = true
  • external-general-entities = false
  • external-parameter-entities = false
  • XIncludeAware = false
  • expandEntityReferences = false

State Transitions

Account balances undergo the following state change on a successful transaction post:

  • Debit account: Balance and AvailableBalance → reduced by amount; LastActivityDate and ModifiedDate → updated to current time
  • Credit account: Balance and AvailableBalance → increased by amount; LastActivityDate and ModifiedDate → updated to current time

No account lifecycle state machine exists (no account status, frozen, or closed states are tracked at the application layer).

Transaction Boundaries

All write operations in the transaction posting workflow (TransactionServlet.postTransaction()) execute within a single JDBC transaction:

  • connection.setAutoCommit(false) before any writes
  • connection.commit() after all inserts and updates succeed
  • connection.rollback() on any SQLException or IllegalArgumentException

This ensures the double-entry invariant: either both the debit and credit legs are posted together, or neither is.

Version & Compatibility Risks

The Servlet API 4.0.1 (javax.servlet) is the last release under the javax.* namespace before the Jakarta EE rename (jakarta.servlet). It corresponds to the Java EE 8 / Servlet 4.0 specification, which is frozen and no longer actively maintained upstream. Modern servlet containers (Tomcat 10+, Jetty 11+) have migrated to jakarta.servlet, meaning a future container upgrade would require namespace migration of all import statements. The mssql-jdbc driver at 12.8.1 is a recent release and carries no immediate compatibility concerns with Java 17, although encrypt=false in the JDBC URL disables TLS in transit, which is a security concern.

Notable Observations

  • No connection pooling library: The application creates a new DriverManager.getConnection() on every request. There is no connection pool (HikariCP, DBCP, c3p0), which limits throughput and may exhaust database connections under load.
  • Legacy javax.* namespace: Dependence on javax.servlet-api:4.0.1 (Java EE 8) rather than jakarta.servlet-api:6.x (Jakarta EE 10) will require a namespace migration before deploying to any modern Jakarta EE-compatible servlet container.
  • No logging library declared: The application has no declared logging dependency (SLF4J, Log4j, Logback). Errors are swallowed silently or returned as HTTP error responses, complicating observability and debugging in production.
  • No security or authentication library: There is no declared security framework (e.g., Spring Security, JASPIC implementation). All API endpoints are open with no authentication or authorization controls.

No caching layer is configured. There is no Redis, EhCache, Caffeine, Hibernate second-level cache, or Spring Cache (@Cacheable) in use. All queries are executed against SQL Server on every request. The TransactionTypes lookup in particular (SELECT TOP 1 TransactionTypeID FROM TransactionTypes WHERE IsDebit = ?) is executed on every transaction post and is a natural caching candidate, as this reference data is unlikely to change frequently.

Data Ownership Boundaries

ZavaLedger is a single-service monolith with one shared SQL Server database (ZavaBankDB). There are no service boundaries, schema-per-service separation, or cross-service data access patterns. There is no CQRS separation; the same service performs both read and write operations. Transaction isolation is enforced at the JDBC connection level using setAutoCommit(false) with explicit COMMIT/ROLLBACK.
