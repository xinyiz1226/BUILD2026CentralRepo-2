Data Storage & External Services

The application uses a single Microsoft SQL Server database (ZavaBankDB) as its only data store. Connection parameters (host, port, database name, username, password) are resolved at runtime from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with a fallback to ledger.properties. No caching layer, message broker, or external API integrations are present.

Key Architectural Decisions

  • Direct JDBC with manual connection management: The application uses raw JDBC (no ORM or connection pool framework), opening and closing connections per request inside LedgerConnectionFactory.
  • XML-over-HTTP protocol: All API endpoints produce and consume XML payloads; LedgerXml.escape() provides minimal XSS/injection protection for XML output.
  • Transaction atomicity via JDBC transactions: TransactionServlet.postTransaction() uses explicit connection.setAutoCommit(false) / commit() / rollback() to ensure double-entry bookkeeping consistency.

DTOs & Contracts

All request and response data transfer is handled via inline XML serialization within servlet handler methods — there are no dedicated DTO or contract classes. Two private static inner classes within TransactionServlet serve as internal data holders:

  • TransactionRequest (private static inner class) — carries parsed XML request fields (debitAccountId, creditAccountId, amount, description, referenceNumber); used only internally within TransactionServlet.postTransaction().
  • TransactionResult (private static inner class) — carries result data (debitTransactionId, creditTransactionId, debitBalance, creditBalance) returned from postTransaction() for XML serialization; used only internally.

Neither class is exposed as an API contract type. There are no OpenAPI/Swagger specifications, protobuf schemas, GraphQL schemas, or Jackson/JAXB serialization configuration. XML output is constructed manually via StringBuilder and LedgerXml.escape().

Communication Patterns

Synchronous only: All client-to-service communication is synchronous HTTP request/response. The application has no asynchronous messaging (no Kafka, RabbitMQ, or equivalent), no event-driven patterns, and no inter-service calls.

No resilience patterns: There is no circuit breaker, retry policy, timeout configuration, or bulkhead pattern implemented at the API layer.

No service discovery: The application does not register with any service registry (Eureka, Consul, Kubernetes DNS). The only external connection is to SQL Server, configured via environment variables or ledger.properties.

No API gateway: Requests arrive directly at servlet endpoints with no intermediary gateway, BFF, or load balancer defined at the application level.

Security posture: No authentication, authorization, or TLS is configured at the application level. All four endpoints (/health, /api/accounts/*, /api/transactions) are publicly accessible with no credential checks, role enforcement, or transport security. There is no Spring Security, servlet filter, or any other access-control mechanism present in the codebase.

Core Business Workflows

ZavaLedger is a banking ledger service that records double-entry bookkeeping transactions and provides account balance and transaction history inquiries for the ZavaBank system.

Primary Workflows

Workflow 1: Post a Double-Entry Ledger Transaction

Entry point: POST /api/transactions (XML request body)

A caller submits a transfer request specifying a debit account, a credit account, and an amount. The service validates the request, verifies both accounts exist, applies the double-entry balance update atomically, and records two transaction ledger entries (one debit leg and one credit leg) in a single JDBC transaction.

Steps:

  1. Parse XML request — DocumentBuilderFactory with XXE protections parses the XML body; extracts debitAccountId, creditAccountId, amount, description, referenceNumber.
  2. Validate input — Check that debit and credit account IDs are different; check that amount is greater than zero.
  3. Open JDBC transaction — A single Connection is obtained; setAutoCommit(false) begins the transaction.
  4. Read current balances — Query Accounts for both account IDs; if either account is missing, throw IllegalArgumentException.
  5. Resolve transaction types — Query TransactionTypes for the active debit type and the active credit type.
  6. Calculate new balances — debitNewBalance = debitCurrentBalance − amount; creditNewBalance = creditCurrentBalance + amount.
  7. Update account balances — Two UPDATE Accounts statements set Balance, AvailableBalance, LastActivityDate, ModifiedDate for each account.
  8. Insert transaction records — Two INSERT INTO Transactions statements record the debit leg (negative amount) and credit leg (positive amount), including BalanceAfter, Description, ReferenceNumber, CounterpartyAccount, and Status = 'Posted'.
  9. Commit or rollback — On success, connection.commit() is called. On any SQLException or IllegalArgumentException, connection.rollback() is called.
  10. Return result — XML <transactionResponse> with both transaction IDs and updated balances is returned (HTTP 200), or an error response (HTTP 400 or 500).

Workflow 2: Query Account Balance

Entry point: GET /api/accounts/{id}/balance

A caller requests the current balance for a specific account. The service queries the Accounts table and returns Balance and AvailableBalance in XML format.

Steps:

  1. Parse account ID — Extract integer account ID from URL path; return HTTP 400 if non-numeric.
  2. Query balance — SELECT Balance, AvailableBalance FROM Accounts WHERE AccountID = ?.
  3. Return result — XML <balanceResponse> with status OK and balance values (HTTP 200), or status NOT_FOUND (HTTP 404) if no account row exists.

Workflow 3: Query Account Transaction History

Entry point: GET /api/accounts/{id}/transactions

A caller requests the full transaction history for a specific account ordered by most recent first. The service queries the Transactions table and streams all entries as XML.

Steps:

  1. Parse account ID — Extract integer account ID from URL path; return HTTP 400 if non-numeric.
  2. Query transactions — SELECT ... FROM Transactions WHERE AccountID = ? ORDER BY TransactionDate DESC, TransactionID DESC.
  3. Stream XML response — Each transaction row is serialized inline via StringBuilder with LedgerXml.escape() applied to string fields; all rows are returned in a single XML <transactionsResponse> (HTTP 200 or HTTP 500 on SQL error).

Workflow 4: Health Check

Entry point: GET /health

A simple liveness indicator that returns a static HTML page with the text "ZavaLedger - Online". No database connectivity is verified; this is a shallow application-process check only.

Cross-Service Data Flows

ZavaLedger has no cross-service data flows. It is a standalone service with no outbound calls to other services, no message broker integration, and no event publication. All data originates from and is written to its own SQL Server database. There is no gateway aggregation, circuit breaker, or fallback behavior involving other services.

Transaction Atomicity

All balance updates and transaction record insertions for a single transfer are wrapped in a single JDBC transaction (explicit setAutoCommit(false) / commit() / rollback()). This guarantees double-entry consistency: either both the debit and credit legs are committed together, or neither is. No saga pattern or eventual consistency is involved.

State Transitions

Transaction records are created in a single final state: Status = 'Posted', Channel = 'API'. There are no pending, cancelled, or reversed transaction states managed by the application. Balance state is updated immediately and synchronously on the Accounts table upon commit.

Error Handling

  • IllegalArgumentException from business rule violations → HTTP 400 with XML error body
  • All other exceptions (including SQLException) → HTTP 500 with generic XML error body ("Unable to post transaction.")
  • No business exception hierarchy, no compensating actions beyond JDBC rollback, and no dead-letter or retry handling

Authorization

No authentication, authorization, role-based access control, or audit logging is implemented. All endpoints are publicly accessible. No business-level authorization rules (e.g., "account owner can only view their own balance") are enforced.

Data Integrity

LedgerXml.escape() applies XML character escaping (&, <, >, ", ') to string fields in XML responses (Description, ReferenceNumber, CounterpartyAccount, Status) to prevent XML injection in output. No equivalent input sanitization beyond XML schema validation is applied to text fields stored in the database.

Version & Compatibility Risks

javax.servlet-api 4.0.1 uses the legacy javax. namespace (Jakarta EE 8 era). The Jakarta EE ecosystem migrated to jakarta. namespaces starting with Jakarta EE 9. Any migration to a modern servlet container (e.g., Tomcat 10+, Jetty 11+) that implements Jakarta EE 10+ would require refactoring all import statements from javax.servlet. to jakarta.servlet.. The mssql-jdbc driver version 12.8.1 is current and actively maintained; however, the application uses direct JDBC without a connection pool (e.g., HikariCP), which poses scalability and resource-leak risks under concurrent load.

Notable Observations

  • No connection pooling library: The application opens a new DriverManager.getConnection() per request with no pool, which can lead to connection exhaustion and degraded performance at scale.
  • No logging framework: Neither SLF4J, Log4j2, nor Logback is declared. Errors are silently swallowed (e.g., catch (IOException ignored) in LedgerConfig), making production troubleshooting difficult.
  • No ORM or query builder: All SQL is hand-written in servlet classes, increasing maintenance burden and exposure to SQL errors; no abstraction layer separates data access from business logic.
  • Minimal dependency footprint: The very small dependency set reduces supply-chain risk but also indicates a lack of standard enterprise library adoption (no security framework, no observability, no caching).

No caching layer is implemented. There is no Redis, EhCache, Caffeine, Spring Cache, or any other in-memory cache. Every request opens a new JDBC connection and executes live SQL queries against SQL Server. Query results are not cached at any layer.

Data Ownership Boundaries

The application operates with a single shared SQL Server database (ZavaBankDB) accessed by a single service (ZavaLedger WAR). There are no service boundaries, database-per-service isolation, or cross-service data access patterns — all data access originates from the same WAR.

No CQRS pattern is applied; all reads and writes share the same JDBC connection and execute against the same database. No event sourcing, outbox pattern, or eventual consistency mechanism is present.
