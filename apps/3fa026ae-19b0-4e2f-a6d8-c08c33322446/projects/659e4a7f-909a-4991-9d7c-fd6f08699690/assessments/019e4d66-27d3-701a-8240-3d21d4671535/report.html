Data Storage & External Services

The service persists and reads all operational data from a single SQL Server database (loan applications, accounts, credit scores, risk factors, and model parameters). External dependencies are limited to runtime configuration sources (environment variables and web.config) used to construct database connection settings.

Key Architectural Decisions

  • Uses a single WCF service contract (IRiskEngineService) with three business operations rather than separate API modules.
  • Uses direct SQL through ADO.NET instead of an ORM, with SQL embedded in service methods.
  • Creates/updates model configuration in-database (RiskModelParameters) during runtime to allow parameter tuning without redeploying binaries.

DTOs & Contracts

Service-level contracts are defined in RiskContracts.cs and exposed through IRiskEngineService. Request/response contract classes include RiskScoreResponse, RiskFactorsResponse, RiskFactorItem, RiskModelUpdateRequest, RiskModelParameter, and RiskModelUpdateResponse. These are mutable C# classes with DataContract/DataMember attributes (not immutable record types). No gateway-level aggregation DTOs, OpenAPI/Swagger specs, GraphQL schemas, or protobuf contracts were detected.

Communication Patterns

  • Synchronous communication: All API operations are synchronous WCF calls over basicHttpBinding; no asynchronous message-broker pattern is present.
  • Persistence interaction: Service methods execute synchronous SQL calls via ADO.NET within request scope.
  • Resilience patterns: No explicit retry policy, circuit breaker, timeout policy, or bulkhead mechanism is configured in code.
  • Service discovery/gateway: No service discovery or API gateway layer is implemented; callers use direct service endpoint addressing.
  • Startup dependency note: API availability depends on SQL Server connectivity at request time because each operation opens a DB connection.
  • Security posture: No explicit API authentication/authorization policy is configured in service contract settings, and no TLS-only requirement is defined in project configuration.

Core Business Workflows

This service supports lending risk decisions by scoring loan applications, producing customer risk factor breakdowns, and allowing controlled updates to scoring model parameters. The primary business value is a deterministic risk rating and approval guidance based on customer financial signals.

Primary Workflows

Workflow 1: Evaluate Loan Application Risk

Entry point: SOAP operation ScoreLoanApplication.

  1. Validate request (applicationId must be greater than zero).
  2. Retrieve latest application/customer financial context from SQL data.
  3. Load model coefficients and normalize credit, balance, term, and requested amount signals.
  4. Compute weighted score, convert to rating band (A-F), and derive max approved amount.
  5. Persist risk assessment record for audit/business reuse.
  6. Return scoring outcome to caller; return a fault when input or data lookup fails.

Workflow 2: Produce Customer Risk Factor Breakdown

Entry point: SOAP operation GetRiskFactors.

  1. Validate request (customerId must be greater than zero).
  2. Aggregate latest credit score, open exposure, average balance, and active account count.
  3. Map raw values into normalized business factors.
  4. Apply factor weights and compute overall weighted risk score.
  5. Return customer factor details and computed total score.

Workflow 3: Update Scoring Model Parameters

Entry point: SOAP operation UpdateRiskModel.

  1. Validate update payload contains at least one parameter.
  2. Open transaction and iterate supplied parameters.
  3. Upsert each parameter into model table with modifier identity.
  4. Commit transaction and return update summary.

Cross-Service Data Flows

This repository implements a single service with no gateway composition between multiple runtime services. Business data is composed through synchronous SQL access to shared domain tables: application/customer context is read, scored, then persisted as a risk assessment artifact. When required source records are unavailable, the business outcome degrades to an explicit fault response rather than partial fallback data.

Business Rules & Decision Logic

  • Validation rules: applicationId and customerId must be positive; model update payload must include at least one valid parameter name.
  • Decision logic: Weighted score combines normalized credit score, balance ratio, term risk, and requested amount using tunable coefficients; final rating maps score bands to A-F outcomes.
  • State transitions: Loan assessment moves from input retrieval to computed outcome to persisted RiskAssessments record.
  • Computed values: CalculatedScore, RiskRating, and MaxApprovedAmount are derived values based on model parameters and score thresholds.
  • Data integrity rules: Model parameter updates execute inside a SQL transaction to preserve consistency across multiple upserts.
  • Error handling and authorization posture: Business errors are surfaced via FaultException; no domain-level authorization rules are implemented in repository code.

Caching Strategy

No explicit caching layer is implemented. Each request executes live SQL reads/writes, and there is no in-memory cache, distributed cache, or second-level ORM cache configured.

Data Ownership Boundaries

The application uses a shared database model where one service owns scoring logic and both reads core banking tables and writes risk-specific tables in the same SQL Server database. Cross-service data exchange is not implemented via APIs or events in this repository; integration occurs through shared relational data. Read/write behavior is strongly request-driven and synchronous, with model parameter writes and assessment inserts occurring in-line during service operations.
