Data Storage & External Services

The application uses a single Microsoft SQL Server database (ZavaBankDB) as its only persistence layer. Two tables are created at startup by KycBootstrapServlet: KYCWatchlist (stores sanctioned or flagged names with their source, e.g. OFAC) and KYCVerification (stores the full audit trail of every customer verification attempt including status, reason, and timestamp). Database connection parameters (host, port, database name, user, password) are read from kyc.properties with environment-variable overrides (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD). There are no external APIs, message brokers, caches, or other third-party services.

Key Architectural Decisions

  • No framework; plain Servlet API: The application deliberately avoids Spring or Jakarta EE CDI — all components are plain HttpServlet subclasses wired via web.xml, keeping the footprint minimal but lacking DI, AOP, or transaction management.
  • Direct JDBC with manual resource management: Database access uses raw DriverManager.getConnection and manually closed Connection/PreparedStatement/ResultSet objects; there is no connection pool (e.g. HikariCP) or ORM layer.
  • Environment-variable-first configuration: KycConfig reads twelve-factor-style environment variables before falling back to the bundled kyc.properties file, enabling container-friendly configuration injection.

DTOs & Contracts

The application defines no explicit DTO or model classes. JSON request and response bodies are constructed inline using org.json.JSONObject objects directly within each servlet handler:

  • KYC Verify Request (POST /api/kyc/verify body): an ad-hoc JSON object containing customerId (int), fullName (string), and optionally idNumber (string). Parsed via JSONObject.optInt / JSONObject.optString.
  • KYC Verify Response: an inline JSONObject with fields customerId, status (PASS or REVIEW), and reason (explanation string).
  • KYC Status Response (GET /api/kyc/status/{customerId}): an inline JSONObject with fields customerId, status, reason, fullName, idNumber, and verifiedDate.
  • Error Response (all endpoints): a minimal inline JSON object {"status":"ERROR","message":"..."}.

There are no OpenAPI/Swagger specifications, no protobuf schemas, no GraphQL schemas, and no Jackson or custom serializer configuration — all serialization is performed manually through the org.json library. None of the data structures are immutable (no records, no Lombok @Value).

Communication Patterns

Synchronous HTTP only: All client interactions are standard synchronous HTTP request/response pairs. There is no inter-service communication — ZavaKYCService is a standalone monolith that does not call any upstream or peer services.

No asynchronous messaging: No message queues, event buses, Kafka topics, or pub/sub patterns are present.

No resilience patterns: There are no circuit breakers (Resilience4j, Hystrix), retry policies, timeouts, or bulkhead configurations. If the SQL Server database is unavailable, all requests that hit the database will receive a 500 Internal Server Error immediately.

No service discovery or API gateway: The service registers with no discovery mechanism. Callers must know the host and port directly. No API gateway, reverse proxy configuration, or load-balancer sidecar is defined in the project source.

Startup dependency chain: KycBootstrapServlet is declared with <load-on-startup>1</load-on-startup> in web.xml, meaning it runs before the application accepts traffic and will fail the entire deployment if the database is unreachable at startup. There is no readiness probe or retry loop in the bootstrap logic.

Security posture: No authentication, authorization, or TLS is configured at any level. All four endpoints — including /internal/bootstrap and all /api/kyc/* endpoints — are publicly accessible with no token validation, no Basic Auth, no role checks, and no HTTPS enforcement. Sensitive customer PII (full name, ID number) is transmitted over plain HTTP.

ZavaKYCService implements a financial compliance domain that screens customer identities against a watchlist of sanctioned individuals and maintains an immutable audit trail of every verification decision for regulatory purposes.

ZavaKYCService is a single-bounded-context monolith. All domain logic, data ownership, and persistence reside within one deployable unit. There is no cross-service communication.

Primary Workflows

Workflow 1: Application Bootstrap and Schema Initialization

Triggered once at application startup via KycBootstrapServlet (load-on-startup=1). This workflow runs before any HTTP traffic is accepted.

Steps:

  1. Servlet container (Tomcat) loads KycBootstrapServlet during deployment.
  2. Servlet opens a JDBC connection to SQL Server.
  3. Executes DDL: creates KYCWatchlist table if it does not already exist.
  4. Executes DDL: creates KYCVerification table if it does not already exist.
  5. Seeds one initial watchlist entry (John Alias, source OFAC-SAMPLE) if the KYCWatchlist table is empty.
  6. Closes the connection and returns control to Tomcat.

If any step fails (e.g., SQL Server is unreachable), the servlet throws a ServletException, the WAR deployment fails, and the application becomes unavailable.

Workflow 2: Customer KYC Verification

Initiated by an upstream service or operator submitting a POST /api/kyc/verify request with the customer's identity information.

Steps:

  1. Receive JSON body containing customerId, fullName, and optionally idNumber.
  2. Input validation: reject with HTTP 400 if the JSON is malformed, customerId is not a positive integer, or fullName is blank.
  3. Open a JDBC connection to SQL Server.
  4. Watchlist check: query KYCWatchlist for a case-insensitive exact match on FullName.
  5. Determination:
    • If a match is found → status = REVIEW, reason = "Name matched watchlist."
    • If no match → status = PASS, reason = "No watchlist match."
  1. Record persistence: insert a KYCVerification row with CustomerID, FullName, IdNumber, determined Status, Reason, and the current timestamp.
  2. Return JSON response: {customerId, status, reason}.

Workflow 3: KYC Status Lookup

Initiated by an upstream system querying the latest verification outcome for a previously submitted customer, via GET /api/kyc/status/{customerId}.

Steps:

  1. Extract customerId from the URL path.
  2. Input validation: reject with HTTP 400 if the path segment is missing or not a valid integer.
  3. Open a JDBC connection to SQL Server.
  4. Query KYCVerification for the most recent record for the given CustomerID (ordered by VerifiedDate DESC, then VerificationID DESC as a tie-breaker).
  5. If no record exists → return HTTP 404 {customerId, status: "NOT_FOUND"}.
  6. If a record exists → return HTTP 200 with full verification details: {customerId, status, reason, fullName, idNumber, verifiedDate}.

Workflow 4: Health Check

Initiated by a load balancer or monitoring system via GET /health.

Returns a static HTML response (ZavaKYCService - Online) with HTTP 200. No business logic or database interaction occurs.

Cross-Service Data Flows

ZavaKYCService is a self-contained service with no outbound calls to other services. There are no aggregation flows, no gateway composition patterns, and no circuit breaker fallbacks involving external services.

The only external dependency is the upstream caller, which is responsible for:

  • Supplying a valid CustomerID that corresponds to a real customer in the upstream customer-management system.
  • Interpreting the returned status field (PASS or REVIEW) to determine whether to proceed with customer onboarding or route the case for manual review.

The service does not validate that CustomerID exists in any upstream system — it trusts the caller.

State Transitions

KYC verification results are immutable once written. There is no update or delete operation on KYCVerification. A customer may be verified multiple times; the status lookup always returns the most recent result, but all historical records are retained.


Customer Submitted → [watchlist check] → PASS or REVIEW
                                          (record persisted, cannot be modified)

Business Constraints

  • No partial name matching: Watchlist screening uses exact equality (UPPER(FullName) = UPPER(?)) — a customer named "John Smith" will not match a watchlist entry of "John A. Smith".
  • No identity-document screening: idNumber is stored in KYCVerification but is never checked against a watchlist or external identity verification service.
  • Single-check per request: Each POST /api/kyc/verify call performs one watchlist check and records one outcome. There is no aggregation of multiple identity signals.
  • No re-verification logic: The service does not check whether a customer has already passed KYC; every call creates a new verification record.

Cross-Cutting Concerns

  • Transactions: No transaction management is declared. Each database operation uses its own Connection object. If insertVerification fails after existsInWatchlist succeeds, the verification record is lost but no rollback occurs. The two operations are not atomic.
  • Error handling: All SQLException exceptions are caught in servlet handlers and return HTTP 500 with a generic error message; the original exception is not exposed to the caller. No dead-letter queue or compensating action exists.
  • Audit trail: KYCVerification serves as the audit log — every verification attempt is persisted with a timestamp. However, there is no separate audit event log, no change tracking, and no mechanism to record who initiated the verification request.
  • Authorization: No authentication or authorization is enforced. Any caller that can reach the service's HTTP port can submit verifications or retrieve status for any customer ID.

Version & Compatibility Risks

The org.json:json:20140107 dependency is a 2014 build, over a decade old, and predates the introduction of semantic versioning in the library. Newer releases (e.g., 20240303) include bug fixes and security improvements; continuing to use the 2014 build carries potential JSON parsing edge-case risks and limits access to modern API improvements. The javax.servlet-api:4.0.1 dependency targets the Java EE 8 namespace (javax.); migration to Jakarta EE 9+ (which uses the jakarta. namespace) would require a namespace refactor across all servlet classes and the web.xml. The mssql-jdbc:12.6.3.jre8 driver is a recent release and poses no immediate compatibility risk, but the jre8 classifier should be replaced with jre11 or jre17 if the JDK target is ever upgraded.

Notable Observations

  • No logging framework declared: The application performs no structured logging; adding SLF4J with Logback or Log4j2 is advisable for observability, especially in a containerised deployment.
  • No connection pooling library: Database connections are created via DriverManager.getConnection on every request. Under moderate load this will exhaust JDBC connections; HikariCP or c3p0 should be added.
  • No security or authentication library: There is no dependency for authentication, authorization, or input sanitization, meaning all security concerns must be handled manually in servlet code.
  • Minimal dependency footprint: With only 3 declared dependencies the application is very lightweight, but this also means it reimplements capabilities (connection pooling, logging, validation) that established libraries provide safely.

Caching Strategy

No caching layer is configured. All read operations execute a direct SQL query against SQL Server on every request. There is no in-memory cache, no Redis, no second-level cache (Hibernate is not used), and no Spring Cache annotations. Because KYCWatchlist data changes infrequently relative to how often it is read (every verification request), introducing a short-lived in-memory or distributed cache for watchlist lookups would reduce database load significantly.

Data Ownership Boundaries

ZavaKYCService is a self-contained single-service application with one isolated SQL Server database. There is no shared database or cross-service data access — no other service reads from or writes to ZavaBankDB through this codebase. CustomerID in KYCVerification is an application-level foreign key referencing an upstream customer identity system, but there is no declared database constraint or API call to validate its existence; the calling service is responsible for providing a valid customer identifier.

All database connections are opened per-request using DriverManager.getConnection with no connection pool. Each servlet opens a single connection, executes its query, and closes the connection in a finally block. Under concurrent load this pattern will create a large number of short-lived TCP connections and may exhaust the SQL Server connection limit.

Summary: Both tables store PII. KYCVerification additionally stores customer identity numbers (IdNumber), which may qualify as government-issued identifier data requiring heightened protection under GDPR, PCI-DSS, or local financial regulation. No encryption-at-rest, no data masking, no field-level access control, and no audit trail mechanism is currently implemented. Database credentials are stored in plain text in kyc.properties and the JDBC URL disables TLS (encrypt=false), meaning PII is transmitted unencrypted between the application and the database.
