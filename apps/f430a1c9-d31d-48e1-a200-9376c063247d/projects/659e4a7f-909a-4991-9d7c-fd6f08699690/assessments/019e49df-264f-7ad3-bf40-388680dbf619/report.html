Data Storage & External Services

The application uses a single SQL Server database for both reference data (KYCWatchlist) and transactional verification history (KYCVerification). There are no outbound API integrations, caches, or message brokers; the only external dependency is the database connection configured through kyc.properties defaults or DB_* environment variables.

Key Architectural Decisions

  • The service is packaged as a traditional servlet WAR and relies on web.xml instead of annotation-driven routing.
  • Data access is implemented with direct JDBC calls inside servlets rather than repository or dependency-injection abstractions.
  • Schema bootstrap and seed data creation happen in KycBootstrapServlet.init(), which makes database availability a startup prerequisite.

DTOs & Contracts

The API contracts are dynamic rather than strongly typed. KycVerifyServlet parses the request body into an org.json.JSONObject, so the effective request contract is an untyped JSON object that must contain customerId and fullName, with idNumber treated as optional enrichment data. Both servlet responses are also emitted as ad-hoc JSON objects, which means there are no dedicated request or response DTO classes, no immutable records, and no schema-first assets such as OpenAPI, Swagger, GraphQL, or protobuf definitions in the repository.

Communication Patterns

Communication is fully synchronous. Clients call servlet endpoints over HTTP, each servlet opens a JDBC connection through KycConnectionFactory, and SQL Server queries or inserts are executed inline before a response is returned. There are no asynchronous messaging patterns, no retry or circuit-breaker libraries, no service discovery, and no gateway aggregation. API availability depends on SQL Server being reachable during startup because KycBootstrapServlet creates and seeds tables in init(). No authentication, authorization, or TLS configuration is implemented in the application; all mapped endpoints are publicly reachable at the servlet layer unless external infrastructure adds protection.

Core Business Workflows

ZavaKYCService supports customer identity screening by checking submitted names against a local watchlist and recording the resulting decision for later retrieval. Its business behavior is centered on verification, audit-style persistence, and startup initialization of reference data.

Primary Workflows

Workflow 1: Perform KYC verification

  1. A client submits POST /api/kyc/verify with a JSON payload.
  2. KycVerifyServlet validates that the body is valid JSON and that customerId and fullName are present.
  3. The service queries KYCWatchlist using a case-insensitive full-name comparison.
  4. If a match is found, the verification is marked REVIEW; otherwise it is marked PASS.
  5. The decision and explanatory reason are stored in KYCVerification with the current timestamp.
  6. The client receives a compact JSON response containing the customer ID, decision, and reason.

Workflow 2: Retrieve latest KYC status

  1. A client calls GET /api/kyc/status/{customerId}.
  2. KycStatusServlet validates that the path parameter exists and is numeric.
  3. The service queries the latest verification record ordered by VerifiedDate and VerificationID.
  4. If no record exists, the client receives a NOT_FOUND response; otherwise the latest decision payload is returned.

Workflow 3: Bootstrap reference data on startup

  1. Tomcat loads the application and triggers KycBootstrapServlet.init().
  2. The servlet ensures the KYCWatchlist and KYCVerification tables exist.
  3. If the watchlist is empty, a sample record for John Alias is inserted.
  4. API availability depends on this initialization completing successfully.

Cross-Service Data Flows

There is no service-to-service composition inside this repository. All business data flow is between an external client and this service, with SQL Server as the persistence boundary. The only cross-context identifier is customerId, which originates outside the service and is stored so callers can later retrieve the most recent decision. When the database is unavailable, the application does not degrade gracefully: verification and status workflows fail with HTTP 500 responses rather than fallback data.

Business Rules & Decision Logic

  • Validation rules: The verification workflow rejects invalid JSON, missing customerId, missing fullName, and non-numeric status path parameters with 400 responses.
  • Decision logic: A case-insensitive exact full-name match in KYCWatchlist forces the result to REVIEW; otherwise the result is PASS.
  • State transitions: Each verification request creates a new immutable history row rather than updating a previous record, and status lookup returns the latest row by timestamp and identity ordering.
  • Business constraints: customerId must be positive for verification, and only the most recent decision is returned for status requests.
  • Transactions and error handling: JDBC operations execute without explicit transaction management or retry logic; SQL exceptions surface as 500 responses.
  • Authorization: No business-level authorization checks are implemented in the service.

Caching Strategy

No caching layer is implemented. Every verification request and status lookup opens a direct JDBC connection and queries SQL Server in real time, so the service follows a simple read-through-from-database pattern with no in-memory or distributed cache regions.

Data Ownership Boundaries

The application owns a single logical persistence boundary: one SQL Server database used exclusively by this service. There are no additional modules or downstream services that share direct table access. customerId acts as a reference to an upstream customer record managed elsewhere, but the service only stores it as an integer identifier; there is no local foreign-key relationship or cross-service query mechanism beyond returning that identifier in API responses.
