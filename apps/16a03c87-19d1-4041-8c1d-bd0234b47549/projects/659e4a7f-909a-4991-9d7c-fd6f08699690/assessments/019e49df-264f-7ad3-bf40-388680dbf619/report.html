Data Storage & External Services

The application stores primary customer, account, account type, and fallback transaction data in a SQL Server database named ZavaBankDB. Authentication is delegated to an external ZavaAuthGateway via configured login/logout URLs, while live balance and transaction history are retrieved from the Zava Ledger service over HTTP and parsed as XML.

Key Architectural Decisions

  • Uses a page-centric WebForms design where UI, orchestration, and persistence logic are colocated in code-behind classes instead of separate service/repository layers.
  • Keeps account master data in SQL Server but prefers the ledger service for balance and transaction reads, falling back to SQL for transactions when the remote call fails.
  • Relies on ASP.NET Forms Authentication for local session state while delegating credential capture to an external authentication gateway.

DTOs & Contracts

The repository does not define explicit API DTO classes or OpenAPI specifications. Contract shapes are implicit:

  • ReturnUrl query string acts as the request contract for the login page and auth redirect flow.
  • WebForms controls (GridView, FormView, TextBox, DropDownList) act as the request/response contract for account management postbacks.
  • Ledger responses are XML documents parsed with XmlDocument; effective response concepts include a balance document and a repeating transaction item document.
  • SQL result sets materialized into DataTable objects act as service-level read models for customers, accounts, account types, and fallback transactions.
  • No immutable record types, protobuf schemas, GraphQL schemas, or Swagger/OpenAPI descriptions were found.

Communication Patterns

The application uses synchronous communication only. Browser requests target WebForms pages over HTTP, login/logout are delegated by redirecting to configured auth-gateway URLs, and live account balance plus transaction history are fetched synchronously from the ledger service using HttpWebRequest. When the transactions API call fails, the page falls back to querying the local SQL database for the most recent transactions. No asynchronous messaging, retry library, circuit breaker, service discovery, or API gateway framework is configured. Security posture is limited to ASP.NET Forms Authentication for page access control; no TLS, token-based authentication, or fine-grained authorization rules are defined in this repository.

ZavaAccountManager supports bank employees who need to inspect customer accounts, open new accounts, close existing accounts, and review balance or transaction activity. The application also brokers sign-in and sign-out through an external authentication gateway.

Primary Workflows

Workflow 1: Employee signs in and opens the account console

  1. An employee requests Default.aspx.
  2. If the browser is not authenticated, the app redirects to Login.aspx and includes the current page as ReturnUrl.
  3. Login.aspx renders a link to the external auth gateway using the configured login URL.
  4. After authentication, the employee returns to the application.
  5. The default page loads customers and account types so the employee can begin account work.

Business rules involved: anonymous users are denied on protected pages, login/logout pages are explicitly allowed, and the auth gateway owns credential capture.

Workflow 2: Employee reviews customer accounts and balance activity

  1. The employee selects a customer from the customer list.
  2. The page stores the selected customer ID and loads that customer's accounts from SQL Server.
  3. The employee chooses either Balance or Transactions for an account.
  4. For balance, the app queries the ledger service and shows current and available balance values.
  5. For transactions, the app first calls the ledger service; if that fails, it loads the latest transaction rows from SQL Server.

Business rules involved: account selection is required before account actions, ledger data is preferred as the freshest source, and transaction history has a degraded local fallback.

Workflow 3: Employee opens or closes an account

  1. To open an account, the employee submits customer ID, account type, and opening balance.
  2. The app validates that customer ID, account type, and balance are parseable numeric values.
  3. The app inserts a new account row with generated account number, active status, and matching opening and available balances.
  4. To close an account, the employee triggers the close action for a selected account row.
  5. The app updates the account status to Closed and stamps close and modified dates.

Business rules involved: invalid numeric inputs cancel account opening, new accounts start active, and closing an account is an immediate status transition applied in the database.

Cross-Service Data Flows

The primary cross-service flow is account activity composition on the default page. ZavaAccountManager reads core customer and account records from SQL Server, then enriches the selected account view with live balance and transaction data from the ledger service. If the transaction service path is unavailable, the business outcome degrades gracefully by showing locally stored transaction history instead of failing the page outright. Authentication is similarly delegated: the app owns the employee console session, but the external auth gateway owns the actual sign-in and sign-out experience.

Business Rules & Decision Logic

  • Authentication rule: all pages except login and logout require an authenticated employee session.
  • Input validation rule: opening an account succeeds only when customer ID, account type, and opening balance can all be parsed into numeric values.
  • Account lifecycle rule: new accounts are inserted as Active; closing an account transitions its status to Closed and stamps close/modified timestamps.
  • Data freshness rule: ledger service data is the preferred source for balances and transactions; transaction history alone has a SQL fallback path.
  • Authorization model: repository configuration enforces only authenticated-vs-anonymous access. No finer employee role checks are defined in the checked-in code.
  • Transaction handling: each page action executes its own SQL commands directly; no broader unit-of-work or saga orchestration pattern is present.
  • Error handling: failures in balance lookup return a user-facing “Ledger balance unavailable” message, while failures in transaction lookup fall back to local SQL data.
