Data Storage & External Services

The application uses a SQL Server database (ZavaBankDB) as its primary persistent store via ADO.NET queries. It also depends on a ledger HTTP API for account balances and transactions and an authentication gateway for login and logout redirects.

Key Architectural Decisions

  • Uses a single WebForms monolith with code-behind handlers for UI and business logic.
  • Uses direct SQL statements through SqlClient instead of an ORM layer.
  • Falls back to local transaction SQL queries when ledger transaction API calls fail.

DTOs & Contracts

The application uses server-side WebForms controls and DataTable objects as internal transport contracts rather than explicit API DTO classes. External contract payloads are XML documents from the ledger service parsed with XmlDocument, and query parameters such as ReturnUrl are passed between pages and the auth gateway. No OpenAPI or protobuf contract artifacts were detected.

Communication Patterns

Communication is synchronous HTTP. ZavaAccountManager calls the ledger service with HttpWebRequest for balance and transaction retrieval. If transaction API calls fail, the app falls back to local SQL transaction queries. Service discovery is based on static configuration values in web.config (LedgerBaseUrl, AuthGatewayLoginUrl, AuthGatewayLogoutUrl). No retry or circuit-breaker policy library is configured. Security posture is forms-auth-based session protection within the app; no explicit TLS enforcement is configured in this repository.

Core Business Workflows

ZavaAccountManager supports branch operations for viewing customer accounts, opening and closing accounts, and checking balance and transaction history. Workflows are driven by authenticated user actions in WebForms pages.

Primary Workflows

Workflow 1: Customer account review

  1. Authenticated user opens Default.aspx.
  2. Application loads customer list and account type options.
  3. User selects a customer and account list is queried.
  4. User can request current balance from ledger API.
  5. User can request transactions from ledger API, with SQL fallback if ledger is unavailable.

Workflow 2: Open new account

  1. User enters customer id, account type, and opening balance in form view.
  2. Application validates numeric input fields.
  3. Application generates an account number and inserts account row with active status.
  4. Account list is refreshed and status message confirms account opening.

Workflow 3: Close account

  1. User triggers close action on an account row.
  2. Application updates status to Closed and sets close and modified timestamps.
  3. Account list refreshes and status message confirms closure.

Cross-Service Data Flows

The main cross-service composition occurs when account balances and transactions are requested. ZavaAccountManager retrieves account context from SQL Server, then enriches displayed details by calling the ledger service. If the ledger transaction endpoint fails, the business fallback returns historical transactions from local SQL data so users can still continue account review.

Business Rules & Decision Logic

  • Access control rule: unauthenticated users are redirected to login flow before account actions.
  • Input validation rule: account opening requires valid integer customer id, valid account type id, and decimal opening balance.
  • State transition rule: account close action sets account status to Closed and writes close and modified dates.
  • Fallback decision rule: if ledger transaction retrieval fails, the application serves local SQL transactions instead.
