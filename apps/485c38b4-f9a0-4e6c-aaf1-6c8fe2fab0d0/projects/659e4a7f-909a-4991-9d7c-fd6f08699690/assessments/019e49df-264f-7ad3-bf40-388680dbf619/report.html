Data Storage & External Services

The service uses a single SQL Server database (ZavaBankDB) as its only data store. Connection details are resolved at runtime: environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) take precedence over the web.config connection string (Server=sqlserver,1433). The database holds at minimum two tables — CurrencyPairs (supported currency pairs and their active status) and ExchangeRates (mid-rates, source, effective/expiry dates). There are no external HTTP APIs, message brokers, or caching layers.

Key Architectural Decisions

  • WCF / SOAP over basicHttpBinding: The service exposes a classic SOAP interface defined by ICurrencyService, making it interoperable with legacy .NET and Java SOAP clients without REST.
  • Direct ADO.NET data access: Data is read with raw SqlConnection/SqlCommand calls and parameterized queries — no ORM or repository abstraction layer is used.
  • Environment-variable-first configuration: DbConfig checks environment variables before falling back to web.config, enabling container-based deployment without config file changes.

DTOs & Contracts

CurrencyRateData is the sole response contract. It is a WCF [DataContract] class decorated with ordered [DataMember] attributes, used as the response type for GetRate. It is a plain C# class (not a C# record) and is therefore mutable; no immutability guarantees are enforced. Field details are described in data-architecture.md.

No request body DTOs exist — all inputs are primitive parameters (string, decimal) passed directly in the SOAP envelope.

There is no OpenAPI/Swagger, Protobuf (.proto), or GraphQL schema. The only formal API description is the auto-generated WSDL at /CurrencyService.svc?wsdl, produced by WCF's serviceMetadata httpGetEnabled="true" behavior. Serialization is handled entirely by WCF's built-in System.Runtime.Serialization stack (DataContractSerializer).

Communication Patterns

Synchronous only: All service operations are synchronous request/response over SOAP (HTTP POST). There is no asynchronous messaging (no Kafka, RabbitMQ, Azure Service Bus, or similar).

No gateway or service discovery: The service is a standalone deployment with no API gateway, client-side load balancing, Eureka/Consul registration, or Kubernetes service mesh. Clients must address it directly by hostname/port (xsp4 on port 8080).

No resilience policies: No circuit breaker, retry, timeout configuration, or bulkhead patterns are implemented at the service level. Fault propagation is handled only via WCF FaultException thrown on invalid input or missing data.

Security posture: No authentication, authorization, or TLS is configured. All three SOAP operations and the health endpoint are publicly accessible with no credentials required. The web.config <authorization> block only explicitly allows anonymous access to /health; the WCF endpoint has no <serviceAuthorization> policy and no transport-level security (<security mode="None"> is the default for basicHttpBinding). Clients communicating with this service in production should be protected by a network boundary or a terminating proxy/gateway that enforces TLS and authentication.

Core Business Workflows

ZavaCurrencyService provides foreign exchange (FX) rate lookups and currency conversion for the Zava Bank platform. It serves as the authoritative source of exchange rate data, exposing operations to retrieve current rates, list supported currencies, and convert monetary amounts between currency pairs.

Domain Entities

EntityService / Bounded ContextDescriptionKey Relationships
CurrencyPairFX Rate ManagementRepresents a tradeable pair of currencies (base and quote), with an active/inactive flagHas zero or more ExchangeRates
ExchangeRateFX Rate ManagementA time-bounded mid-rate for a specific currency pair, with source attribution and effective/expiry datesBelongs to one CurrencyPair
CurrencyRateDataFX Rate Management (API boundary)A projection DTO returned to callers — combines pair identity with the most recent applicable rateDerived from CurrencyPair + ExchangeRate

Service-to-Domain Mapping

ServiceDomain ContextOwned EntitiesExternal Dependencies
ZavaCurrencyServiceFX Rate ManagementCurrencyPair, ExchangeRate (read-only consumer)SQL Server (ZavaBankDB) — populated by an external process

There is a single bounded context. The service does not own the write path for exchange rates — it reads rates that are assumed to be populated by an external market data feed or administrative process (not present in this repository).

Primary Workflows

Workflow 1: Look Up an Exchange Rate

A client (another Zava Bank service or integration) requests the current mid-rate for a specific currency pair (e.g., USD → EUR).

Steps:

  1. Client sends a GetRate SOAP request with fromCurrency and toCurrency.
  2. Both currency codes are validated (must be exactly 3 non-whitespace characters) and normalized to uppercase.
  3. The service queries CurrencyPairs joined with ExchangeRates to find the most recent active, non-expired rate for the pair.
  4. If a valid rate is found, a CurrencyRateData response is returned with the rate, source label, and effective date.
  5. If no rate is found, a SOAP fault is returned to the caller.

Business rules applied: Currency code format validation, currency normalization, active-pair filter, expiry-date filter, most-recent-rate selection.

Workflow 2: List Supported Currencies

A client requests the set of all currency codes for which at least one active trading pair exists.

Steps:

  1. Client sends a GetSupportedCurrencies SOAP request.
  2. The service queries CurrencyPairs for all distinct BaseCurrency and QuoteCurrency codes where IsActive = 1.
  3. The union of both columns is returned as a sorted string[].

Business rules applied: Active-pair filter only — no input validation required.

Workflow 3: Convert a Monetary Amount

A client provides an amount and a source/target currency and receives the converted amount.

Steps:

  1. Client sends a ConvertAmount SOAP request with amount, fromCurrency, and toCurrency.
  2. The amount is validated to be non-negative (≥ 0).
  3. The service internally invokes Workflow 1 (GetRate) to retrieve the applicable mid-rate.
  4. The converted amount is calculated as Round(amount × rate, 4, MidpointRounding.AwayFromZero).
  5. The result is returned as a decimal.

Business rules applied: Non-negative amount check, all rules from Workflow 1 (currency validation, rate lookup).

Cross-Service Data Flows

ZavaCurrencyService is a leaf service with no outbound service-to-service calls. All data flows are inbound (client → service → database).

The exchange rate data in ZavaBankDB is assumed to be populated by an upstream market data pipeline or administrative service that is not part of this repository. ZavaCurrencyService acts purely as a read-only query facade over that data; there is no feedback loop or event-driven refresh from this service.

No gateway aggregation, circuit breaker fallback, or cross-service composition is present. If the database is unavailable, all three SOAP operations will return a SOAP fault with the SQL exception detail (because includeExceptionDetailInFaults="true" is currently configured).

State Transitions

There are no mutable entity state transitions in this service — all operations are read-only. The IsActive flag on CurrencyPairs and the ExpiryDate on ExchangeRates represent state, but they are managed externally.

Transactions

No explicit transaction management is configured. Each operation opens a single SqlConnection, executes one SqlCommand, and disposes the connection via using blocks. The read-only nature of all operations makes transaction isolation a non-issue for this service.

Error Handling

All business rule violations and missing-data conditions surface as WCF FaultException instances, which are serialized as SOAP faults and returned to the client. Because includeExceptionDetailInFaults="true" is set in web.config, SQL-level exceptions also propagate their full detail to callers — a security concern in production.

Authorization

No authentication or authorization is implemented. All operations are publicly accessible. See api-service-contracts.md for the security posture assessment.

Version & Compatibility Risks

All dependencies are BCL assemblies bundled with .NET Framework 4.8, which is the final feature release of the .NET Framework line. Microsoft has placed .NET Framework 4.8 in maintenance-only mode — it receives security patches but no new features. The WCF server stack (System.ServiceModel host-side) is not available in .NET / .NET Core; migrating to modern .NET would require replacing the SOAP endpoint with ASP.NET Core (e.g., gRPC or REST via Minimal APIs / CoreWCF). System.Data.SqlClient shipped in-box with .NET Framework 4.8 is the legacy SQL Server driver; Microsoft.Data.SqlClient is its modern, actively maintained successor.

Notable Observations

  • Zero third-party NuGet packages: The project relies exclusively on in-box .NET Framework assemblies, which simplifies the dependency surface but means all functionality is tied to the .NET Framework 4.8 runtime.
  • WCF server-side hosting is the primary migration blocker: System.ServiceModel (WCF) is available on .NET / .NET Core only as a client library or via the community-maintained CoreWCF package — the built-in ASP.NET hosting of .svc files does not exist outside .NET Framework.
  • Direct ADO.NET without ORM: Using raw SqlConnection/SqlCommand reduces dependencies but means there is no abstraction layer to ease a potential database driver swap (e.g., from System.Data.SqlClient to Microsoft.Data.SqlClient).
  • No logging, caching, or observability libraries: The service has no structured logging, distributed tracing, or health metrics beyond the basic /health endpoint, which will need to be addressed during modernization.

Entity Model

CurrencyRateData (defined in CurrencyRateData.cs) is a WCF [DataContract] DTO returned by GetRate. It is populated from query results and does not map to a single table — it is a projection across CurrencyPairs and ExchangeRates. It is a mutable C# class with ordered [DataMember] attributes for SOAP serialization.

Key Repository Methods

There is no repository interface layer. Data access is implemented directly in CurrencyService.cs using inline ADO.NET (SqlConnection / SqlCommand).

ServiceMethodSQL PatternPurpose
CurrencyServiceGetRate(fromCurrency, toCurrency)SELECT TOP 1 ... FROM CurrencyPairs JOIN ExchangeRates WHERE BaseCurrency=@From AND QuoteCurrency=@To AND IsActive=1 AND (ExpiryDate IS NULL OR ExpiryDate > GETDATE()) ORDER BY EffectiveDate DESCRetrieves the most recent active exchange rate for a currency pair
CurrencyServiceGetSupportedCurrencies()SELECT DISTINCT BaseCurrency UNION SELECT DISTINCT QuoteCurrency FROM CurrencyPairs WHERE IsActive=1 ORDER BY CurrencyCodeReturns the full list of currencies that appear in at least one active pair
CurrencyServiceConvertAmount(amount, fromCurrency, toCurrency)Calls GetRate internally; no additional SQLMultiplies amount by the retrieved rate with 4 decimal place rounding

All queries use AddWithValue parameterization to prevent SQL injection. Connections are opened per-request inside using blocks (no connection pooling configuration beyond ADO.NET defaults). There is no transaction management — all operations are single-statement reads.

Caching Strategy

No caching layer is implemented. Every call to GetRate or GetSupportedCurrencies opens a new database connection and executes a live SQL query. There is no in-memory cache, Redis, session cache, or second-level query cache. For a service that serves relatively static reference data (exchange rates with explicit EffectiveDate/ExpiryDate), the absence of caching is a potential performance concern for high-throughput scenarios.

Data Ownership Boundaries

ZavaCurrencyService is a single-service, single-database deployment. It does not share its SQL Server instance with other services at the application level (though the Zava Bank platform uses a shared ZavaBankDB SQL Server instance as implied by the default connection string Server=sqlserver,1433;Database=ZavaBankDB).

All data access is read-only. The service performs no writes; it acts purely as a query service over the CurrencyPairs and ExchangeRates tables, which are presumably populated by a separate process or service. There are no CQRS, event sourcing, or outbox patterns.

Cross-service data access: not applicable — this service does not call any other service's API or database.
