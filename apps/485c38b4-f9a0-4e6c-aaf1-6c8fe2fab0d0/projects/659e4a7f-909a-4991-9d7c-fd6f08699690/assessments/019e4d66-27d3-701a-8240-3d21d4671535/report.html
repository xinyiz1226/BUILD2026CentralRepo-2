Data Storage & External Services

The application relies exclusively on a SQL Server database (ZavaBankDB) accessed via ADO.NET without any ORM. Two key tables are involved: CurrencyPairs (mapping base/quote currency pairs with an active flag) and ExchangeRates (mid-rates with effective and expiry dates, linked by PairID). There are no caches, message brokers, or additional external APIs — all data is fetched in real time from SQL Server on each request. Database connectivity is resolved through environment variables at runtime, falling back to the web.config connection string.

Key Architectural Decisions

  • WCF with basicHttpBinding: The service exposes a SOAP endpoint using basicHttpBinding, making it consumable by any SOAP-capable client; service metadata (WSDL) is exposed via httpGetEnabled=true.
  • Direct ADO.NET data access: No ORM or repository abstraction is used; raw SqlCommand queries are executed inline within the service implementation, coupling business logic and data access in CurrencyService.
  • Environment-variable-first configuration: DbConfig prefers environment variables for the database connection string over web.config, enabling containerized deployment (see Dockerfile).

DTOs & Contracts

CurrencyRateData is the sole DTO in the service. It is decorated with [DataContract] and its fields with [DataMember(Order = N)], which governs SOAP/XML serialization order. It is used exclusively as the response type for the GetRate operation. ConvertAmount returns a primitive decimal and GetSupportedCurrencies returns string[] — neither requires a dedicated DTO class.

CurrencyRateData is a mutable class (not a record or readonly struct). Full field details are documented in data-architecture.md.

There is no OpenAPI/Swagger specification, no .proto protobuf schema, and no GraphQL schema. The authoritative contract is the WSDL served at CurrencyService.svc?wsdl.

Communication Patterns

Synchronous only. All client interactions are synchronous request-response over SOAP/HTTP. Clients connect directly to the .svc endpoint — there is no API gateway, service mesh, or load balancer defined within this repository.

No resilience patterns. There is no circuit breaker, retry policy, timeout configuration, or bulkhead pattern implemented in the service code (e.g., no Polly usage). Database calls use a plain SqlConnection without any timeout override.

No service discovery. Clients must address the service by a hardcoded or externally configured URL. There is no Eureka, Consul, Kubernetes DNS–based discovery, or client-side load balancing configured.

Security posture: No authentication or authorization is configured at the WCF transport or message layer. The health endpoint is explicitly allowed for all users (<allow users="*" />), and the WCF service does not define any <serviceCredentials> or security behavior. Transport security (HTTPS/TLS) is not configured in web.config — the binding uses plain HTTP. All SOAP operations and the WSDL metadata endpoint are publicly accessible with no authentication or authorization checks.

Startup order: The service has no startup dependency chain management. If SQL Server is unavailable at request time, operations throw a FaultException without retry.

Core Business Workflows

ZavaCurrencyService enables financial applications and clients at Zava Bank to retrieve live exchange rates, list supported currencies, and convert monetary amounts between currency pairs stored in the bank's internal SQL Server database.

Domain Entities

CurrencyPairsCurrency Rate ManagementRepresents an active or inactive tradeable pair of currencies (e.g., USD/EUR) identified by a base and quote currency codeHas many ExchangeRates; active pairs (IsActive=1) are the authoritative set of supported conversions

Service-to-Domain Mapping

ServiceDomain ContextOwned EntitiesExternal Dependencies
ZavaCurrencyServiceCurrency Rate ManagementCurrencyPairs, ExchangeRates (read-only access)SQL Server (ZavaBankDB) — all data originates from this database; no downstream services

This is a single-service application. There are no microservice boundaries, no event-driven cross-service flows, and no gateway aggregation layer.

Primary Workflows

Workflow 1: Exchange Rate Lookup (GetRate)

A client requests the current mid-rate for a currency pair (e.g., USD → EUR).

Steps:

  1. Client sends a SOAP request with fromCurrency and toCurrency string values.
  2. The service normalises both codes (trims whitespace, uppercases) and validates that each is exactly 3 characters.
  3. A parameterised SQL query selects the most recent, non-expired rate from ExchangeRates joined to the active CurrencyPairs record matching the requested pair.
  4. If a row is returned, the service constructs and returns a CurrencyRateData response.
  5. If no row is returned (pair not supported or all rates expired), the service raises a SOAP fault: "No exchange rate found for X/Y.".

Business rules involved: Currency code normalisation, 3-letter code validation, active pair filter (IsActive=1), expiry filter (ExpiryDate IS NULL OR ExpiryDate > GETDATE()), most-recent-first ordering.

Workflow 2: Amount Conversion (ConvertAmount)

A client requests conversion of a monetary amount from one currency to another.

Steps:

  1. Client sends a SOAP request with amount (decimal), fromCurrency, and toCurrency.
  2. The service validates that amount is non-negative; if negative, it raises a SOAP fault: "Amount must be non-negative.".
  3. The service delegates to GetRate to retrieve the current mid-rate (see Workflow 1), inheriting all its validation and database lookup steps.
  4. The converted amount is calculated as amount × rate, rounded to 4 decimal places using MidpointRounding.AwayFromZero.
  5. The rounded result is returned as a plain decimal.

Business rules involved: Non-negative amount constraint, delegated rate lookup (currency validation + rate fetch), 4-decimal-place rounding with banker's rounding avoided (AwayFromZero).

Workflow 3: Supported Currency Discovery (GetSupportedCurrencies)

A client requests the list of all currency codes the service can handle.

Steps:

  1. Client sends a SOAP request with no parameters.
  2. The service queries CurrencyPairs for all distinct currency codes (both BaseCurrency and QuoteCurrency columns) where IsActive=1.
  3. The merged, deduplicated, alphabetically sorted list is returned as string[].

Business rules involved: Active pair filter (IsActive=1), union of base and quote sides, alphabetical ordering.

Workflow 4: Liveness Check (GET /health)

An infrastructure or orchestration system checks whether the service process is alive.

Steps:

  1. An HTTP client sends GET /health.
  2. HealthHandler returns HTTP 200 with body "OK" immediately — no database check is performed.

Note: This is a process liveness check only, not a readiness probe. A healthy response does not guarantee that SQL Server is reachable.

Cross-Service Data Flows

ZavaCurrencyService is a standalone service with no upstream or downstream service dependencies beyond SQL Server. There are no API gateway aggregation patterns, no event-driven flows, no message broker integrations, and no circuit breaker fallback paths to document.

All data originates from the ZavaBankDB SQL Server database. The service has no write paths — it performs read-only queries only. If the database is unavailable, all operations (except /health) fail immediately with a SqlException propagated as a SOAP fault.

Transactions & Error Handling

  • No explicit transactions: Each database call opens its own SqlConnection with using disposal. There are no SqlTransaction or TransactionScope wrappings — operations are single-query reads and do not require atomicity.
  • SOAP faults: Business validation errors are surfaced as WCF FaultException (unmapped fault), which serialises to a SOAP 1.1 Fault envelope. Clients must handle FaultException to distinguish business errors from transport errors.
  • No retry logic: Database failures propagate immediately as unhandled exceptions, resulting in a generic SOAP fault visible to the client.

Authorization

No authentication or authorization is enforced at the business logic layer. All three service operations are publicly accessible to any caller that can reach the SOAP endpoint. See api-service-contracts.md for the security posture assessment.

Version & Compatibility Risks

All dependencies are built-in .NET Framework 4.8 assemblies — there are no third-party NuGet packages to version or patch independently. However, several assemblies represent the most significant migration risks: System.ServiceModel (WCF) and System.Web (classic ASP.NET) are not available in .NET Core, .NET 5+, or .NET 10. Migrating to modern .NET would require replacing WCF with CoreWCF (a community-maintained port), gRPC, or REST APIs, and replacing the System.Web HTTP pipeline with ASP.NET Core middleware. Additionally, System.Configuration's ConfigurationManager is available as a NuGet package (System.Configuration.ConfigurationManager) on .NET Core but the preferred approach is Microsoft.Extensions.Configuration.

Notable Observations

  • No external NuGet packages: The project has an empty packages.config — all dependencies are .NET Framework built-in assemblies. This eliminates NuGet vulnerability surface but also means the entire dependency risk is in the framework itself.
  • WCF lock-in: System.ServiceModel is the core transport; migrating away from .NET Framework 4.8 requires either adopting CoreWCF (SOAP compatibility) or re-architecting the service interface entirely.
  • System.Web dependency: The use of IHttpHandler (Health.ashx) pulls in System.Web, which is one of the most significant blockers for .NET modernization — it cannot be directly ported and must be replaced with ASP.NET Core middleware/minimal API endpoints.
  • Direct ADO.NET without SqlClient NuGet: SQL Server access uses the framework-bundled System.Data.SqlClient. On .NET Core, this should be replaced with the Microsoft.Data.SqlClient NuGet package, which receives active security patches.

Key Repository Methods

There is no repository interface or data-access abstraction layer. All database calls are inline within CurrencyService.cs using SqlConnection / SqlCommand.

ServiceMethodSQL / Purpose
CurrencyService.GetRateSqlCommand (SELECT)SELECT TOP 1 cp.BaseCurrency, cp.QuoteCurrency, er.MidRate, er.Source, er.EffectiveDate FROM CurrencyPairs cp INNER JOIN ExchangeRates er ON er.PairID = cp.PairID WHERE cp.BaseCurrency = @FromCurrency AND cp.QuoteCurrency = @ToCurrency AND cp.IsActive = 1 AND (er.ExpiryDate IS NULL OR er.ExpiryDate > GETDATE()) ORDER BY er.EffectiveDate DESC
CurrencyService.GetSupportedCurrenciesSqlCommand (SELECT DISTINCT)SELECT DISTINCT BaseCurrency / QuoteCurrency AS CurrencyCode FROM CurrencyPairs WHERE IsActive = 1 UNION … ORDER BY CurrencyCode
CurrencyService.ConvertAmountDelegates to GetRateCalls GetRate then multiplies and rounds to 4 decimal places

No transaction management is applied (no TransactionScope, no SqlTransaction). Each operation opens its own connection.

Caching Strategy

No caching layer is implemented. Every call to GetRate, GetSupportedCurrencies, or ConvertAmount issues a live database query against SQL Server. There is no in-memory cache, distributed cache (Redis, Memcached), or output cache configured.

Given that exchange rates change infrequently, a short-TTL cache on GetRate and GetSupportedCurrencies results would be a straightforward performance improvement opportunity during modernization.

Data Ownership Boundaries

ZavaCurrencyService is the sole consumer of ZavaBankDB within this repository. There is no multi-service architecture, no database-per-service pattern, and no evidence of cross-service data access. Data ownership is entirely monolithic within this single service.

All reads are simple point queries or aggregation queries on two tables. There are no write operations (INSERT/UPDATE/DELETE) in the service code — the database is treated as read-only from the service's perspective.
