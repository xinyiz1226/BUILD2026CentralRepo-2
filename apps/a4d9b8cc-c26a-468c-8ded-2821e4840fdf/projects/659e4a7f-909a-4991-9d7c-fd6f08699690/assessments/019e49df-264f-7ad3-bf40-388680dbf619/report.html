Data Storage & External Services

The application persists all document metadata and binary content in a single SQL Server table named Documents. Its only external runtime dependency is that SQL Server instance; configuration enters through doc-vault.properties or matching environment variables, and there are no outbound API, cache, or messaging integrations in the repository.

Key Architectural Decisions

  • Uses a monolithic servlet deployment with request handling and persistence logic implemented directly inside servlets rather than a separate service or repository layer.
  • Creates the Documents table at servlet startup through DocVaultBootstrapServlet, so schema initialization is embedded in application startup instead of a dedicated migration tool.
  • Resolves database host, port, name, user, and password from environment variables first and falls back to the bundled properties file for local defaults.

DTOs & Contracts

The application does not define dedicated request or response DTO classes. Instead, the upload endpoint accepts a multipart form contract (customerId, documentType, file), the customer listing endpoint assembles ad hoc JSONObject and JSONArray payloads for document metadata, and the download endpoint returns raw binary content with HTTP headers. There is no OpenAPI or Swagger document, no protobuf or GraphQL schema, and no immutable record-style transport model in the repository; serialization is handled manually through org.json and servlet response writers.

Communication Patterns

All runtime interactions are synchronous. Clients call servlet endpoints over HTTP, and each servlet performs its own validation, resolves database configuration through DocVaultConfig, opens a JDBC connection via DocVaultConnectionFactory, and then issues SQL directly against SQL Server. No asynchronous messaging, service discovery, API gateway, client-side load balancing, retry policy, timeout policy, bulkhead pattern, or circuit breaker implementation is present in the codebase. API-level security controls are also absent: there is no authentication, authorization, or TLS enforcement in the repository, so the endpoints are effectively public unless an external platform adds those protections.

Startup Dependency Chain

  1. Tomcat starts the ROOT.war package and loads servlet definitions from web.xml.
  2. DocVaultBootstrapServlet runs at startup (load-on-startup=1) and attempts to open a SQL Server connection.
  3. The application becomes fully ready only if SQL Server is reachable and the Documents table creation statement succeeds.
  4. After startup, /health, upload, list, and download endpoints are available through the servlet mappings in web.xml.

Core Business Workflows

ZavaDocVault supports the banking platform by capturing customer documents, exposing document history, and returning stored files on demand. Its business logic is concentrated in a few servlet workflows rather than a deep domain model.

Primary Workflows

Workflow 1: Upload customer document

A user or upstream system submits POST /api/documents/upload with a multipart form. DocumentUploadServlet validates that customerId is numeric and that a non-empty file is present, defaults documentType to General when omitted, sanitizes the submitted filename by replacing slash characters, and inserts the document into SQL Server. On success it returns a compact JSON payload containing the generated documentId and echoed metadata.

Workflow 2: List documents for a customer

A caller requests GET /api/documents/customer/{customerId} to review all stored documents for one customer. CustomerDocumentsServlet validates the path segment, queries the Documents table for that customer, sorts results by newest upload first, and returns a JSON summary containing a document count and metadata array.

Workflow 3: Download a stored document

A caller requests GET /api/documents/{id} for a specific document. DocumentDownloadServlet validates the path segment, loads the matching row, and streams the binary payload back with a download-friendly Content-Disposition header. If the ID does not exist, the servlet returns a 404 JSON error response.

Workflow 4: Bootstrap persistence on startup

When the application starts, DocVaultBootstrapServlet executes before user traffic because it is configured with load-on-startup. Its only responsibility is to create the Documents table if it is missing, ensuring the document workflows can run after startup.

Cross-Service Data Flows

No multi-service orchestration or gateway aggregation flow is present in this repository. The only cross-context handoff is the customerId value supplied by a caller that already knows the customer identity from another system. ZavaDocVault uses that identifier to partition document records but does not fetch or enrich customer data from an external service, and there is no fallback composition logic when other systems are unavailable.

Business Rules & Decision Logic

  • customerId must be present and numeric for upload and list operations; malformed values produce immediate 400 responses.
  • Upload requests must include a non-empty file part. Missing multipart content or empty files are rejected before persistence begins.
  • If documentType is blank, the service assigns the default business category General.
  • Filenames are normalized by replacing / and \ with _ before persistence to avoid storing path-like values.
  • Customer document history is presented in reverse chronological order using UploadedDate DESC, DocumentID DESC.
  • Startup initialization is mandatory for normal operations: if table creation fails, the servlet throws a ServletException and the application startup path is considered unsuccessful.
  • No business-level authorization rules, approval flows, or audit trail implementation are present in the repository.

Version & Compatibility Risks

The most visible compatibility risk is not the dependency count but the runtime baseline around them: the build targets Java 8, and AppCAT flagged that Java version as end-of-support. javax.servlet-api:3.1.0 anchors the app to the older javax.* servlet namespace and a traditional application server model, while org.json:json:20140107 is a very old release that may require modernization review during platform or framework upgrades.

Notable Observations

  • The project has a very small dependency surface area, which should simplify modernization, but nearly all business logic is therefore implemented manually inside servlets.
  • No logging, security, cache, messaging, or observability libraries are declared in the build.
  • The SQL Server JDBC driver already targets Java 8, which aligns with the current runtime but reinforces the legacy baseline identified by AppCAT.
  • The build file does not declare any test libraries, so there is no dependency-based automated test harness in the repository today.

Caching Strategy

No caching strategy is implemented. The application does not declare a cache provider, does not keep document metadata in memory, and does not use cache-aside or second-level cache patterns; every upload, list, and download request goes directly to SQL Server.

Data Ownership Boundaries

The data-store topology is a single shared SQL Server database from the perspective of this service, but only one application module in this repository accesses it. There is no evidence of cross-service direct database access, REST-based data composition, or CQRS patterns. The only external business boundary is the CustomerID value carried into and out of the Documents table, which lets another system supply the customer context while ZavaDocVault remains the source of truth for stored document blobs and their metadata.
