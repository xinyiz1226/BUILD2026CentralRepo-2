Data Storage & External Services

ZavaDocVault stores all document data — including binary file contents — in a single SQL Server database table (Documents). The schema is bootstrapped automatically at startup via DocVaultBootstrapServlet. Connection parameters (host, port, database name, user, password) are read from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with fallback to a bundled doc-vault.properties file. There are no caches, message brokers, or other external services.

Key Architectural Decisions

  • Plain JDBC over ORM: The application uses raw JDBC with PreparedStatement and ResultSet rather than any ORM framework (JPA/Hibernate), keeping the dependency set minimal but requiring manual SQL and resource management.
  • Binary storage in SQL Server: Document file contents are stored as VARBINARY(MAX) in the database rather than a dedicated blob/object store, making the service fully self-contained at the cost of database size growth.
  • Schema self-bootstrapping: DocVaultBootstrapServlet runs at startup (load-on-startup=1) and issues a conditional DDL statement (IF OBJECT_ID ... IS NULL CREATE TABLE) to create the Documents table when it doesn't exist, eliminating the need for external migration tooling.

ZavaDocVault has no formal DTO or contract classes. All response bodies are built inline using org.json.JSONObject and org.json.JSONArray directly inside servlet handler methods. There are no Java records, POJOs, or request-model classes.

Response shapes by endpoint:

Communication Patterns

Synchronous only. ZavaDocVault is a standalone service with no outbound HTTP calls, no message queue consumers or producers, and no inter-service dependencies at runtime.

Resilience patterns: None. There are no circuit breakers, retry policies, timeouts, or bulkhead patterns configured at any layer. If a JDBC call to SQL Server hangs, the servlet thread blocks indefinitely.

Service discovery: None. The application does not register with any service registry (Eureka, Consul, Kubernetes DNS). Callers connect to it by direct hostname and port.

API gateway: None. The service is accessed directly; no gateway, BFF, or reverse proxy is declared in the project.

Security posture: No authentication, no authorization, and no TLS are configured at the application level. All 5 endpoints — including document upload, list, and download — are publicly accessible with no credential checks, no role validation, and no session management. HTTP is the only transport.

Core Business Workflows

ZavaDocVault is the document storage and retrieval service for Zava Bank, allowing bank systems and staff to securely store, list, and retrieve customer documents such as bank statements, identity records, and financial reports.

ZavaDocVault is a leaf service with no outbound service calls. It owns the Document entity fully and exposes its data only through its own API endpoints.

Primary Workflows

Workflow 1: Document Upload

A caller (bank system, staff tool, or API consumer) submits a file for a specific customer. The service validates the request inputs, sanitizes the filename, stores the binary content in SQL Server, and returns the assigned document ID.

Steps:

  1. Caller submits POST /api/documents/upload with a multipart form containing customerId, an optional documentType, and the file binary.
  2. Validation: customerId must be present and parseable as an integer; if not, the request is rejected with a 400 error.
  3. Default assignment: If documentType is absent or blank, it is set to "General".
  4. Validation: The uploaded file part must be present and non-empty (size > 0); an empty or missing file is rejected with a 400 error.
  5. Filename sanitization: The submitted filename has all \ and / characters replaced with _ to prevent path-separator injection; if the filename is blank, it defaults to upload.bin.
  6. Persistence: The sanitized filename, content type, document type, customer ID, and raw binary content are written as a new row in the Documents table.
  7. The service returns the generated DocumentID along with the submission metadata.

Workflow 2: List Customer Documents

A caller retrieves a metadata list of all documents belonging to a specific customer, ordered by most recently uploaded first.

Steps:

  1. Caller submits GET /api/documents/customer/{customerId}.
  2. Validation: The {customerId} path segment must be parseable as an integer; non-numeric values are rejected with a 400 error.
  3. Query: All document metadata rows for the customer are retrieved (document ID, type, filename, content type, upload date). Binary content is NOT retrieved in this query.
  4. The service returns a JSON object with the customer ID, count of documents, and an array of document metadata records.

Workflow 3: Document Download

A caller retrieves the full binary content of a specific document by its ID, which is streamed back as a file attachment.

Steps:

  1. Caller submits GET /api/documents/{id}.
  2. Validation: The {id} path segment must be parseable as a long integer; non-numeric values are rejected with a 400 error.
  3. Lookup: The document row (filename, content type, binary data) is retrieved. If no matching document is found, a 404 response is returned.
  4. The binary content is written to the HTTP response stream with appropriate Content-Type and Content-Disposition: attachment headers.

Workflow 4: Schema Bootstrap (Startup)

At application startup, before any API request is served, the service ensures the Documents table exists in SQL Server.

Steps:

  1. Tomcat initializes DocVaultBootstrapServlet with load-on-startup=1.
  2. The servlet opens a database connection and executes a conditional DDL statement: create the Documents table only if it does not already exist.
  3. If the database is unreachable, initialization fails and Tomcat marks the servlet container as unavailable; no API endpoints become active until this step succeeds.

Cross-Service Data Flows

ZavaDocVault has no outbound calls to other services. It is a terminal service in the Zava Bank platform — it receives requests from callers and responds with its own data.

The CustomerID value is passed in by the caller and stored verbatim. ZavaDocVault does not validate it against a customer registry, does not look up customer names or contact details, and does not emit events to other services when a document is uploaded or downloaded. Any business logic that links documents to specific customer profiles must be implemented by the calling service.

There are no circuit breakers, fallback flows, or partial-result behaviors — if the database is unavailable, all API operations return a 500 error with no degraded response.

Default Values & Computed Fields:

  • documentType defaults to "General" when the submitted value is absent or blank.
  • uploadedDate is set server-side by SQL Server's GETDATE() function at INSERT time; the caller cannot supply or override it.
  • documentId is assigned by the SQL Server IDENTITY column; the caller has no influence over its value.
  • Filename defaults to "upload.bin" if the submitted filename is blank.

Data Integrity Rules:

  • Filename sanitization replaces \ and / with _ before persistence, preventing path-separator characters from reaching the stored filename.
  • No uniqueness constraint is enforced on filename or (CustomerID, FileName) — duplicate uploads are permitted without conflict detection.
  • No document size enforcement beyond the Tomcat multipart limits declared in web.xml (10 MB per file, 12 MB per request).

Transactions:

No explicit transaction management is used. Each JDBC operation is executed in its own auto-commit connection. Upload operations are single-statement INSERT statements — partial write failures are not possible at the SQL level, but JDBC connection failures mid-request will result in a 500 error with no compensating action.

Error Handling:

Business exceptions are surfaced as HTTP status codes with inline JSON error bodies ({"status":"ERROR","message":"..."}). There are no typed business exception classes, no dead-letter queues, and no compensating transactions.

Audit & Logging:

No application-level audit trail, business event log, or change tracking is implemented. UploadedDate in the Documents table provides a basic timestamp of when each document was stored, but there is no record of read operations (list or download), no access log per customer, and no change history for updates or deletions (neither operation is currently supported by the API).

Authorization:

No authorization rules are enforced. Any caller that can reach the network endpoint can upload documents for any customer ID, list documents for any customer, or download any document by ID. There is no concept of document ownership enforcement, role-based access, or customer-scoped access control.

Version & Compatibility Risks

javax.servlet-api 3.1.0 uses the legacy javax. namespace (Java EE), which is incompatible with Jakarta EE 9+ tooling that uses jakarta.. Migrating to a modern Tomcat 10+ or any Jakarta EE-compatible container would require a namespace migration across all source files. org.json:json:20140107 is pinned to a 2014 snapshot artifact; current releases are versioned by date (e.g., 20240303) and include security and correctness fixes accumulated over a decade. The mssql-jdbc 12.6.3.jre8 JRE 8 variant is the latest stable release line, but the .jre8 suffix ties the application to Java 8; upgrading to a newer LTS Java would allow switching to the .jre11 or .jre17 classifier.

ZavaDocVault has no formal repository interfaces. All data access logic is embedded inline within servlet handler methods using PreparedStatement and ResultSet. The following table documents the four SQL operations by their owning class:

Caching Strategy

No caching is configured at any layer. Every read request — including the customer document list — executes a live SQL query against SQL Server. There is no second-level cache, query result cache, session cache, or application-level cache (EhCache, Redis, Caffeine, Spring Cache, etc.).

Data Ownership Boundaries

ZavaDocVault is the sole owner of the SQL Server database it connects to and of the single Documents table within it. There is no shared database with other Zava Bank services. CustomerID is stored as a plain integer reference — no foreign-key relationship, no join query, and no cross-service database access exists; the link to customer records in other services is maintained purely by convention at the application level.

The data access pattern is simple read/write with no CQRS separation: the same database instance and table serve both write operations (upload) and read operations (list, download).
