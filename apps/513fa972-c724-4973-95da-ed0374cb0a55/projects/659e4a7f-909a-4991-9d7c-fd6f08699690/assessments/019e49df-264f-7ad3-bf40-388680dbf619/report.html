Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) to store Users, Roles, UserRoles, and SessionTokens tables. All user authentication (credential lookup, password hash verification) and session lifecycle management (create, validate, deactivate tokens) are handled through direct ADO.NET calls. There are no caches, message brokers, or other external service integrations; the SQL Server is the sole persistence dependency.

Key Architectural Decisions

  • ASHX HTTP Handlers for APIs: Token validation (/api/auth/validate) and user identity (/WhoAmI.ashx) are implemented as lightweight IHttpHandler implementations rather than MVC controllers, avoiding framework overhead for these high-frequency API endpoints.
  • Dual session token approach: After login, both an ASP.NET Forms Authentication cookie (for browser clients) and a raw GUID session token (stored in SessionTokens and embedded in the ticket's UserData) are issued, enabling cross-ecosystem SSO where Java services cannot decrypt the Forms Auth cookie.
  • Environment-variable-first connection string: DbConfig checks environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) before falling back to web.config, supporting containerized deployment without config file changes.

DTOs & Contracts

The application has no formal DTO classes or OpenAPI/Swagger specification. All request and response contracts are implicit:

  • Login request: HTML form POST with txtUsername and txtPassword fields. No explicit request model class — values are read directly from Page.Request.Form via ASP.NET Web Forms control bindings.
  • ApiAuthValidate response: Inline JSON string constructed by ApiAuthValidateHandler.ProcessRequest using string concatenation. No response model class. Fields: valid (bool), userId (int, present when valid), username (string, present when valid), expiresAt (ISO 8601 datetime, present when valid).
  • WhoAmI response: Inline JSON string constructed in WhoAmIHandler.ProcessRequest via StringBuilder. No response model class. Fields: authenticated (bool), username (string), displayName (string), sessionToken (string — raw GUID for cross-ecosystem SSO), roles (string array).
  • Internal data models: AuthUser and SessionValidationResult are internal sealed classes used only within the data access layer; they are never serialized or exposed directly in responses. See data-architecture.md for field-level details.

No protobuf schemas, GraphQL schemas, or contract-first API specifications exist. Serialization is performed by hand (string concatenation / HttpUtility.JavaScriptStringEncode) with no JSON serialization library.

Communication Patterns

Synchronous only. ZavaAuthGateway is a single-service application with no inter-service HTTP calls, message queues, or event-driven patterns. All communication is client-to-service request/response over HTTP.

Service discovery: Not applicable. The gateway is addressed by its container hostname/port directly. No Eureka, Consul, or Kubernetes DNS-based discovery is used.

Resilience patterns: None configured. There are no retry policies, circuit breakers, or timeout handling — not needed for a single-service deployment. Each request completes within a single synchronous SQL query execution.

Gateway aggregation: Not applicable. The service is itself the gateway; it does not aggregate responses from downstream services.

Security posture:

  • Authentication: ASP.NET Forms Authentication is enforced globally via web.config (<deny users="?"/>) with explicit opt-outs for public pages (Login.aspx, Logout.aspx, Default.aspx, /health, /WhoAmI.ashx, /api/auth/validate).
  • Authorization: No role-based authorization ([Authorize(Roles=...)]) is applied at the endpoint level. WhoAmI.ashx returns the user's roles but does not enforce them.
  • Transport security: No HTTPS/TLS configuration is present. The application listens on plain HTTP port 8080. TLS termination is expected to occur at an upstream reverse proxy or load balancer.
  • Token scheme: Session tokens are random GUIDs stored in SQL Server; validity is confirmed on every call to /api/auth/validate via a live database lookup (no local cache or stateless JWT validation).
  • Password hashing: SHA-1 with a per-user salt. SHA-1 is cryptographically weak and considered deprecated for password storage.
  • Machine key: A static <machineKey> is hardcoded in web.config, which is used to encrypt the Forms Authentication cookie. Leaking this key would allow cookie forgery.

Secrets Provisioning Workflow

Current state (problematic): Two categories of secrets are hardcoded directly in web.config and committed to source control:

  1. SQL Server credentials (User Id=sa; Password=Zava123!) — present as the fallback connection string. These are valid production credentials in the default configuration.
  2. Machine keys (validationKey, decryptionKey) — static keys used to encrypt and validate all Forms Authentication cookies. Any party with access to the repository can forge arbitrary .ZAVAAUTH cookies.

Container override path: The five database environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) can override the connection string at runtime via DbConfig.GetConnectionString(). When all five are set, the hardcoded web.config credentials are bypassed. However, no secrets manager, vault, or managed identity is wired up to supply these variables — they must be provided manually or via a CI/CD pipeline secret.

Machine keys are not overridable: There is no environment-variable or configuration override mechanism for machineKey. Rotating the machine key requires modifying web.config and redeploying; all existing Forms Auth cookies become invalid on rotation.

Recommended remediation: Remove credentials from web.config, configure machine key via environment-variable-backed configuration (not supported by ASP.NET Web Forms natively without custom code), and store all secrets in a secrets manager (Azure Key Vault, AWS Secrets Manager, or Kubernetes Secrets with external-secrets-operator).

Core Business Workflows

ZavaAuthGateway is an authentication gateway that manages the full lifecycle of user sessions for the ZavaBank platform — handling login, logout, session token issuance, token validation for downstream services, and authenticated identity resolution.

ZavaAuthGateway is a single-service bounded context with no cross-service aggregation. Downstream services depend on it — they call /api/auth/validate to verify session tokens — but the gateway itself makes no outbound service calls.

Primary Workflows

Workflow 1: User Login

A user submits their username and password via the Login page. The gateway validates credentials against the database, issues a session token on success, and sets an encrypted Forms Authentication cookie for browser-based SSO.

Steps:

  1. User submits the login form (POST /Login.aspx) with txtUsername and txtPassword.
  2. Gateway retrieves the user record by username from the Users table.
  3. Credential validation: If the user does not exist, IsActive = false, or IsLockedOut = true, login is rejected immediately.
  4. Password verification: Gateway computes SHA1(salt + password) and compares it to PasswordHash. If mismatch, login is rejected.
  5. On success: a random GUID session token is generated and inserted into SessionTokens with a 30-minute expiry.
  6. LastLoginDate is updated on the Users row.
  7. An encrypted FormsAuthenticationTicket (version 2, 30-minute expiry) is created with the session token embedded in UserData.
  8. The encrypted ticket is set as the .ZAVAAUTH HTTP cookie (HttpOnly, expires with ticket).
  9. User is redirected to Default.aspx.

Workflow 2: User Logout

A logged-in user navigates to Logout.aspx. The gateway revokes the active session token in the database and clears the auth cookie.

Steps:

  1. Browser issues GET /Logout.aspx with the .ZAVAAUTH cookie.
  2. Gateway decrypts the Forms Auth ticket from the cookie.
  3. If the ticket contains a non-empty UserData (session token GUID), the SessionTokens row is marked IsActive = 0 and ExpiresAt = GETDATE().
  4. FormsAuthentication.SignOut() is called to clear the server-side session.
  5. An expired replacement cookie is sent to the browser to purge the client-side .ZAVAAUTH cookie.
  6. Browser is redirected to Login.aspx?loggedOut=1.

Workflow 3: Cross-Ecosystem Session Token Validation

Downstream services (Java or other non-.NET systems that cannot decrypt the Forms Auth cookie) call /api/auth/validate with the raw session token GUID to check whether a user's session is currently valid.

Steps:

  1. Service calls GET /api/auth/validate?token=<GUID> (or sends X-Session-Token: <GUID> header).
  2. Gateway queries SessionTokens JOIN Users for an active, non-expired token whose owner is also active.
  3. If found: returns {"valid":true,"userId":N,"username":"...","expiresAt":"..."}.
  4. If not found (token missing, expired, deactivated, or user inactive): returns {"valid":false}.

Workflow 4: Browser Identity Resolution (WhoAmI)

A browser-based JavaScript client calls /WhoAmI.ashx to retrieve the currently authenticated user's identity and roles without re-authenticating.

Steps:

  1. Browser sends GET /WhoAmI.ashx with the .ZAVAAUTH cookie.
  2. Gateway decrypts the Forms Auth ticket from the cookie.
  3. If the ticket is missing, invalid, or expired, returns {"authenticated":false}.
  4. Gateway fetches the user record by username and the user's role list from the database.
  5. If the user is not active or is locked out, returns {"authenticated":false}.
  6. Constructs and returns: {"authenticated":true,"username":"...","displayName":"...","sessionToken":"...","roles":[...]} — the sessionToken field (raw GUID from UserData) enables downstream Java services to perform token validation without the Forms Auth cookie.

Cross-Service Data Flows

ZavaAuthGateway is the upstream identity provider; it has no downstream service dependencies itself. The cross-service data flow is unidirectional:


[ZavaAuthGateway] ← credentials/cookie ← [Browser]
[ZavaAuthGateway] → session token GUID → [Browser]
[ZavaAuthGateway] ← GET /api/auth/validate?token= ← [Downstream Java/Other Services]
[ZavaAuthGateway] → {"valid":true,"userId":...} → [Downstream Services]

The sessionToken field in the WhoAmI response is the key cross-ecosystem integration point: the browser obtains it from /WhoAmI.ashx and passes it to Java backend services, which then call /api/auth/validate to verify the session. This avoids sharing the .NET Forms Auth machine key with Java services.

No circuit breaker, fallback, or graceful degradation is implemented. If the SQL Server is unavailable, all authentication and validation requests fail with unhandled exceptions.

State Transitions

SessionToken lifecycle:


[Created on login] → IsActive=1, ExpiresAt=now+30min
        ↓
[Valid while] IsActive=1 AND ExpiresAt > GETDATE() AND Users.IsActive=1
        ↓
[Deactivated] IsActive=0, ExpiresAt=GETDATE() (on logout)
[Expired naturally] ExpiresAt passes (no cleanup job — rows accumulate)

No scheduled job or background cleanup process is present to purge expired SessionTokens rows; they accumulate indefinitely.

Business Constraints

  • Session duration: Fixed at 30 minutes from issuance. Sliding expiration is enabled for the Forms Auth cookie (browser session), but the raw SessionToken in the database is not extended on activity — it always expires 30 minutes after creation, regardless of continued usage.
  • No account lockout automation: IsLockedOut is a static flag; there is no failed-login counter or automated lockout trigger in the gateway code.
  • No password change or registration: The gateway only authenticates existing accounts; no self-service account management is exposed.
  • Cross-ecosystem SSO: The session token GUID stored in FormsAuthenticationTicket.UserData is the sole mechanism for Java/non-.NET services to verify identity. If the UserData field is empty (e.g., legacy tickets), WhoAmI returns the token as an empty string.

Error Handling & Audit

  • Authentication failures return generic error messages ("Invalid username or password.") to prevent username enumeration.
  • No business event logging or audit trail is written on successful or failed login attempts beyond updating LastLoginDate.
  • No compensating action exists for partial failures (e.g., if CreateSessionToken succeeds but UpdateLastLogin fails, the session is still valid — the user just has a stale LastLoginDate).
  • No transaction scope wraps the multi-step login operation (CreateSessionToken + UpdateLastLogin).

Authorization

Role data is returned by /WhoAmI.ashx for browser clients and is available to downstream services, but no role-based access control is enforced within the gateway itself. Authorization decisions based on roles must be implemented by consuming services.

Version & Compatibility Risks

The entire dependency surface of this application consists of .NET Framework 4.8 BCL assemblies. While .NET 4.8 itself is in Microsoft's long-term maintenance mode (no new features), it remains supported. The critical compatibility risk is System.Web, which is tightly coupled to the Windows IIS/ASP.NET pipeline and does not exist in .NET Core / .NET 5+. Migrating to modern .NET would require replacing System.Web (Web Forms pages, IHttpHandler, FormsAuthentication, HttpContext, HttpCookie) with ASP.NET Core equivalents (Razor Pages / Minimal API, cookie authentication middleware). The SQL Server data access uses System.Data.SqlClient, which has a NuGet-available successor (Microsoft.Data.SqlClient) compatible with all modern .NET versions. The application runs on Mono 6.12 in Docker rather than the Windows-native CLR, which introduces Mono-specific compatibility constraints and limits access to Windows-only .NET Framework APIs.

Notable Observations

  • Zero third-party NuGet packages: The entire application is built exclusively on .NET Framework BCL assemblies, making the dependency surface extremely small but tightly bound to System.Web, which is non-portable.
  • No ORM — raw ADO.NET: All database interactions use direct SqlConnection/SqlCommand calls with inline SQL strings. There is no Entity Framework, Dapper, or any other data-access library. This simplifies migration of the data layer but requires manual query porting.
  • Mono runtime as Linux host: The Dockerfile uses mono:6.12 / xsp4 instead of a native Windows container. Mono 6.12 reached end-of-life in 2024, and its compatibility with .NET Framework 4.8 assembly features is not guaranteed.
  • SHA-1 password hashing: The cryptography dependency is entirely in-box (System.Security.Cryptography.SHA1). SHA-1 is cryptographically weak; migrating to bcrypt or Argon2 would require adding an external NuGet package (e.g., BCrypt.Net-Next).

Caching Strategy

No caching layer is configured. Every call to /api/auth/validate and /WhoAmI.ashx performs a live synchronous SQL query against the SessionTokens or Users table. There is no in-memory cache, distributed cache (Redis, Memcached), or second-level query cache.

Session token validity is entirely database-driven: the ValidateSessionToken query checks IsActive = 1 and ExpiresAt > GETDATE() inline, so revoked or expired tokens are always rejected on the next call without any cache-invalidation problem — at the cost of one DB round-trip per validation request.

Data Ownership Boundaries

Shared vs isolated stores: ZavaAuthGateway is a single-service application with sole ownership of the ZavaBankDB SQL Server database. No other service is known to share this database from within the repository. However, the SourceSystem column in SessionTokens (populated as "DotNet") implies that multiple systems may write to this database — Java applications may also create session tokens identified by a different SourceSystem value.

Cross-service data access: Downstream services that need to verify a user's identity call GET /api/auth/validate?token=<guid> over HTTP; they do not query the database directly. The sessionToken GUID returned by /WhoAmI.ashx enables Java-ecosystem services (which cannot decrypt the ASP.NET Forms Auth cookie) to perform token validation via the REST endpoint, providing cross-ecosystem SSO without sharing DB credentials.

Read/write patterns: The service follows a simple read-on-every-request pattern for session validation; there is no CQRS, event sourcing, or read replica in place. SessionTokens is the highest-write table (insert on login, update on logout); Users is primarily read-heavy with a single update on each login (LastLoginDate).
