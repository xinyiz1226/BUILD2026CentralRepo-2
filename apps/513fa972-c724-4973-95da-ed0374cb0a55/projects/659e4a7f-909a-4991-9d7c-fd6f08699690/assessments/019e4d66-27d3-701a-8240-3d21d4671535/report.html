Data Storage and External Services

The sole data store is a Microsoft SQL Server instance (database name ZavaBankDB), accessed exclusively through raw ADO.NET (SqlConnection / SqlCommand). No ORM or caching layer is present. The connection string is resolved at runtime from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD), falling back to the ZavaBankDb entry in web.config. There are no external third-party service integrations.

Key Architectural Decisions

  • No ORM — raw ADO.NET: All database access is performed via SqlConnection and SqlCommand directly, keeping the data layer thin but tightly coupled to SQL Server syntax.
  • Forms Authentication for SSO: A signed and encrypted .ZAVAAUTH cookie carries a session token in its UserData field, enabling cross-ecosystem SSO by sharing that raw GUID token with non-.NET consumers via the WhoAmI handler.
  • IHttpHandler endpoints alongside WebForms pages: Lightweight ASHX handlers expose machine-readable JSON APIs (token validation, user identity, health probe) without introducing a separate API framework.

There are no formal DTO classes. All API responses are hand-built JSON strings constructed inline within the handler methods using StringBuilder and HttpUtility.JavaScriptStringEncode. No OpenAPI/Swagger, protobuf, or GraphQL specifications exist.

Internal model classes (not part of any public API contract):

  • SessionValidationResult — internal record returned from session token validation; carries UserId, Username, ExpiresAt. See data-architecture.md for field details.

Serialization is performed manually (no Jackson, System.Text.Json, or Newtonsoft.Json). This means there is no schema, no versioning, and no contract enforcement — the JSON shape is implicitly defined by the string-building code in each handler.

Communication Patterns

Synchronous only: All communication is synchronous HTTP. There are no message queues, event streams, or asynchronous patterns. The application does not call any downstream services; it communicates only with its SQL Server database (via ADO.NET).

Resilience: No circuit-breaker, retry, timeout, or bulkhead patterns are implemented. A database connection failure will propagate as an unhandled exception.

Service discovery: Not applicable — the application is a single deployable unit with no service registry.

API gateway / aggregation: Not applicable — the application itself acts as the authentication gateway; it does not aggregate responses from multiple backends.

Security posture: Transport security depends entirely on IIS/reverse-proxy configuration external to the application. At the application layer, authentication is enforced via ASP.NET Forms Authentication (<authentication mode="Forms"> in web.config) with a cookie named .ZAVAAUTH. The <authorization><deny users="?"/> rule globally restricts unauthenticated access. Public paths (Login.aspx, Logout.aspx, Default.aspx, /api/auth/validate, /health, /WhoAmI.ashx) are explicitly exempt. The session token transmitted in UserData of the Forms Auth ticket and in the sessionToken JSON field is a plain GUID — there is no JWT or OAuth2 involvement. The machineKey used to encrypt/decrypt the auth cookie is hardcoded in web.config (a known security concern). No HTTPS/TLS is enforced at the application level.

Core Business Workflows

ZavaAuthGateway provides centralized employee authentication and session management for the ZavaBank ecosystem, enabling users to log in with credentials, maintain a shared session token across .NET and Java consumers, and log out securely.

ZavaAuthGateway is the single source of truth for user identity and session state in the ZavaBank ecosystem. External consumers (Java portals, etc.) do not access the database directly — they validate sessions via the /api/auth/validate endpoint or retrieve user identity via /WhoAmI.ashx.

Primary Workflows

Workflow 1: Employee Login

An employee submits their username and password via the Login.aspx form. The application looks up the user record, verifies the password, creates a session token, and issues a Forms Authentication cookie.

Steps:

  1. Employee submits username and password via POST to Login.aspx.
  2. Input validation: both fields must be non-empty (client-side guard, server enforced).
  3. User lookup: query Users table by username.
  4. Account eligibility check: user must exist, be active (IsActive=true), and not be locked out (IsLockedOut=false).
  5. Password verification: SHA-1 hash of (salt + password) is compared to PasswordHash stored in the database.
  6. Session token creation: a new GUID token is inserted into SessionTokens with a 30-minute expiry, recording IP address and user agent.
  7. Last-login update: LastLoginDate and ModifiedDate on the Users row are updated.
  8. Forms Auth ticket issuance: a FormsAuthenticationTicket is created (version 2, 30-minute expiry, non-persistent) with the session GUID stored in UserData, then encrypted and set as the .ZAVAAUTH cookie (HttpOnly, path-scoped).
  9. Redirect to Default.aspx.

Workflow 2: Session Token Validation (Cross-Ecosystem SSO)

An API consumer (e.g., a Java service) submits a session token (GUID) to verify its validity without needing to decrypt the .ZAVAAUTH cookie.

Steps:

  1. Consumer sends GET /api/auth/validate with ?token=GUID query parameter or X-Session-Token header.
  2. Token presence check: if no token is provided, return 400 {"valid":false,"error":"token is required"}.
  3. Database lookup: query SessionTokens JOIN Users where token matches, session is active (IsActive=1), session has not expired (ExpiresAt > GETDATE()), and user is active (IsActive=1).
  4. If no matching record: return 200 {"valid":false}.
  5. If valid: return 200 {"valid":true,"userId":N,"username":"...","expiresAt":"..."}.

Workflow 3: WhoAmI — Authenticated User Identity

A consumer (browser or API) presents the .ZAVAAUTH cookie to retrieve full user identity including display name and roles.

Steps:

  1. Consumer sends GET /WhoAmI.ashx with the .ZAVAAUTH cookie.
  2. Cookie presence check: if absent or empty, return {"authenticated":false}.
  3. Cookie decryption: FormsAuthentication.Decrypt() decodes the ticket; if decryption fails or ticket is expired, return {"authenticated":false}.
  4. User lookup: load user by ticket.Name (username); verify IsActive=true and IsLockedOut=false.
  5. Roles lookup: load all roles for the user via UserRoles JOIN Roles.
  6. Compose and return JSON: {"authenticated":true,"username":"...","displayName":"...","sessionToken":"...","roles":[...]}.
    • displayName is FirstName + " " + LastName, falling back to Username if blank.
    • sessionToken is the raw GUID from ticket.UserData, enabling Java consumers to use the token for cross-ecosystem SSO without decrypting the cookie.

Workflow 4: Employee Logout

An employee visits Logout.aspx to end their session.

Steps:

  1. Employee navigates to Logout.aspx (GET).
  2. Cookie retrieval: read .ZAVAAUTH cookie from the request.
  3. If cookie is present and non-empty: decrypt the ticket and extract UserData (session token GUID).
  4. Session deactivation: UPDATE SessionTokens SET IsActive=0, ExpiresAt=GETDATE() WHERE Token=GUID.
  5. Forms Auth sign-out: FormsAuthentication.SignOut() clears the server-side session.
  6. Cookie expiry: an expired .ZAVAAUTH cookie (past-dated) is written to the response to delete it from the browser.
  7. Redirect to Login.aspx?loggedOut=1.

Cross-Service Data Flows

ZavaAuthGateway is a standalone service with no upstream service dependencies. Data flows are:

  • Inbound: Browser/client → ZavaAuthGateway (login form, logout, WhoAmI)
  • Inbound (API consumers): Java or other services → /api/auth/validate to check session validity
  • Outbound: ZavaAuthGateway → SQL Server only (no calls to downstream services)

The sessionToken field in the /WhoAmI.ashx response is the primary cross-ecosystem SSO bridge: a Java consumer retrieves this raw GUID and can then present it to /api/auth/validate to confirm the session is still active, without needing to parse or decrypt the .NET Forms Auth cookie format.

No circuit-breaker or fallback behavior exists for the database dependency — a SQL Server failure results in an unhandled exception propagated to the client.

Business Rules and Decision Logic

Validation Rules

  • Login inputs: Both username and password must be non-empty strings (trimmed); empty inputs are rejected before any database query.
  • Token input: The token query parameter or X-Session-Token header must be non-empty; an empty token returns HTTP 400.

Decision Logic

  • Account eligibility (Login): A user must exist in the database, have IsActive=true, and have IsLockedOut=false. Any one of these conditions failing produces the same generic error message to prevent user enumeration.
  • Password verification: SHA-1 hash of (salt || password) must exactly match PasswordHash (case-insensitive hex comparison). SHA-1 is cryptographically weak — this is a known security risk.
  • Session validation: A session token is valid only if IsActive=1, ExpiresAt > GETDATE(), and the owning user has IsActive=1. All conditions must hold simultaneously.
  • WhoAmI identity: If FirstName and LastName are both blank, Username is used as the display name.

UserActive / Locked / InactiveState is managed externally (no workflow in this application to lock/unlock/activate users)

Business Constraints

  • Session duration: 30 minutes from creation; sliding expiration is configured on the Forms Auth cookie but the SessionTokens row has a fixed ExpiresAt (no sliding update on use).
  • Session token uniqueness: Token is a Guid.NewGuid() — statistically unique; no explicit uniqueness constraint is enforced in application code.
  • Generic error messages: Login failures deliberately use a single error message regardless of failure reason (not found, locked, wrong password) to prevent user enumeration.

Cross-Cutting Concerns

  • Transactions: No explicit transaction scope (TransactionScope or @Transactional) is used. The login workflow performs multiple database writes (INSERT session token, UPDATE last login) without wrapping them in a transaction — a partial failure would leave inconsistent state.
  • Error handling: All exceptions in WhoAmI.ashx are caught silently and return {"authenticated":false}. Logout.aspx catches decryption errors silently. No structured error logging is present.
  • Audit / logging: No audit trail, structured logging, or event emission is implemented. Login attempts, failures, and logouts are not logged.
  • Authorization: Role-based data is returned in /WhoAmI.ashx for downstream consumers to enforce RBAC; ZavaAuthGateway itself does not enforce role-based access beyond <deny users="?"/> (authenticated vs unauthenticated).

Version and Compatibility Risks

The application's only runtime dependencies are assemblies shipped with .NET Framework 4.8, which Microsoft supports in maintenance mode. While .NET Framework 4.8 itself receives security fixes, its core web stack (System.Web) is fundamentally incompatible with .NET 5/6/7/8/10 — it will never be ported. Migrating to modern .NET requires replacing System.Web (WebForms, HttpContext, FormsAuthentication, HttpHandler) with ASP.NET Core equivalents, and replacing System.Data.SqlClient with Microsoft.Data.SqlClient. No third-party packages are present that would introduce additional version-compatibility concerns, which simplifies the migration surface.

Notable Observations

  • No third-party NuGet packages: The application achieves its full functionality with zero external package dependencies. This eliminates transitive-dependency risk but means all functionality is tightly coupled to the .NET Framework BCL.
  • System.Web is a hard migration blocker: System.Web is not available on modern .NET. All WebForms pages (ASPX), ASHX handlers, FormsAuthentication, and HttpContext APIs must be rewritten or replaced during any cloud-native migration.
  • System.Data.SqlClient is legacy: The BCL System.Data.SqlClient namespace has been superseded by the separately distributed Microsoft.Data.SqlClient NuGet package, which receives active security updates and new SQL Server features.
  • No logging, observability, or caching libraries: The application lacks structured logging, distributed tracing, or caching dependencies — gaps that will need to be addressed when migrating to a cloud-native target.

Caching Strategy

No caching layer is configured. Every request to /api/auth/validate or /WhoAmI.ashx issues live SQL queries against the database. Session token data is stored exclusively in the SessionTokens table; there is no in-memory cache, distributed cache (Redis, Memcached), or second-level query cache.

Data Ownership Boundaries

ZavaAuthGateway is a single-service, single-database application. All four tables reside in a shared SQL Server database (ZavaBankDB), owned and accessed exclusively by this service. No other service is known to share these tables directly; external consumers authenticate via the /api/auth/validate REST endpoint rather than querying the database directly.

There are no CQRS, event sourcing, or read-replica patterns. All reads and writes flow through the same AuthRepository using the same connection string.

The application does not implement an outbox pattern; session token creation and the Forms Auth cookie issuance are not atomic — a failure after CreateSessionToken but before the cookie response would leave an orphaned token row.
