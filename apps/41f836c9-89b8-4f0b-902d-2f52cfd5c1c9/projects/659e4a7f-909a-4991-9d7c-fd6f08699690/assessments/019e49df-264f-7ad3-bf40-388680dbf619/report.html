Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) accessed via ADO.NET (SqlDataAdapter and SqlCommand) to store loan product definitions and loan application records. There is no ORM or caching layer. Two external HTTP services are integrated: the ZavaAuthGateway handles all authentication redirects (login and logout), and the Loan Origination API (zava-loan-origination-api:8080) receives new loan application submissions via HTTP POST with XML payloads.

Key Architectural Decisions

  • Web Forms with code-behind: UI logic is embedded directly in page code-behind classes (Default.aspx.cs, Login.aspx.cs) rather than using a MVC or MVVM separation of concerns.
  • Direct ADO.NET data access: All database interactions use raw SqlConnection/SqlCommand calls within page code-behind methods, with no repository or service layer abstraction.
  • External authentication delegation: Login and logout flows are fully delegated to an external ZavaAuthGateway; the portal relies on the resulting Forms Authentication cookie for session management.

DTOs & Contracts

No formal DTO or contract classes are defined in ZavaLoanPortal. All data flows through:

  • ASP.NET Web Forms postback state: Form field values are carried in __VIEWSTATE and __EVENTVALIDATION hidden fields. There are no request/response model classes.
  • ADO.NET DataTable: Used as the in-memory data container for loan products (LoanProducts) and loan history (LoanApplications). DataTable is bound directly to DropDownList and GridView controls.
  • XML string payload (outbound): The loan application is hand-serialized into an XML string (<loanApplication><customerId>…</customerId>…</loanApplication>) and sent to the Loan Origination API via HttpWebRequest. There is no typed DTO class or schema contract.
  • No OpenAPI/Swagger/protobuf/GraphQL: No API specification files are present. The outbound XML payload has no accompanying XSD schema.

Communication Patterns

Synchronous patterns:

  • Browser ↔ ZavaLoanPortal: Standard HTTP request/response using ASP.NET Web Forms postbacks. UpdatePanel wraps the loan wizard and history grid, enabling partial-page async postbacks (ASP.NET AJAX / ScriptManager), though these are still synchronous HTTP calls from the browser's perspective.
  • ZavaLoanPortal → SQL Server: Synchronous ADO.NET calls (SqlConnection.Open(), SqlCommand.ExecuteNonQuery(), SqlDataAdapter.Fill()) made on the request thread with no async/await.
  • ZavaLoanPortal → Loan Origination API: Synchronous HttpWebRequest.GetResponse() call. The response body is read and discarded; exceptions are silently swallowed. No timeout is configured, so the default socket timeout applies.
  • ZavaLoanPortal → ZavaAuthGateway: HTTP redirect-based; the portal issues a Response.Redirect to the external gateway URL, and the gateway redirects back with a Forms Authentication cookie. No direct server-to-server HTTP call occurs.

Asynchronous patterns: None. No message queues, event-driven patterns, or async/await usage are present.

Resilience patterns: None. The outbound HTTP call to the Loan Origination API wraps GetResponse() in a bare try/catch that silently discards all exceptions. There is no retry, circuit breaker, timeout configuration, or fallback behavior.

Service discovery: Hardcoded URLs. Both AuthGatewayLoginUrl/AuthGatewayLogoutUrl and LoanOriginationApiUrl are read from AppSettings in web.config. No service registry or DNS-based discovery is used.

Security posture: Forms Authentication (cookie-based, .ZAVAAUTH) is enforced at the application level via web.config. All pages except Login.aspx and Logout.aspx deny unauthenticated users. However, no TLS/HTTPS is configured (all URLs in web.config use http://), the machine key (validationKey/decryptionKey) is hardcoded in web.config in plain text, and the Loan Origination API endpoint receives sensitive financial data over plain HTTP. No authorization beyond authentication (roles, claims, or RBAC) is implemented.

Primary Workflows

Workflow 1: User Authentication

A unauthenticated user accesses the portal and is redirected to the external ZavaAuthGateway to authenticate. Upon successful authentication, the gateway issues a .ZAVAAUTH Forms Authentication cookie and redirects the browser back to the portal's original target URL.

Steps:

  1. User navigates to /Default.aspx (or any protected page).
  2. Portal detects unauthenticated request; issues 302 redirect to /Login.aspx?ReturnUrl=....
  3. Login page renders a link to ZavaAuthGateway login URL (from AuthGatewayLoginUrl config), encoding the ReturnUrl.
  4. User clicks the link; browser navigates to ZavaAuthGateway.
  5. ZavaAuthGateway validates credentials and sets the .ZAVAAUTH cookie; redirects back to ReturnUrl.
  6. Portal reads the cookie, confirms authentication, and serves the requested page.

Business Rules Involved: Only authenticated users may access the loan portal (all pages deny unauthenticated users except /Login.aspx and /Logout.aspx).

Workflow 2: Loan Application Submission

An authenticated user completes a five-step wizard to submit a loan application. The application is simultaneously recorded in the local database and forwarded to the external Loan Origination API.

Steps:

  1. User lands on /Default.aspx; active loan products are loaded from LoanProducts and displayed in a dropdown.
  2. Step 1 – Personal Info: User enters CustomerID, first name, last name, and email.
  3. Step 2 – Employment: User enters employer, job title, years employed, and annual income.
  4. Step 3 – Loan Details: User selects a loan product, enters requested amount and term (months), and provides a purpose statement.
  5. Step 4 – Review: Portal renders a summary of name, amount, and term for the user to confirm.
  6. Step 5 – Submit (Finish): User clicks Finish; portal:

a. Validates that requested amount is a valid decimal and term is a valid integer.

b. Sends an XML payload to the Loan Origination API via HTTP POST (fire-and-forget; errors are silently discarded).

c. Inserts the application record into LoanApplications with status Submitted and AssignedOfficer set to the authenticated user's identity name.

  1. Portal displays "Application submitted." and refreshes the loan history grid.

Business Rules Involved:

  • Amount and term must be valid numeric values (checked at Step 3 → Step 4 navigation); wizard navigation is cancelled if validation fails.
  • Status is always set to Submitted on creation; no other initial status is possible.
  • AssignedOfficer is automatically set to the authenticated user's identity (User.Identity.Name).
  • The external API call and the DB insert are independent; failure of the API call does not prevent the DB insert.

Workflow 3: Loan History Review

On every page load and after each application submission, the portal retrieves and displays the 25 most recent loan applications.

Steps:

  1. BindLoanHistory() executes SELECT TOP 25 ... FROM LoanApplications ORDER BY ApplicationDate DESC.
  2. Results are bound to the gvLoanHistory GridView.
  3. Each row includes a "Review" button; clicking it triggers gvLoanHistory_RowCommand, which sets a status label ("Application selected for review.") but performs no further action (stub implementation).

Workflow 4: User Logout

The user clicks the logout link in the site header; the portal clears the Forms Authentication session and redirects to the external ZavaAuthGateway logout URL.

Steps:

  1. User navigates to /Logout.aspx.
  2. Portal calls FormsAuthentication.SignOut(), invalidating the .ZAVAAUTH cookie.
  3. Portal issues 302 redirect to AuthGatewayLogoutUrl (from web.config).
  4. ZavaAuthGateway completes the logout flow on its end.

Cross-Service Data Flows

ZavaLoanPortal is a single-service application with no gateway aggregation or inter-service composition patterns. Data flows are:

  • Inbound (read): Portal queries ZavaBankDB for loan products and application history on each page load. No caching; data is always fresh from the database.
  • Outbound (write, fire-and-forget): On application submission, the portal serializes the loan application into a hand-crafted XML string and POSTs it to the Loan Origination API. The response is discarded. If the API is unavailable, the exception is silently swallowed and the portal continues to insert the record into the local database.
  • Outbound (redirect): Authentication and logout are delegated to ZavaAuthGateway via browser redirects — no server-to-server HTTP calls for auth.

Fallback behavior: When the Loan Origination API is unavailable, the application record is still saved to LoanApplications in the local database. The user sees "Application submitted." regardless of whether the external API received the data. There is no retry, notification, or compensating mechanism.

Business Rules & Decision Logic

Validation Rules

  • Loan amount: Must parse as a valid decimal. Checked when the user navigates from Step 3 (Loan Details) to Step 4 (Review). If invalid, wizard navigation is cancelled and an error label is shown.
  • Term months: Must parse as a valid int. Checked at the same point as loan amount.
  • No other field-level validation is present: CustomerID, first name, last name, email, employer, job title, income, and purpose are accepted as free-text without format or range checks.

Decision Logic

  • Access control gate: On every page load of Default.aspx, Request.IsAuthenticated is checked. If false, the user is immediately redirected to the login page; no further processing occurs.
  • Already-authenticated redirect on Login: If an authenticated user navigates to /Login.aspx, they are immediately redirected to the ReturnUrl (defaulting to ~/Default.aspx).
  • Loan product availability: Only products with IsActive=1 are shown in the dropdown; inactive products are excluded from selection.
  • Wizard step review population: When the wizard advances to the Review step (index 3), the summary label is programmatically populated from wizard field values.

Business Constraints

  • Top 25 history: Only the 25 most recent loan applications are displayed; older records are not paginated or accessible through the UI.
  • AssignedOfficer auto-set: The officer field is always set to the currently authenticated user's identity name; it cannot be overridden by the applicant.
  • CustomerID is self-reported: The CustomerID field defaults to "1" and is not validated against any customer registry; the portal trusts the user's input.

Cross-Cutting Concerns

  • Transactions: No explicit transaction management. The HTTP POST to the Loan Origination API and the INSERT into LoanApplications are two independent, non-transactional operations with no saga or compensating logic.
  • Error handling: The outbound HTTP call to the Loan Origination API is wrapped in a try/catch that silently discards all exceptions. No logging, alerting, or retry occurs on failure.
  • Audit/logging: No structured logging or audit trail is implemented. AssignedOfficer and ApplicationDate/CreatedDate/ModifiedDate in LoanApplications provide a minimal record of who submitted and when.
  • Authorization: Authentication is enforced (deny all unauthenticated users), but no role-based or attribute-based authorization is implemented. Any authenticated user can submit any loan application with any CustomerID.

Version & Compatibility Risks

All dependencies are in-box .NET Framework 4.8 assemblies — no third-party NuGet packages are declared. While .NET Framework 4.8 itself remains in long-term support, ASP.NET Web Forms (System.Web) is the primary migration blocker: it is not available on .NET 6/7/8/10 and has no direct upgrade path. FormsAuthentication (part of System.Web.Security) is equally tied to the legacy System.Web pipeline and must be replaced with ASP.NET Core's cookie authentication middleware when targeting modern .NET. System.Data.SqlClient can be replaced by Microsoft.Data.SqlClient (drop-in NuGet replacement) on modern .NET, and the raw ADO.NET pattern can optionally be migrated to Entity Framework Core or Dapper.

Notable Observations

  • No NuGet packages: The project declares zero third-party NuGet packages (packages.config is empty), making dependency hygiene straightforward — the only migration effort is at the framework/BCL level.
  • System.Web is the critical blocker: System.Web (Web Forms, HttpContext, FormsAuthentication, HttpWebRequest pipeline) is the largest single obstacle to moving to modern .NET; the entire presentation and authentication stack must be rewritten.
  • HttpWebRequest for external HTTP calls: System.Net.HttpWebRequest is used to call the Loan Origination API; this should be replaced with System.Net.Http.HttpClient (available in .NET 4.5+ and all modern .NET versions) as HttpWebRequest is considered legacy.
  • No logging framework: The application relies entirely on inline try/catch blocks with swallowed exceptions and no structured logging library (e.g., Serilog, NLog, Microsoft.Extensions.Logging), which is a gap that should be addressed during modernization.

Caching Strategy

No caching layer is present. Every page request or postback triggers fresh SQL queries against SQL Server:

  • Loan products list is re-queried from LoanProducts on every Page_Load (non-postback).
  • Loan history is re-queried after every wizard submission and on initial page load.

No HTTP response caching, output caching (OutputCache), in-memory caching (MemoryCache), or distributed caching (Redis, Memcached) is configured. The System.Web.Caching.Cache API available in ASP.NET Web Forms is unused.

Data Ownership Boundaries

ZavaLoanPortal is a single-service application with sole ownership of both database tables (LoanProducts, LoanApplications). There is no database-per-service topology and no inter-service data sharing at the database level.

Cross-service data flow occurs only outbound: when a loan application is submitted, the portal fires an HTTP POST to the external Loan Origination API in addition to inserting the record into LoanApplications. These two write operations are not coordinated in a transaction — the HTTP call is made first, then the DB insert occurs. If the HTTP call fails (exception swallowed), the DB insert still proceeds; if the DB insert fails, no compensating action removes the application from the external API.

No CQRS, event sourcing, outbox pattern, or read/write separation is implemented.
