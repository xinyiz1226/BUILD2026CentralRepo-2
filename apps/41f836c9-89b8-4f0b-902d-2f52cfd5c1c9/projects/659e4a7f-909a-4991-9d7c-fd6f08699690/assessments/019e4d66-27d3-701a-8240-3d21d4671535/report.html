Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) accessed directly through ADO.NET (SqlConnection / SqlCommand / SqlDataAdapter). Two tables are referenced at runtime: LoanProducts (for the dropdown of available loan types) and LoanApplications (for submitting and listing past applications). There is no caching layer or message broker. Two external HTTP services are consumed: the ZavaAuthGateway handles all authentication and sign-out via browser redirects (configured via AuthGatewayLoginUrl / AuthGatewayLogoutUrl), and the LoanOriginationApi receives raw XML loan-application payloads via an HTTP POST (configured via LoanOriginationApiUrl).

Key Architectural Decisions

  • No ORM: All data access uses raw ADO.NET with inline SQL strings, meaning schema changes require manual query updates throughout the code-behind files.
  • Delegated authentication: Rather than handling credentials locally, the application redirects unauthenticated users to an external ZavaAuthGateway and relies on Forms Authentication cookies (ZAVAAUTH) issued by that gateway.
  • AJAX partial rendering: The main loan wizard is wrapped in an UpdatePanel to enable partial-page postbacks without full page reloads, which is a classic ASP.NET Web Forms AJAX pattern.

DTOs & Contracts

The application defines no explicit DTO or contract classes. Data is passed between the UI and the database through direct ADO.NET DataTable objects and plain .NET primitive types (int, decimal, string). The only structured payload is the outbound loan-application XML body, which is constructed as an inline string concatenation in Default.aspx.cs:


<loanApplication>
  <customerId>{int}</customerId>
  <loanProductId>{int}</loanProductId>
  <requestedAmount>{decimal:F2}</requestedAmount>
  <termMonths>{int}</termMonths>
</loanApplication>

There is no OpenAPI/Swagger specification, no protobuf schema, no GraphQL schema, and no serialization configuration class. The XML payload has no schema validation on the sending side.

Communication Patterns

Synchronous – Inbound (browser → portal): All browser interactions use standard HTTP form POST with ASP.NET ViewState. The UpdatePanel / ScriptManager pair enables partial-page postbacks (ASP.NET AJAX), but these are still synchronous HTTP requests from the browser's perspective.

Synchronous – Outbound (portal → LoanOriginationApi): The Default.aspx.cs SubmitApi method uses System.Net.HttpWebRequest to POST an XML payload to the configured LoanOriginationApiUrl. The call is fire-and-forget within the same request thread; any exception is caught and silently discarded (empty catch block). There is no retry policy, no circuit breaker, no timeout configuration, and no fallback behaviour; a slow or unavailable LoanOriginationApi will block the IIS thread for the duration of the default OS-level TCP timeout.

Service Discovery: Services are located via hardcoded URLs read from web.config <appSettings> (AuthGatewayLoginUrl, AuthGatewayLogoutUrl, LoanOriginationApiUrl). There is no service registry, DNS-based discovery, or load balancing.

Authentication / Authorisation: ASP.NET Forms Authentication is configured with <deny users="?" />, requiring all users to be authenticated to access any page except Login.aspx and Logout.aspx. The actual credential verification is fully delegated to the external ZavaAuthGateway via browser redirect. The .ZAVAAUTH cookie is protected with static machineKey values committed directly in web.config (fixed validationKey and decryptionKey), which is a security risk in multi-node or containerised deployments. TLS is not enforced in the application configuration; there is no HTTPS redirect or HSTS header.

No asynchronous messaging (Kafka, RabbitMQ, Azure Service Bus, etc.) is present.

Startup Dependency Chain

The application has a flat startup dependency chain with no orchestration tooling:

  1. SQL Server (sqlserver:1433) — must be available before the portal can serve any authenticated page (all page loads query the database). No health-check or wait mechanism is configured; failure results in a runtime SqlException.
  2. ZavaAuthGateway — must be reachable for users to authenticate. Unavailability causes the login link to point to an unreachable URL but does not prevent application startup.
  3. LoanOriginationApi — must be reachable to receive submitted loan applications. Unavailability is silently swallowed (empty catch block); the portal continues operating but outbound API calls are lost.
  4. ZavaLoanPortal — starts after IIS / xsp4 initialises the ASP.NET application domain.

No dockerize, Kubernetes readiness probes, Spring Cloud Config retry, or depends_on health checks are configured.

Secrets Provisioning Workflow

There is no secrets management workflow. All secrets are committed as plain text in web.config in the source repository:

  • Source: Source-controlled web.config file.
  • Identity / access model: None — no managed identity, service principal, RBAC, or vault integration is configured.
  • Provisioning sequence: Secrets are embedded at build time; rotating them requires editing web.config and redeploying the application.
  • Risk: The SQL Server sa password and both machineKey values are static and publicly readable in the repository. A compromised repository or Docker image exposes full database access and the ability to forge authentication cookies.

Recommended remediation: move all secrets to Azure Key Vault or environment variable injection at container startup, and regenerate the static machineKey values.

Primary Workflows

Workflow 1: Officer Authentication

An unauthenticated visitor (or an officer whose cookie has expired) is redirected from any protected page to Login.aspx, which renders a link to the external ZavaAuthGateway. The gateway performs the actual credential check and issues a .ZAVAAUTH Forms Authentication cookie. On successful authentication the user is returned to the originally requested URL.

Steps:

  1. Officer navigates to /Default.aspx.
  2. ASP.NET Forms Authentication checks for .ZAVAAUTH cookie.
  3. Cookie absent or invalid → 302 redirect to Login.aspx?ReturnUrl=/Default.aspx.
  4. Login.aspx renders a link to AuthGatewayLoginUrl?ReturnUrl=<encoded-return-url>.
  5. Officer clicks the link and authenticates at ZavaAuthGateway.
  6. Gateway issues .ZAVAAUTH cookie and redirects back to ReturnUrl.
  7. Officer lands on /Default.aspx as an authenticated user.

Business rules involved:

  • All pages except Login.aspx and Logout.aspx require authentication (<deny users="?"/>).
  • The ReturnUrl is passed through so officers return to their original destination after login.

Workflow 2: Loan Application Submission (5-Step Wizard)

An authenticated officer completes a five-step wizard to submit a new loan application on behalf of a customer. On completion the application is persisted in ZavaBankDB and a notification XML payload is forwarded to the downstream LoanOriginationApi.

Steps:

StepWizard StepFields CollectedBusiness Activity
1Personal InfoCustomerID, First Name, Last Name, EmailOfficer identifies the applicant customer
2EmploymentEmployer, Job Title, Years Employed, Annual IncomeOfficer captures income and employment details
3Loan DetailsLoan Product (dropdown), Requested Amount, Term (months), PurposeOfficer selects product and financial terms; numeric validation enforced
4ReviewSummary of steps 1–3Officer reviews before committing
5Submit(no new inputs)Officer clicks Finish to commit

On Finish:

  1. Loan details are parsed from form fields.
  2. An XML payload is sent to LoanOriginationApi via HTTP POST (fire-and-forget).
  3. A new row is inserted into LoanApplications with Status = 'Submitted' and ApplicationDate = GETDATE().
  4. The loan history grid is refreshed to show the newly submitted application.
  5. The wizard resets to step 1 for the next application.

Workflow 3: Loan History Review

On every non-postback load of Default.aspx, the officer sees the 25 most recent loan applications across all customers, ordered by application date descending. Each row has a "Review" button that sets a status label but performs no further navigation (the review detail feature is not yet implemented).

Steps:

  1. Page loads; BindLoanHistory() queries LoanApplications for the top 25 records.
  2. GridView is populated with ApplicationID, CustomerID, Amount, Term, Status, and Date.
  3. Officer clicks "Review" on a row → gvLoanHistory_RowCommand fires → status label updated to "Application selected for review." (placeholder; no routing occurs).

Workflow 4: Officer Sign-Out

An authenticated officer navigates to Logout.aspx. The application immediately invalidates the local Forms Authentication ticket and redirects the browser to the external ZavaAuthGateway logout URL to terminate the gateway session as well.

Steps:

  1. Officer clicks Logout in the navigation bar.
  2. FormsAuthentication.SignOut() clears the .ZAVAAUTH cookie.
  3. 302 redirect to AuthGatewayLogoutUrl.
  4. ZavaAuthGateway terminates its own session and confirms logout.

Cross-Service Data Flows

This is a single-service application with two outbound integrations; there is no gateway-style aggregation pattern.

LoanApplication → LoanOriginationApi: After the wizard finishes, ZavaLoanPortal constructs a raw XML document (<loanApplication>) from the form values and POSTs it synchronously to LoanOriginationApi. There is no response body consumed and no error handling — if the downstream API is unavailable, the exception is silently discarded. The loan record is still written to the local database regardless of the API call outcome, meaning the portal database and the downstream processing system can diverge (data inconsistency risk).

ZavaAuthGateway → ZavaLoanPortal: ZavaAuthGateway issues the .ZAVAAUTH Forms Authentication cookie. ZavaLoanPortal does not call the gateway programmatically after that; it trusts the cookie's User.Identity.Name for all subsequent identity assertions (including the AssignedOfficer database field). No token introspection or session validation against the gateway occurs during page requests.

State Transitions

LoanApplication.Status has only one transition visible in this portal: an application is always created with Status = 'Submitted'. No further status transitions (e.g., Under Review → Approved → Disbursed) are implemented in this codebase. The "Review" button on the history grid is a placeholder with no state-change logic.

Authorization Rules

  • Page-level: All pages except Login.aspx and Logout.aspx require an authenticated user (Forms Auth <deny users="?"/>).
  • Role-level: No role-based authorization is implemented. Any authenticated user can submit applications, view all 25 most recent applications from all customers, and trigger the (stub) review action.
  • Data-level: No row-level security; the history grid shows all applications regardless of which officer submitted them.
  • AssignedOfficer: Set to User.Identity.Name of the currently authenticated officer at submission time; no validation that the officer is authorised to act on behalf of the specified CustomerID.

Transaction Boundaries

No explicit transaction scope (TransactionScope, SqlTransaction) is used. The INSERT INTO LoanApplications statement is executed as a single auto-committed statement. The outbound HTTP call to LoanOriginationApi occurs before the database insert; there is no rollback if the database write subsequently fails, and no retry or compensating action if the HTTP call fails. This creates a potential inconsistency between the downstream system and the local database.

Audit & Logging

No structured logging framework is present. The application relies entirely on ASP.NET's default error page (customErrors mode="Off") for exception visibility. No business event logging, no audit trail for application submissions, and no change-tracking mechanism are implemented.

Error Handling

  • SubmitApi() wraps the HttpWebRequest call in a bare catch with an empty body — all HTTP failures are silently discarded.
  • int.Parse() and decimal.Parse() in wizLoanApplication_FinishButtonClick are not wrapped in try/catch; a malformed value that bypasses step-3 validation (e.g., via direct form manipulation) will throw an unhandled exception.

Version & Compatibility Risks

All dependencies are .NET Framework 4.8 built-in assemblies and carry no independent version number beyond the framework itself. .NET Framework 4.8 is in long-term servicing mode and will not receive new features; it is not cross-platform and cannot run on Linux containers without Windows Server Core base images. System.Web (the core of ASP.NET Web Forms) does not exist in .NET 5/6/7/8/9/10, which is the single largest migration blocker for this application. System.Data.SqlClient is superseded by Microsoft.Data.SqlClient in the modern ecosystem. System.Net.HttpWebRequest (used for the LoanOriginationApi call) is superseded by System.Net.Http.HttpClient.

Notable Observations

  • Zero third-party NuGet packages: The entire application depends only on the .NET Framework BCL, making the package-level dependency surface area minimal, but also indicating a very low-level implementation style (raw ADO.NET SQL, raw HttpWebRequest).
  • System.Web is a hard migration blocker: ASP.NET Web Forms and System.Web are not available in .NET Core or any .NET 5+ runtime; migration requires either rewriting to ASP.NET Core (Razor Pages / MVC) or using the third-party YARP + Windows Compatibility Pack approach.
  • No logging framework: There is no Serilog, NLog, log4net, or equivalent library present; the application performs no structured logging, which will complicate observability in a cloud environment.
  • No dependency injection container: The application uses no DI framework; all dependencies (SQL connections, configuration values) are resolved inline within code-behind files, which increases coupling and testing difficulty.

Caching Strategy

No caching layer is present. Every page load executes fresh SQL queries against SQL Server:

  • BindLoanProducts() runs a SELECT on every non-postback page load (!IsPostBack).
  • BindLoanHistory() runs a SELECT TOP 25 on every non-postback load and again after each loan submission.

There are no [OutputCache] attributes, no HttpRuntime.Cache usage, no distributed cache (Redis, Memcached), and no second-level query cache.

Data Ownership Boundaries

The application uses a single shared SQL Server database (ZavaBankDB). There is only one service in this deployment, so there are no cross-service data access concerns within this codebase. However, the database is likely shared with other ZavaBank systems (the SQL Server hostname sqlserver and the database name ZavaBankDB suggest a shared instance), which means the application has no isolated data store boundary.

Cross-service data flow is outbound-only via HTTP: after a loan application is persisted locally, an XML payload is also sent to the external LoanOriginationApi. There is no inbound data synchronisation mechanism.

Read/write pattern: The portal is primarily read-heavy on the loan product catalogue and write-once on loan applications. No CQRS pattern is implemented; reads and writes share the same ADO.NET connection path and the same database user credentials.
