Data Storage & External Services

The application uses a single SQL Server database (ZavaBankDB) accessed directly via ADO.NET. It queries three main tables: LoanApplications (loan portfolio summary), LoanPayments (delinquency report), and Transactions (daily transaction volume). Authentication is delegated to an external ZavaAuthGateway service configured via appSettings keys (AuthGatewayLoginUrl, AuthGatewayLogoutUrl), using Forms Authentication cookies shared across the platform.

Key Architectural Decisions

  • Code-behind pattern with no service layer: All data access and business logic is implemented directly in ASP.NET Web Forms code-behind classes (Default.aspx.cs), with no separate service or repository abstractions.
  • ADO.NET for direct SQL execution: The application uses raw SqlConnection/SqlCommand/SqlDataAdapter without an ORM, executing parameterized SQL queries for all reporting data.
  • Externalized authentication via redirect: Rather than handling authentication internally, the app redirects unauthenticated users to a separate ZavaAuthGateway service and relies on shared Forms Authentication cookies.

There are no DTO or contract classes in this project. Communication Patterns

Synchronous only: All communication is synchronous. The application makes blocking ADO.NET calls to SQL Server for each report section on every page load or postback. There is no asynchronous processing, message queue integration, or background job infrastructure.

Authentication via redirect: The application delegates authentication to ZavaAuthGateway using HTTP redirects and shared Forms Authentication cookies (.ZAVAAUTH). No OAuth2, JWT, or token-based auth is implemented. The auth gateway URL is hardcoded in web.config appSettings.

No resilience patterns: There are no retry policies, circuit breakers, timeouts, or fallback mechanisms configured for either database access or authentication gateway calls. A SQL Server outage will result in unhandled exceptions surfacing to the user.

No service discovery: External URLs (auth gateway, SQL Server) are hardcoded in web.config. There is no service registry, DNS-based discovery, or load balancing.

Security posture: Forms Authentication is enforced for all pages except Login.aspx and Logout.aspx. However, the machineKey in web.config contains hardcoded cryptographic keys (validationKey, decryptionKey), which is a critical security risk in shared or cloud environments. Transport security (HTTPS/TLS) is not configured in the application — it relies on the host (XSP4/Mono) or a reverse proxy. There is no role-based authorization ([Authorize(Roles=...)] or <allow roles="...">) beyond the blanket <deny users="?"/> rule.

Primary Workflows

Workflow 1: Authenticated Access to Reporting Dashboard

When an unauthenticated user navigates to the dashboard, they are redirected to the ZavaAuthGateway login page. After the gateway authenticates the user and sets the .ZAVAAUTH Forms Authentication cookie, the user is redirected back to the dashboard, which loads the default 7-day reporting window automatically.

Steps:

  1. User navigates to /Default.aspx
  2. ASP.NET Forms Authentication middleware detects missing or invalid cookie
  3. User is redirected to /Login.aspx?ReturnUrl=/Default.aspx
  4. Login page renders a link to the ZavaAuthGateway login URL (from appSettings/AuthGatewayLoginUrl)
  5. User clicks the link and authenticates via ZavaAuthGateway
  6. ZavaAuthGateway sets the .ZAVAAUTH cookie and redirects back to /Default.aspx
  7. Dashboard loads with default date range (last 7 days) and renders all three report sections

Workflow 2: Run Reports for a Date Range

The primary user-facing workflow: a manager selects a start and end date, submits the filter, and the dashboard refreshes all three reporting sections with data for the selected range.

Steps:

  1. Manager views the dashboard (pre-populated with the last 7 days)
  2. Manager optionally modifies Start Date and End Date fields
  3. Manager clicks Run Reports button (triggers ASPX postback via UpdatePanel)
  4. Server validates the date range (both dates must parse; end must be ≥ start)
  5. If invalid: status label shows "Enter valid date range." — no queries run
  6. If valid: all three data methods execute in sequence against SQL Server:
    • BuildLoanSummary() — loan portfolio by status
    • BindDelinquency() — overdue payments (top 50)
    • BindDailyVolume(start, end) — transaction volume by day
  1. Dashboard sections re-render with fresh data; status label shows "Reports refreshed."
  2. Selected date range is stored in ViewState for subsequent postbacks

Workflow 3: User Sign-Out

Manager clicks Logout, which invalidates the auth cookie and redirects to the ZavaAuthGateway logout endpoint.

Steps:

  1. Manager navigates to /Logout.aspx
  2. FormsAuthentication.SignOut() clears the .ZAVAAUTH cookie
  3. User is redirected to appSettings/AuthGatewayLogoutUrl (ZavaAuthGateway's logout page)

Cross-Service Data Flows

ZavaReportDashboard does not aggregate data from multiple services via API calls. All three reporting queries go directly to a single shared SQL Server database (ZavaBankDB). The three queries are independent and execute sequentially:

  1. Loan Portfolio Summary: reads LoanApplications — groups by Status, sums RequestedAmount and ApprovedAmount
  2. Delinquency Report: reads LoanPayments — filters for records where DueDate < today and Status != 'Paid', ordered by oldest due date first, limited to 50 rows
  3. Daily Transaction Volume: reads Transactions — filters by the user-supplied date range, groups by calendar day, sums absolute transaction amounts

There is no circuit breaker, retry, or fallback behavior. If SQL Server is unavailable, all three queries fail with unhandled exceptions — there is no graceful degradation (e.g., showing cached data or a partial report).

Business Rules & Decision Logic

Validation Rules:

  • Date range validation: Both txtStartDate and txtEndDate must parse as valid DateTime values, and end date must be greater than or equal to start date. If either condition fails, a status message is shown and no database queries are executed.

Decision Logic:

  • Authentication gate: Every request to Default.aspx checks Request.IsAuthenticated. Unauthenticated users are unconditionally redirected to the login page before any business logic executes.
  • Delinquency filter: A payment is considered delinquent if DueDate < GETDATE() AND Status != 'Paid'. Only the oldest 50 records are surfaced (business decision: show the most critical overdue items).
  • Transaction volume: Amounts use ABS(Amount) so that both debits and credits contribute positively to the daily volume figure.

State Transitions:

  • No state mutations occur — ZavaReportDashboard is read-only. All entity state (loan status, payment status) is managed by upstream services.

Authorization:

  • ASP.NET <deny users="?"/> enforces that no page (except Login and Logout) is accessible without authentication. There is no role-based authorization — any authenticated user can view all reports regardless of their role.

Error Handling:

  • Date validation errors are surfaced as a status label message (soft error).
  • SQL Server connectivity failures result in unhandled SqlException propagating to the user (hard error — no graceful fallback).
  • No business exception types, compensating actions, or dead-letter handling are defined.

Audit/Logging:

  • No audit trail or business event logging is implemented. There is no record of which manager ran which report or when.

Computed Values:

  • Loan Portfolio Summary computes COUNT(*) as application count and SUM(RequestedAmount) / SUM(ApprovedAmount) as totals — all computed at query time, not cached.
  • Daily Transaction Volume computes COUNT(*) as transaction count and SUM(ABS(Amount)) as total volume, grouped by calendar day.

Version & Compatibility Risks

The application is built on .NET Framework 4.8, which is in long-term servicing mode (supported but no new features). ASP.NET Web Forms is not available in .NET 5 or later — migration to .NET requires replacing Web Forms with ASP.NET Core MVC, Razor Pages, or Blazor. The use of System.Data.SqlClient (the legacy namespace bundled in the framework) should be replaced with Microsoft.Data.SqlClient (a separate NuGet package) when migrating to .NET Core/5+, as the old namespace is not supported cross-platform. There are no third-party dependencies to update, but the core platform itself is the primary risk.

Notable Observations

  • No third-party NuGet packages: The project is entirely self-contained using only .NET Framework BCL assemblies — this simplifies dependency auditing but limits modernization tooling options.
  • ADO.NET with raw SQL strings: Without an ORM (Entity Framework, Dapper), schema changes require manual SQL maintenance across multiple code-behind methods.
  • Web Forms is a migration blocker: ASP.NET Web Forms has no direct equivalent in .NET 5+; moving to a modern runtime requires architectural changes to the presentation layer.
  • Static connection string with embedded credentials: web.config stores SQL Server credentials in plain text, which is a security concern in cloud deployments.

Data Architecture & Persistence Layer

ZavaReportDashboard uses ADO.NET (no ORM) to query a single SQL Server database (ZavaBankDB) across three tables — LoanApplications, LoanPayments, and Transactions — all accessed read-only for reporting purposes.

Key Repository Methods

There are no repository classes or interfaces in this project. Data access is implemented directly as private methods in Default.aspx.cs:

ServiceMethodSQL / Purpose
Default.aspx.csBuildLoanSummary()SELECT Status, COUNT(*) AS ApplicationCount, SUM(RequestedAmount), SUM(ApprovedAmount) FROM LoanApplications GROUP BY Status ORDER BY Status — returns HTML table string
Default.aspx.csBindDelinquency()SELECT TOP 50 PaymentID, ApplicationID, AccountID, DueDate, PaymentAmount, Status FROM LoanPayments WHERE DueDate < GETDATE() AND Status <> 'Paid' ORDER BY DueDate ASC — binds to GridView
Default.aspx.csBindDailyVolume(DateTime s, DateTime e)SELECT CAST(TransactionDate AS DATE) AS TransactionDay, COUNT(*) AS TransactionCount, SUM(ABS(Amount)) AS TotalVolume FROM Transactions WHERE TransactionDate >= @s AND TransactionDate < DATEADD(DAY,1,@e) GROUP BY CAST(TransactionDate AS DATE) ORDER BY TransactionDay — binds to GridView

All methods use SqlConnection + SqlDataAdapter + DataTable with no connection pooling configuration beyond ADO.NET defaults.

Caching Strategy

No caching is implemented. Each page load or postback triggers fresh ADO.NET queries to SQL Server for all three report sections. There is no use of ASP.NET Output Caching, HttpRuntime.Cache, MemoryCache, or any external cache provider (Redis, Memcached).

This means every user interaction results in three sequential database round-trips, which is a performance risk under concurrent usage.

Data Ownership Boundaries

ZavaReportDashboard is a read-only reporting consumer of a shared SQL Server database. It does not own or manage any of the three tables it reads — LoanApplications, LoanPayments, and Transactions are presumed to be owned and written by other Zava Bank platform services (e.g., a loan origination service, a payment processing service, and a transaction/ledger service).

All three tables are accessed directly via SQL from this application, rather than through an API or messaging layer. This creates tight coupling to the shared database schema — any table or column rename in the owning services will break reporting queries. There is no CQRS separation, read replica, or event-sourced projection; this application queries the operational database directly.
