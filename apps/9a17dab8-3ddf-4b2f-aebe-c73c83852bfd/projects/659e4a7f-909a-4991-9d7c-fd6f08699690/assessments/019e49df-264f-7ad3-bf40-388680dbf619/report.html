Data Storage & External Services

The application persists and queries fraud operations data in a shared SQL Server database named ZavaBankDB. Fraud-specific screens read and update fraud alerts and fraud rules, while authentication relies on both shared session tables in the database and a synchronous HTTP call to the external ZavaAuthGateway WhoAmI.ashx endpoint when only the .ZAVAAUTH cookie is available.

Key Architectural Decisions

  • Uses a classic server-rendered Struts MVC stack instead of a REST API, so browser navigation and form posts are the dominant interaction style.
  • Reads and writes data with inline SQL in action classes and service helpers rather than through an ORM or repository abstraction.
  • Reuses shared platform identity data by validating session tokens against common database tables and, when needed, a central auth gateway.

DTOs & Contracts

The action layer exchanges mostly HTML pages, but several internal DTOs define the view contract. SessionUser represents the authenticated analyst stored in the Struts session map. FraudAlertRecord is the response model used by the queue and detail pages, while FraudRuleRecord is the response and edit model for fraud rule management. Request contracts are implicit in mutable action properties such as sessionToken, alertId, decision, notes, ruleName, severity, thresholdAmount, and timeWindowMinutes, which Struts binds from form or query parameters.

No gateway-level aggregation DTOs, OpenAPI documents, protobuf schemas, or GraphQL schemas were found. Serialization is mostly framework-managed request parameter binding plus JSP expression rendering; the only custom parsing is the manual JSON extraction of sessionToken from the auth gateway response in SsoSessionService.

Communication Patterns

All observed communication is synchronous. Browser requests enter through the Struts filter, flow through AuthTokenInterceptor, and then execute action classes that either query SQL Server directly via JDBC or call SsoSessionService. SsoSessionService performs one outbound HTTP GET to the configured auth gateway URL and sets both connect and read timeouts to 3000 ms. No retry, circuit breaker, bulkhead, or asynchronous messaging framework is present.

Service discovery is not dynamic: the auth gateway URL is a hardcoded property that can only be overridden by environment variables. API availability depends on SQL Server being reachable and, for cookie-based sign-in, on the auth gateway being reachable. Security at the API contract layer is limited to session-token validation in the interceptor; there is no TLS termination configuration, no role-based authorization, and no evidence of fine-grained access checks beyond “authenticated analyst or redirect to login.”

Core Business Workflows

ZavaFraudDetector supports fraud analysts who need to authenticate, review suspicious transactions, record dispositions, and maintain detection rules. The main workflows are centered on manual analyst decisioning rather than automated event processing.

Primary Workflows

Workflow 1: Analyst authentication and queue access

  1. An analyst opens the dashboard or submits a session token through login.action.
  2. The application checks for an existing SessionUser in the Struts session map.
  3. If no session exists, SsoSessionService looks for a token in the request, header, or .ZAVAAUTH cookie.
  4. When only the cookie is present, the service calls the auth gateway to translate that cookie into a session token.
  5. The service validates the token against shared SessionTokens and Users tables and stores the resolved analyst in the session.
  6. After authentication, the queue workflow loads the newest fraud alerts with account and transaction context for review.

Business rules involved: valid active session required for all actions except login and health; inactive or expired tokens return the analyst to the login page.

Workflow 2: Alert review and decisioning

  1. An analyst selects a queued alert from the flagged queue.
  2. The detail workflow loads the full alert context, including severity, description, amount, and related account.
  3. The analyst submits an approval or rejection decision with optional notes.
  4. The application maps any non-Approved decision to Rejected, appends the analyst username to the resolution notes, and updates the alert record timestamps.
  5. The analyst is redirected back to the queue to continue triage.

Business rules involved: alertId and decision must be present before an update occurs; only Approved preserves that exact final state, while all other values become Rejected.

Workflow 3: Fraud rule maintenance

  1. An analyst browses the rule catalog from rules.action.
  2. The analyst opens an existing rule or starts a new rule draft from ruleEdit.action.
  3. On save, the application verifies that a rule name and severity are present.
  4. Threshold amount and time-window fields must parse as numbers if supplied.
  5. The application inserts a new rule or updates the selected existing rule, then returns the analyst to the rules list.

Business rules involved: rule name and severity are mandatory; malformed numeric thresholds or time windows block persistence; delete requests remove the targeted rule immediately.

Cross-Service Data Flows

The main cross-service business composition happens during authentication. Cookie-based sign-in starts in ZavaFraudDetector, calls ZavaAuthGateway to recover a usable session token, then validates that token against shared identity tables in SQL Server before granting access to the fraud dashboard. Queue and detail views similarly compose fraud-owned alert data with platform-owned account and transaction data through direct SQL joins.

Fallback behavior is simple and business-visible. If the auth gateway is unavailable or returns a non-200 response, the analyst cannot be auto-signed-in from the cookie and must retry or provide a direct session token. If a queue or detail query fails, the analyst sees an error or is redirected away from the requested record rather than receiving partial data.

Business Rules & Decision Logic

  • Protected workflows require an authenticated analyst session; only login and health bypass the auth interceptor.
  • Queue processing shows only alerts whose status is New or InReview, prioritizing the newest items first.
  • Alert resolution normalizes analyst choices into two final states: Approved or Rejected.
  • Resolution notes always include the acting analyst name to preserve a lightweight audit trail.
  • Rule maintenance enforces required rule name and severity values, plus numeric parsing for threshold and time-window inputs.
  • Persistence and error handling are request-scoped with direct JDBC operations; there is no explicit transactional annotation, saga, or compensating workflow logic in the repository.

Data Ownership Boundaries

The service operates against a shared relational store rather than a database-per-service model. Fraud-specific writes target FraudAlerts and FraudRules, but the queue and detail views also join against Accounts and Transactions, and authentication reads SessionTokens and Users. That means the service depends on platform-owned tables for both identity and financial context, with direct SQL as the cross-context access mechanism.

Read and write behavior is request-scoped and synchronous. There is no CQRS split, no background projection, and no evidence of transaction annotations or saga coordination. The only non-database data flow is the auth gateway HTTP lookup that helps derive a session token before the service performs its own database validation.
