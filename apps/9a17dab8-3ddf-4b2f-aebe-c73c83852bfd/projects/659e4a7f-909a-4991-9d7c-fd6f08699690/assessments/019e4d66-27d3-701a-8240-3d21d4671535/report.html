Data Storage & External Services

The application uses a single Microsoft SQL Server instance (ZavaBankDB) as its primary data store, accessed via raw JDBC through FraudConnectionFactory. Tables referenced include FraudAlerts, Transactions, Accounts, Users, SessionTokens, FraudRules, and FraudRuleConditions. There is no ORM layer; all queries are constructed as hand-written SQL using PreparedStatement. An external Auth Gateway (a .NET service exposing WhoAmI.ashx) is called over HTTP to resolve .ZAVAAUTH SSO cookies into session tokens; this is the only external HTTP dependency.

Key Architectural Decisions

  • Plain JDBC without ORM: All database interactions use FraudConnectionFactory.openConnection() and raw PreparedStatement calls; there is no JPA, Hibernate, or Spring Data layer, making every action class responsible for its own SQL.
  • Struts 2 interceptor-based authentication: A custom AuthTokenInterceptor wraps every action to validate sessions, delegating SSO token resolution to SsoSessionService which checks the SQL Server SessionTokens table and, if needed, calls the external Auth Gateway.
  • Configuration via properties file with environment variable overrides: FraudConfig reads frauddetector.properties from the classpath and allows any property to be overridden by a corresponding environment variable (e.g. DB_HOST, AUTH_GATEWAY_WHOAMI_URL), supporting both local and containerised deployment.

DTOs & Contracts

The application uses two plain JavaBean transfer objects to carry data between action classes and JSP views:

  • FraudAlertRecord — Response model used by FlaggedQueueAction and TransactionDetailAction to expose fraud alert data to JSP views. Contains alert metadata (ID, type, severity, status, description, dates) and denormalised fields from joined tables (account number, transaction amount). Not immutable; fields are set via JDBC result set mapping. No serialization annotations.
  • FraudRuleRecord — Response model used by RuleListAction and RuleEditAction to expose fraud rule data and conditions to JSP views. Not immutable. No serialization annotations.
  • SessionUser — Minimal authentication context object stored in the HTTP session under the key zavaFraudDetector.user; carries userId (int) and username (String) resolved from the Users and SessionTokens tables.

There are no OpenAPI/Swagger specifications, protobuf schemas, GraphQL schemas, or formal API contracts. All "contracts" are implicit HTML form field names matched by Struts 2's OGNL property binding.

Communication Patterns

Synchronous (inbound — browser to application):

All client interactions are synchronous HTTP request/response cycles. The Struts 2 StrutsPrepareAndExecuteFilter intercepts all requests (/*), dispatches to the matching action class, executes business logic (including direct JDBC calls to SQL Server), and returns a full JSP HTML page. There is no REST/JSON API, no AJAX, and no WebSocket communication.

Synchronous (outbound — application to Auth Gateway):

SsoSessionService makes a direct HttpURLConnection HTTP GET to the .NET Auth Gateway at {auth.gateway.whoami.url}/WhoAmI.ashx, forwarding the .ZAVAAUTH cookie. Connect timeout: 3,000 ms; read timeout: 3,000 ms. No retry logic or circuit breaker is implemented — if the Auth Gateway is unavailable the token resolution silently fails and the user is redirected to the login page.

Synchronous (outbound — application to SQL Server):

All data access is synchronous JDBC via FraudConnectionFactory.openConnection(). There is no connection pool — every action opens a new DriverManager connection on each request. No retry, timeout configuration, or transaction management beyond auto-commit.

Asynchronous / Messaging: None. No message queue, event bus, or pub/sub integration exists.

Service Discovery: None. The Auth Gateway URL is hardcoded via the auth.gateway.whoami.url property/environment variable. SQL Server host/port are similarly static. There is no Eureka, Consul, or Kubernetes-native DNS-based service discovery.

API Gateway: None. The application is deployed directly as a WAR with no API gateway in front of it.

Resilience Patterns: None. No circuit breaker (Resilience4j, Hystrix), retry policy, bulkhead, or rate limiter is configured.

Security Posture:

Session-based authentication is enforced by AuthTokenInterceptor on all actions except /login.action and /health.action. Token validation checks the SessionTokens table in SQL Server (and optionally calls the Auth Gateway). There is no HTTPS/TLS configuration in the application itself — TLS must be handled externally by the servlet container or a reverse proxy. There is no OAuth2, OIDC, or JWT implementation. There is no CSRF protection — Struts 2's built-in token mechanism is not configured. There is no role-based access control (RBAC) — any authenticated user can perform all actions including rule creation, deletion, and alert decision. Authorization is binary: authenticated or not authenticated.

Primary Workflows

Workflow 1: Analyst Authentication

An analyst opens the application and must be authenticated before accessing any protected page. Authentication can happen via three paths: a session token query parameter, an X-Session-Token/Authorization: Bearer HTTP header, or an existing .ZAVAAUTH SSO cookie issued by the .NET Auth Gateway.

Steps:

  1. Browser issues an HTTP request to any protected action.
  2. AuthTokenInterceptor checks whether a SessionUser is already stored in the HTTP session.
  3. If no session user is present, SsoSessionService.resolveSessionUser attempts to extract a token from the request (parameter → header → SSO cookie).
  4. If a token is found directly, it is validated against the SessionTokens table in SQL Server (IsActive = 1, ExpiresAt > now()).
  5. If only an SSO cookie is found, SsoSessionService calls the Auth Gateway's WhoAmI.ashx endpoint (3-second timeout) to exchange the cookie for a session token, then validates the resolved token against SQL Server.
  6. On success, a SessionUser (userId, username) is stored in the HTTP session and the original action proceeds.
  7. On failure (no valid token, Auth Gateway unreachable, token expired), the analyst is redirected to the login page.

Workflow 2: Review Fraud Alert Queue

The fraud alert queue is the primary working view for analysts. It shows the 100 most recently created alerts with status New or InReview.

Steps:

  1. Authenticated analyst navigates to /flaggedQueue.action.
  2. FlaggedQueueAction queries FraudAlerts (joined with Accounts and Transactions) for alerts with status New or InReview, ordered by creation date descending, limited to 100 rows.
  3. Each alert is mapped to a FraudAlertRecord carrying: alert ID, transaction ID, alert type, severity, status, description, created date, account number, and formatted transaction amount.
  4. The JSP view renders the queue as a table; the analyst's username (from session) is displayed in the header.

Workflow 3: Approve or Reject a Fraud Alert

An analyst reviews the details of a single alert and records a decision (Approved or Rejected) with optional resolution notes.

Steps:

  1. Analyst clicks an alert in the queue, navigating to /transactionDetail.action?alertId={id}.
  2. TransactionDetailAction loads the full alert record from SQL Server.
  3. Analyst reviews the alert details and submits the decision form to /alertDecision.action.
  4. AlertDecisionAction validates that alertId and decision are non-null and non-empty.
  5. The alert status is set to Approved or Rejected based on the decision field value.
  6. A resolution string is composed: optional analyst notes plus "Decision by {username}".
  7. SQL Server FraudAlerts row is updated: Status, ResolutionNotes, ResolvedDate = GETDATE(), ModifiedDate = GETDATE().
  8. Analyst is redirected back to the flagged queue.

Workflow 4: Manage Fraud Detection Rules

Analysts (or administrators) can view, create, edit, and delete the fraud detection rules stored in SQL Server.

Steps — View rules:

  1. Analyst navigates to /rules.action.
  2. RuleListAction retrieves all rows from FraudRules ordered by RuleID.
  3. Rules are rendered in a table with name, description, severity, threshold amount, and time window.

Steps — Create or edit a rule:

  1. Analyst navigates to /ruleEdit.action (new) or /ruleEdit.action?ruleId={id} (edit).
  2. RuleEditAction loads the existing rule from SQL Server if ruleId is provided.
  3. Analyst submits the form to /ruleSave.action.
  4. RuleEditAction.save validates: rule name and severity are required; threshold amount and time window must be valid numbers if provided.
  5. If ruleId is absent, a new row is inserted into FraudRules.
  6. If ruleId is present, the existing row is updated.
  7. Analyst is redirected to the rules list.

Steps — Delete a rule:

  1. Analyst submits a delete request to /ruleDelete.action?ruleId={id}.
  2. RuleDeleteAction deletes the row from FraudRules by primary key.
  3. Analyst is redirected to the rules list.

Cross-Service Data Flows

ZavaFraudDetector is a monolith with no inter-service aggregation. All data is read from a single SQL Server database instance. The only cross-service data flow is the outbound call to the Auth Gateway during SSO cookie resolution:

  • Auth Gateway → ZavaFraudDetector: When a .ZAVAAUTH cookie is present but no direct session token, SsoSessionService sends an HTTP GET to WhoAmI.ashx forwarding the cookie. The Auth Gateway returns a JSON object containing a sessionToken field. ZavaFraudDetector extracts this token and validates it against its own SessionTokens table copy.
  • Fallback: If the Auth Gateway does not respond within 3 seconds or returns a non-200 status, the token resolution silently fails. The analyst is redirected to the login page (no partial result is shown).

State Transitions

FraudAlert lifecycle:

                                        → Rejected

State changes are written directly to the Status column in SQL Server. The application does not enforce a strict state machine; AlertDecisionAction will update the status unconditionally (including overwriting an already-resolved alert if submitted again).

Transaction Management

No @Transactional annotation or explicit JDBC transaction management is used. All SQL operations execute in auto-commit mode. Multi-statement operations (e.g., rule INSERT followed by condition inserts) are not wrapped in a transaction, creating a risk of partial writes.

Authorization

Authentication is binary: authenticated session users can perform all actions with no role differentiation. Any logged-in analyst can approve alerts, delete rules, and access all data. There is no RBAC, no resource ownership check, and no separation between analyst and administrator roles.

Error Handling

All action classes catch java.sql.SQLException and add an action error message via addActionError(...). No exceptions propagate to the Struts 2 framework or the user as stack traces. However, after adding an error message, most actions still return SUCCESS (for mutations) or INPUT (for forms), which may silently ignore partial failures from the user's perspective.

Audit Trail

No audit trail, change log, or business event log is implemented. The FraudAlerts table records ResolvedDate and ModifiedDate on decision actions, and FraudRules records CreatedDate and ModifiedDate on insert/update. The resolving analyst's username is appended to ResolutionNotes as a human-readable string. There is no structured audit log or external audit service.

Version & Compatibility Risks

Apache Struts 2 (v2.5.30) is a significant migration risk. The Struts 2.5.x branch is no longer actively maintained and has a long history of critical remote code execution vulnerabilities (e.g. CVE-2017-5638, CVE-2018-11776, CVE-2023-50164). Version 2.5.30 is not the latest even within the 2.5 branch, and the entire Struts 2 framework is considered legacy for new cloud-native deployments. The Servlet API 4.0.1 targets the Java EE 8 specification; migrating to Jakarta EE 9+ would require updating package imports from javax.servlet to jakarta.servlet. The mssql-jdbc 12.8.1 driver is current and compatible with SQL Server on Azure (Azure SQL Database, Azure SQL Managed Instance), presenting no immediate compatibility risk.

Notable Observations

  • No logging framework declared: The project declares no SLF4J, Log4j, or Logback dependency; Struts 2 bundles its own logging via commons-logging, but application-level structured logging is absent — a gap for production observability on Azure.
  • No dependency injection framework: The codebase uses manual object creation (e.g. new SsoSessionService() inside action classes) rather than a DI container, making unit testing and refactoring harder.
  • No security library: Authentication is implemented by hand via raw JDBC queries and HTTP calls; there is no dedicated security framework (e.g. Spring Security, OIDC/OAuth2 library), which increases the risk surface.
  • Minimal dependency footprint: With only 3 declared dependencies the project relies heavily on the Struts 2 transitive dependency tree (which includes OGNL, XWork, Freemarker, etc.) — actual runtime classpath is considerably larger than declared dependencies suggest.

Caching Strategy

No caching is implemented. There is no in-memory cache, no distributed cache, no second-level ORM cache (since no ORM is used), and no session-level result caching. Every page request issues fresh JDBC queries to SQL Server. No cache annotations (e.g. @Cacheable), no Redis/EhCache/Caffeine dependency, and no HTTP Cache-Control headers are configured.

Data Ownership Boundaries

The application follows a shared monolithic database pattern: all tables reside in the single ZavaBankDB SQL Server instance and are accessed directly by a single WAR deployment. There are no bounded contexts, schema separations, or service-level data isolation boundaries. All data access is performed by one application component with a single database credential set.

Cross-service data access is limited to the outbound call to the .NET Auth Gateway (WhoAmI.ashx), which resolves .ZAVAAUTH SSO cookies. The Auth Gateway does not directly share the same database; it maintains its own session token store which is mirrored into the SessionTokens table in ZavaBankDB for server-side validation.

Read/write patterns are straightforward OLTP: no CQRS, no event sourcing, no read replicas, and no eventual consistency mechanisms.
