Data Storage & External Services

The application does not manage its own persistent data store. ACH file state is tracked through file-system directories (/shared/ach-incoming, /shared/ach-incoming/processed, /shared/ach-incoming/error). Transaction data is forwarded via HTTP POST to an external Ledger API (http://zavaledger:8080/api/ledger/ach). Processing outcome events are published to a RabbitMQ broker: success events go to the zava.events topic exchange with routing key ach.processed, and failure events go to the zava.dlx dead-letter exchange with routing key ach.failed. The mssql-jdbc driver is included as a dependency but is not directly invoked in the current codebase.

Key Architectural Decisions

  • File-based ingestion with polling: The service continuously polls a shared directory for new ACH files rather than consuming from a queue, providing simple integration with file-drop-based upstream systems.
  • Properties + environment variable configuration: All connection parameters (RabbitMQ, Ledger URL, file paths) are loaded from a classpath properties file and overridable via environment variables, enabling container-friendly configuration management.
  • Synchronous HTTP + fire-and-forget RabbitMQ: Each transaction entry is posted synchronously to the Ledger API before the corresponding event is published to RabbitMQ, ensuring events are only emitted after a successful (or failed) ledger post.

Communication Patterns

Synchronous — HTTP POST to Ledger API:

The application calls the Ledger API using the Java built-in java.net.HttpURLConnection. A single HTTP POST is made per ACH entry within a file. Connect and read timeouts are configurable via http.timeout.ms (default 15 000 ms). There is no retry logic, circuit breaker, or fallback: a non-2xx response or connection exception marks the entry as failed and the entire file is moved to the error directory.

Asynchronous — RabbitMQ events:

Events are published to RabbitMQ using the amqp-client library. A new TCP connection and channel are opened for every publish call and closed immediately after — there is no connection pooling or channel reuse. The exchange is declared as topic and durable. Publish is fire-and-forget: exceptions are caught and logged but not retried. Two exchanges are used: zava.events (routing key ach.processed) for successes and zava.dlx (routing key ach.failed) for failures.

Service discovery:

No service discovery mechanism is present. The Ledger API URL and RabbitMQ host are hardcoded in achprocessor.properties and overridable via environment variables (LEDGER_URL, RABBITMQ_HOST, etc.).

Resilience:

No circuit breaker, retry policy, bulkhead, or timeout escalation is implemented beyond the single fixed HTTP connect/read timeout (15 s). A single transactional post failure causes the entire ACH file to be routed to the error directory.

Security posture:

No authentication, TLS, or authorization is configured for either the outbound Ledger API calls or the RabbitMQ connection. The Ledger API is called over plain HTTP. RabbitMQ credentials (guest/guest by default) are stored in the properties file; environment-variable overrides are supported but no secret management integration is present. No inbound API surface exists to secure.

Primary Workflows

Workflow 1: ACH File Batch Processing

This is the core and only business workflow. The service continuously polls a shared directory and, upon finding a NACHA file, executes the following end-to-end processing sequence:

Steps:

  1. Discover: Poll /shared/ach-incoming every 5 seconds (configurable via ach.poll.interval.ms). Skip sub-directories named processed or error.
  2. Parse: Read the NACHA flat file line by line. Extract:
    • Batch header (record type 5): batch number, company name, effective date.
    • Entry detail records (record type 6): transaction code, routing number, account number, amount in cents, individual ID, name, trace number.
    • Batch control (record type 8): entry count, total debit, total credit.
    • If no entry detail records are found, the file is considered invalid.
  1. Post to Ledger (per entry): Send an HTTP POST to the Ledger API with a JSON payload containing: source, file name, batch number, routing number, account number, amount in cents, and trace number. A 2xx HTTP response indicates success; any other status code or connection exception marks the entry as failed.
  2. Publish processing event (per entry):
    • On successful Ledger post → publish ach.processed event to the zava.events exchange.
    • (Failure events are published at file-level, not entry-level.)
  1. Resolve file outcome:
    • If all entries posted successfully → move file to /shared/ach-incoming/processed (no failure event published).
    • If any entry failed → move file to /shared/ach-incoming/error and publish a single ach.failed event to the zava.dlx dead-letter exchange.
  1. Handle parse failure:
    • If the file cannot be parsed → move file to /shared/ach-incoming/error and publish ach.failed with reason parse_error.

Business Rules Involved: See Business Rules & Decision Logic section.

Cross-Service Interactions:

  • Calls Ledger API synchronously per entry.
  • Publishes to RabbitMQ asynchronously after each entry or on file-level failure.

Cross-Service Data Flows

ZavaACHProcessor is a one-directional data pipeline with no upstream data retrieval:

  • ZavaACHProcessor → Ledger API: Each AchEntry is serialised to a JSON payload and HTTP-posted to the Ledger. The Ledger API is the authoritative destination for committed ACH data. ZavaACHProcessor receives only an HTTP status code in response; no data is read back and stored.
  • ZavaACHProcessor → RabbitMQ: On each successful Ledger post, an ach.processed event containing file name, batch number, trace number, amount, and timestamp is published to zava.events. On any file-level failure, a single ach.failed event with the failure reason is published to zava.dlx. Downstream consumers of these events are unknown to this service; no response is awaited.

There is no gateway aggregation, no data fetched from other services, and no circuit breaker pattern. The only fallback is logging the failure and classifying the file or entry as errored.

Business Rules & Decision Logic

Validation Rules

  • Non-empty entry set: A NACHA file must contain at least one record type 6 (Entry Detail) line. Files with no entries are treated as invalid and moved to the error directory.
  • Fixed-width field slicing: All NACHA record fields are extracted by character position per the NACHA specification (e.g., transaction code at positions 1–3, routing number at 3–11, account number at 12–29, amount in cents at 29–39, individual name at 54–76, trace number at 79–94). No length validation beyond safe-slice bounds checking is applied.
  • Directory filter: Only files directly in the incoming directory are processed. Sub-directories (including processed and error) are skipped by name comparison.
  • File type filter: Any file that is not a directory and whose name is not processed or error is eligible for processing — there is no file extension filter.

Error Handling

  • Parse errors: Caught via try/catch; file moved to error directory; ach.failed event published with reason parse_error.
  • Ledger HTTP failures: Per-entry failure increments a counter; processing continues for remaining entries. At file-level, any non-zero fail count triggers error disposition.
  • RabbitMQ publish failures: Caught and logged; no retry or compensating action. Events may be silently lost if the broker is unavailable.
  • File move failures: Logged; the file remains in the incoming directory and may be reprocessed on the next poll cycle, risking duplicate processing.

Cross-Cutting Concerns

  • No transaction management: There is no distributed transaction, saga, or two-phase commit coordinating the Ledger API call and the RabbitMQ publish. It is possible for a transaction to be posted to the Ledger but the corresponding ach.processed event to fail to publish (or vice versa on error).
  • No idempotency guard: If a file move fails after a successful Ledger post, the file will be reprocessed on the next poll, potentially causing duplicate Ledger entries.
  • Audit / logging: All significant state transitions are logged to stdout with the prefix [ZavaACHProcessor]. No structured audit trail, change tracking, or correlation IDs are used.
  • Authorization: No authentication or authorization is applied. Any process with filesystem access to the incoming directory can trigger processing.

Version & Compatibility Risks

The amqp-client 5.21.0 is a recent release of the RabbitMQ Java client and is compatible with Java 8 through Java 21; however, migrating to Azure requires replacing RabbitMQ with Azure Service Bus or Azure Event Hubs, which would entail adopting the Azure SDK for Java. The mssql-jdbc 12.6.1 is a current release and supports Java 8, but its presence in the dependency list without any usage in the source code suggests it may be an orphaned or placeholder dependency. The application uses Java 8, which reached end of active support in 2019 (OpenJDK) and limits access to modern language features; upgrading to Java 17 or 21 LTS is strongly recommended for Azure deployments.

Notable Observations

  • No web framework: The service exposes no HTTP endpoints; it is a pure background processor. The java.net.HttpURLConnection used to call the Ledger API is the built-in JDK HTTP client with no external library dependency, which is verbose and limited compared to modern HTTP clients (Apache HttpClient, OkHttp, or Java 11's java.net.http.HttpClient).
  • Orphaned SQL Server driver: mssql-jdbc is declared as a dependency but is not imported or used anywhere in Main.java. This may indicate a removed feature or future planned capability; it should be removed if not needed to reduce image size and attack surface.
  • No logging framework: The application uses plain System.out.println for all log output rather than a structured logging library (SLF4J + Logback/Log4j2). This prevents log-level control, structured JSON logging, and integration with Azure Monitor / Application Insights.
  • No observability dependencies: There is no Micrometer, OpenTelemetry, or Application Insights SDK dependency. Metrics, traces, and distributed correlation are unavailable, which will be a gap when deploying to Azure Container Apps or AKS.

Data Architecture & Persistence Layer

ZavaACHProcessor has no ORM layer or managed relational database; its data model consists of two in-memory inner classes used to represent parsed NACHA records during processing, with the file system acting as the durable store and a remote Ledger HTTP API as the system of record for committed transactions.

Entity Model

The application does not use a relational database or ORM. The two in-memory model classes represent parsed records from NACHA-formatted flat files:

Both classes are private static inner classes within Main.java. All fields are raw String types regardless of the underlying NACHA field semantics (numeric amounts, dates, codes). There is no JPA, Hibernate, MyBatis, or any other ORM mapping. No @Entity, @Table, @Id, or persistence annotations exist.

Caching Strategy

No caching layer is present. The application processes each file end-to-end in a single synchronous pass with no in-memory result caching, query caching, or second-level cache. ACH batch and entry objects are created, used, and garbage-collected within the processing of a single file.

Data Ownership Boundaries

The service has a single-node, file-system-based data boundary. There is no shared or replicated database. The only durable state transitions are:

  • Incoming ACH files move from /shared/ach-incoming → /shared/ach-incoming/processed or /shared/ach-incoming/error.
  • Committed transaction records are written exclusively to the external Ledger API's data store; this service holds no local copy.
  • RabbitMQ events are fire-and-forget; no local event store or outbox pattern is used.

Cross-service data access is one-directional: ZavaACHProcessor writes to the Ledger API via HTTP POST and reads nothing back beyond the HTTP response status code.

The accountNumber and routingNumber fields constitute PCI-DSS sensitive payment data. The individualName and individualId fields are PII. No encryption-at-rest, field-level masking, or access control is applied to these values while they are held in memory or logged. Additionally, the log statement in postTransactionToLedger() may write traceNumber values to stdout; structured redaction is absent. The JSON payloads sent to the Ledger API and RabbitMQ contain account and routing numbers in plain text over unencrypted HTTP and AMQP connections.
