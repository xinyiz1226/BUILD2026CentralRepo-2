Data Storage & External Services

The processor reads ACH files from a configured shared directory, moves files to processed/error folders, posts transaction payloads to the ledger HTTP API, and publishes processing outcomes to RabbitMQ topic exchanges.

Key Architectural Decisions

  • Uses a single-process polling loop instead of a web server runtime.
  • Keeps ingestion state in filesystem paths rather than a database.
  • Separates integration channels: synchronous HTTP for ledger writes and asynchronous RabbitMQ events for notifications.

Security posture is minimal in current code: no TLS enforcement, no authentication headers for ledger calls, and no explicit authorization logic in this service.

Core Business Workflows

This service processes inbound ACH batch files and coordinates transaction posting plus operational event publication for downstream banking workflows.

Primary Workflows

Workflow 1: Process ACH file and dispatch transactions

  1. Poll incoming ACH directory and pick the next file.
  2. Parse NACHA batch header, entries, and control records into in-memory domain objects.
  3. For each entry, submit transaction payload to ledger API.
  4. If all entries succeed, publish processed event(s) and move file to processed folder.
  5. If any entry fails or parse fails, publish failure event and move file to error folder.

Workflow 2: Failure handling and operational signaling

  1. Detect parse/post exceptions or non-success ledger responses.
  2. Record failure through standard logs.
  3. Publish ach.failed message for asynchronous handling.
  4. Route original file to error folder for remediation/replay.

Cross-Service Data Flows

The processor composes file-derived ACH batch data with outbound integration contracts by posting each transaction to the ledger API and then publishing result events to RabbitMQ. Ledger response status controls business degradation behavior: success keeps the file in the processed path; any failure triggers ach.failed signaling and file quarantine in the error path.

Business Workflow Sequence

sequenceDiagram participant Ops as "Operations" participant ACH as "ZavaACHProcessor" participant Ledger as "Ledger Service" participant MQ as "RabbitMQ" Ops->>ACH: Drop ACH batch file ACH->>ACH: Parse batch and entries ACH->>Ledger: Submit entry transaction alt All entries accepted Ledger-->>ACH: Success statuses ACH-->>MQ: Publish ach.processed events ACH->>ACH: Move file to processed else Parse or post failure detected Ledger-->>ACH: Error status or exception ACH-->>MQ: Publish ach.failed event Note over ACH: Quarantine file for retry or investigation ACH->>ACH: Move file to error end

Business Rules & Decision Logic

  • A file is considered valid only when at least one ACH entry record is parsed.
  • Processing success requires every entry post to return a 2xx ledger response.
  • Any parse failure or partial post failure forces the whole file into the error path.
  • Event type is derived from outcome: successful entries emit ach.processed; failed files emit ach.failed.
