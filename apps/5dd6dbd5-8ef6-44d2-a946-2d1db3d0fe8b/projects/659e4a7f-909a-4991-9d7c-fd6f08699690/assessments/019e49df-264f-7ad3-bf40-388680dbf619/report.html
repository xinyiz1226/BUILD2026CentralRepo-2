Data Storage & External Services

The application uses a single Microsoft SQL Server instance (ZavaBankDB, port 1433) as its only data store. It reads from and writes to several tables: ComplianceReports (regulatory report records), Transactions and Accounts (transaction data for CTR generation), FraudAlerts (alert data for SAR generation), and SessionTokens + Users (SSO session validation). All database access is performed using plain JDBC via a shared ComplianceConnectionFactory.

An external Auth Gateway (a .NET service exposing WhoAmI.ashx) handles decryption of the .ZAVAAUTH FormsAuth cookie. When a request carries this cookie, SsoSessionService calls the gateway over HTTP to retrieve a session token, which is then validated against the SQL Server SessionTokens table.

Key Architectural Decisions

  • No ORM or connection pooling: All data access uses raw JDBC with manual try-with-resources connection management; no Hibernate, JPA, or connection pool is configured.
  • SSO integration via shared SQL token table: Session authentication is dual-path — direct token lookup in SQL Server, or cookie-based delegation to the .NET auth gateway — avoiding any native Java decryption of .NET FormsAuth tickets.
  • Classic servlet/filter architecture: The application uses a pre-Spring, standard Java EE model with web.xml configuration, Servlet filters for cross-cutting security, and JSP views — no dependency injection framework is present.

DTOs & Contracts

Two internal value objects act as the application's data contracts:

  • SessionUser — immutable (final fields, no setters); represents an authenticated principal carrying userId (int) and username (String). Created by SsoSessionService and stored in the HTTP session. Not serialized to JSON; lives only in server-side session memory.
  • ReportRecord — mutable POJO (conventional getters/setters); represents a single row from the ComplianceReports table. Used as the response model passed to reports.jsp via request attributes. Not serialized to JSON; bound to the JSP view layer only.

No OpenAPI/Swagger specification, protobuf schemas, GraphQL schemas, or JSON serialization configuration (Jackson, Gson, etc.) are present. There is no formal API contract document — the application is a server-rendered web app, not a REST API.

For field-level details and persistence mappings, see data-architecture.md.

Communication Patterns

Synchronous — Outbound HTTP (auth resolution):

SsoSessionService makes a synchronous HTTP GET call to the external Auth Gateway (WhoAmI.ashx) when a .ZAVAAUTH cookie is present. The call uses java.net.HttpURLConnection with a hardcoded 3-second connect timeout and 3-second read timeout. There is no retry policy, circuit breaker (Resilience4j, Hystrix, etc.), or fallback — if the gateway is unreachable, the method returns an empty token and the user is redirected to the login page.

Synchronous — JDBC (database access):

All SQL Server communication is synchronous and blocking JDBC, opened per-request with no connection pool. If the database is unavailable, exceptions are silently swallowed and the affected servlet returns an empty result or redirects.

Service Discovery:

No service discovery mechanism is in place. The Auth Gateway URL is hardcoded in compliance.properties (auth.gateway.whoami.url=http://zava-auth-gateway:8080/WhoAmI.ashx) and resolved via Docker DNS within the compose network.

Asynchronous / Messaging:

No message broker (Kafka, RabbitMQ, Azure Service Bus) or event-driven pattern is used. All communication is synchronous request/response.

Security Posture:

All /reports and /reports/* URL patterns are protected by AuthFilter, which enforces SSO session validation before request processing. The filter checks for a valid SessionUser in the HTTP session; if absent, it attempts to resolve one from the request (token parameter, X-Session-Token header, Authorization: Bearer header, or .ZAVAAUTH cookie via the Auth Gateway). Unauthenticated requests are redirected to /login.

No HTTPS/TLS is configured at the application level — TLS termination is expected to happen at the infrastructure layer (load balancer, reverse proxy). No CSRF protection, no content-security-policy headers, and no role-based authorization (RBAC) beyond the blanket authentication check are implemented. The /health endpoint is publicly accessible with no authentication.

Core Business Workflows

ZavaComplianceReporter enables bank compliance officers to generate, review, and download regulatory reports — specifically Currency Transaction Reports (CTR) and Suspicious Activity Reports (SAR) — filed with FinCEN as part of the Zava Bank platform's regulatory obligations.

Primary Workflows

Workflow 1: User Authentication via SSO

A compliance officer accesses any protected /reports URL. AuthFilter intercepts the request and checks for a valid authenticated session. If no session exists, it attempts to resolve the user from available credentials in the request (explicit token parameter, X-Session-Token header, Authorization: Bearer header, or a .ZAVAAUTH FormsAuth cookie forwarded to the Auth Gateway). On success, the resolved SessionUser is stored in the HTTP session for the duration of the browser session. On failure, the user is redirected to /login.

Steps:

  1. Request arrives at /reports/*
  2. AuthFilter checks HTTP session for SessionUser
  3. If absent: SsoSessionService.resolveSessionUser() is invoked
    • Checks request parameter sessionToken, then X-Session-Token header, then Authorization: Bearer header
    • If .ZAVAAUTH cookie is present: calls Auth Gateway WhoAmI.ashx (3-second timeout) to resolve the token
  1. Resolved token is validated against SessionTokens + Users tables in SQL Server
  2. On valid token: SessionUser stored in HTTP session; request forwarded to target servlet
  3. On invalid/missing token: redirect to /login

Workflow 2: Generate Currency Transaction Report (CTR)

A compliance officer submits a form on the reports page requesting a CTR above a specified transaction threshold. The system queries large transactions from the database, formats a report, and saves it as a new ComplianceReport record.

Steps:

  1. Officer submits POST /reports/generate with reportType=CTR and optional thresholdAmount
  2. GenerateReportServlet validates reportType — rejects non-CTR/SAR values with redirect
  3. Threshold is parsed; defaults to 10,000.00 if absent or invalid
  4. SQL query fetches top 200 transactions where ABS(Amount) >= threshold, joined with Accounts for account numbers, ordered by date descending
  5. Report text is formatted as a plain-text CTR document with header and transaction lines
  6. Report is saved to ComplianceReports with Status = 'Generated', RegulatoryBody = 'FinCEN', FiledBy = username
  7. Officer is redirected to /reports list

Workflow 3: Generate Suspicious Activity Report (SAR)

A compliance officer requests a SAR, which aggregates unresolved high-severity fraud alerts from the fraud detection system.

Steps:

  1. Officer submits POST /reports/generate with reportType=SAR
  2. GenerateReportServlet validates reportType
  3. SQL query fetches top 200 FraudAlerts where Severity IN ('High', 'Critical') OR Status IN ('New', 'InReview'), ordered by creation date descending
  4. Report text is formatted as a plain-text SAR document with header and alert lines
  5. Report is saved to ComplianceReports with Status = 'Generated', RegulatoryBody = 'FinCEN'
  6. Officer is redirected to /reports list

Workflow 4: View Compliance Reports

An authenticated officer views the list of previously generated compliance reports.

Steps:

  1. Officer requests GET /reports
  2. AuthFilter verifies authentication
  3. ReportsServlet queries top 100 ComplianceReports ordered by FilingDate DESC
  4. Report list (ID, type, date, status, filed-by) is passed to reports.jsp for rendering

Workflow 5: Download Compliance Report

An officer downloads the full text of a specific compliance report.

Steps:

  1. Officer requests GET /reports/download?reportId=N
  2. DownloadReportServlet parses the reportId integer parameter; redirects on invalid input
  3. Fetches ReportType and ReportData for the given ID from ComplianceReports
  4. Returns response as text/plain attachment with filename {ReportType}-{reportId}.txt

Cross-Service Data Flows

ZavaComplianceReporter does not call other Zava Bank microservices for business data. All data flows are direct JDBC reads from the shared ZavaBankDB database:

  • CTR generation: reads Transactions (joined with Accounts) owned by the banking/ledger domain
  • SAR generation: reads FraudAlerts owned by the fraud detection domain
  • Authentication: reads SessionTokens and Users owned by the identity/IAM domain

The only outbound service call is to the Auth Gateway (WhoAmI.ashx) for SSO cookie decryption. This call has a 3-second timeout and no retry. If the Auth Gateway is unavailable, cookie-based authentication silently fails and the user is redirected to /login — users with an explicit session token or header are not affected by gateway unavailability.

There are no event-driven flows, message queues, or async callbacks. All cross-context data access is synchronous and tightly coupled to the shared database schema.

Business Rules & Decision Logic

Validation Rules

  • Report type: reportType must be exactly "CTR" or "SAR" (case-insensitive); any other value causes an immediate redirect to /reports with no error message
  • CTR threshold: thresholdAmount must be a valid decimal number; if absent, blank, or non-numeric, it defaults to 10,000.00
  • Authentication: All /reports and /reports/* URLs require an active SessionUser in the HTTP session; unauthenticated requests redirect to /login
  • Report ID: reportId parameter on download must be parseable as an integer; invalid values redirect to /reports

Business Decision Logic

  • CTR data selection: Transactions are included in the CTR if ABS(Amount) >= threshold — both deposits and withdrawals above the threshold are captured; limited to TOP 200 results per report generation
  • SAR data selection: FraudAlerts are included in the SAR if Severity IN ('High', 'Critical') OR Status IN ('New', 'InReview') — the union of high-risk severity AND unresolved status ensures neither dimension alone filters out relevant alerts; limited to TOP 200 results
  • Regulatory body: All reports are automatically attributed to FinCEN (Financial Crimes Enforcement Network); this is hardcoded and not configurable

State Transitions

ComplianceReport lifecycle is minimal — reports are created with Status = 'Generated' and there are no transitions to subsequent states (e.g., "Submitted", "Acknowledged") within this service. Status progression to a submitted or filed state would require external intervention or a future workflow extension.

Cross-Cutting Concerns

  • Transactions: No @Transactional annotation or explicit transaction management. Each JDBC operation opens and closes its own connection. If the INSERT INTO ComplianceReports fails after the data has been read, no compensating action is taken and no error is surfaced to the user.
  • Error handling: All SQLException and data-access exceptions are silently swallowed. If report data cannot be retrieved, an empty or partial report text is saved (the report entry is created regardless). Download attempts for non-existent reports redirect silently to /reports.
  • Audit / logging: No structured audit log is written. The FiledBy column in ComplianceReports records which username generated the report, but no application-level audit trail, change log, or event log is maintained.
  • Authorization: Authentication is enforced (valid session required), but there is no role-based authorization — any authenticated user can generate or download any report, including reports filed by other users.

Version & Compatibility Risks

The project targets Java 8 and Servlet 3.1, both of which are significantly outdated. Java 8 reached its Oracle public support end-of-life in March 2022 and is now in extended support only; migrating to Java 17 or Java 21 (current LTS releases) is strongly recommended for cloud deployments. The javax.servlet-api 3.1.0 and javax.servlet:jstl:1.2 packages use the legacy javax. namespace, which was superseded by Jakarta EE's jakarta. namespace starting with Jakarta EE 9 — any migration to a modern application server (e.g., Tomcat 10+, WildFly 27+) requires a namespace migration. The Microsoft SQL Server JDBC driver (mssql-jdbc 12.8.1.jre8) is current and up to date.

Notable Observations

  • No ORM or connection-pool library declared: The application uses raw JDBC with no Hibernate, JPA, or HikariCP dependency. Every servlet opens its own Connection manually, which is a known scalability and reliability concern in cloud environments.
  • No logging framework declared: There is no SLF4J, Log4j2, or Logback dependency. All error handling silently swallows exceptions (catch (Exception ignored)), making observability and troubleshooting significantly harder.
  • No security library declared: Authentication is implemented from scratch using custom AuthFilter and a SQL-based session token lookup. No Spring Security, OWASP ESAPI, or any other security framework is used.
  • Extremely lean dependency tree: With only 3 declared dependencies, the application is easy to migrate in terms of library compatibility, but the absence of modern infrastructure libraries (connection pooling, logging, security) indicates that foundational cloud-readiness work will be needed before migration.

Caching Strategy

No caching layer is configured. There is no Redis, EhCache, Caffeine, Hibernate second-level cache, or Spring @Cacheable annotation in the project. Every request that requires data results in a new JDBC connection and fresh SQL query against SQL Server. Reference data (e.g., users, account numbers) is re-queried on every request.

Data Ownership Boundaries

The application uses a shared database model — all six tables (ComplianceReports, Transactions, Accounts, FraudAlerts, SessionTokens, Users) reside in a single SQL Server database (ZavaBankDB) that is shared across the entire Zava Bank platform. ZavaComplianceReporter is the logical owner only of ComplianceReports; the remaining tables (Transactions, Accounts, FraudAlerts, SessionTokens, Users) are owned by other Zava Bank services but accessed directly by this service via shared JDBC connections.

Cross-service data access is performed through direct table reads rather than API calls, coupling this service tightly to the platform's shared schema. There is no CQRS separation: the same database is used for both report generation (read-heavy queries across Transactions and FraudAlerts) and report persistence (writes to ComplianceReports).
