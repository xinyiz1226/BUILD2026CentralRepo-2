Data Storage & External Services

The application connects to a shared Microsoft SQL Server instance (ZavaBankDB) using raw JDBC (no ORM). It reads from and writes to five tables: Users, SessionTokens, Transactions, Accounts, FraudAlerts, and ComplianceReports. Connection parameters (host, port, database name, credentials) are supplied via environment variables or compliance.properties. An external .NET Auth Gateway (WhoAmI.ashx) is called over HTTP when a .ZAVAAUTH cookie is present and no direct session token is available, bridging the Java app into the platform-wide SSO scheme.

Key Architectural Decisions

  • Direct JDBC over ORM: All database access uses PreparedStatement through a single ComplianceConnectionFactory, avoiding any ORM dependency to keep the WAR lightweight and compatible with Java 8.
  • Filter-based authentication: AuthFilter intercepts protected URL patterns (/reports, /reports/*) and delegates session resolution to SsoSessionService, centralizing auth logic without a security framework.
  • Environment-variable-first configuration: ComplianceConfig checks environment variables before falling back to compliance.properties, enabling container-friendly deployment without code changes.

Communication Patterns

Synchronous patterns only. The application uses two synchronous outbound communication channels:

  1. JDBC (SQL Server) — All five servlets that touch data open a new JDBC connection on each request through ComplianceConnectionFactory. There is no connection pool, no retry policy, and no circuit breaker. A failure to connect results in a silently swallowed SQLException, with the page either rendering an empty list or returning a static error string in the report body.
  2. HTTP (Auth Gateway) — SsoSessionService calls the .NET WhoAmI.ashx endpoint via HttpURLConnection with a hardcoded 3-second connect/read timeout. There is no retry or circuit-breaker logic; a non-200 response or any exception returns an empty token string and the user is redirected to the login page.

Resilience policies: None. No Resilience4j, Spring Retry, or equivalent library is present. Failure handling is limited to try/catch blocks that suppress exceptions and return empty results.

Service discovery: None. The Auth Gateway URL and SQL Server host/port are configured statically via environment variables or compliance.properties. No Eureka, Consul, or Kubernetes DNS-based discovery is used.

Security posture: The application implements session-token-based authentication through a custom AuthFilter and SsoSessionService. Protected routes (/reports, /reports/*) require a valid session. Token resolution supports query parameters, the X-Session-Token header, Authorization: Bearer headers, and the .ZAVAAUTH cookie bridged via the auth gateway. No TLS/HTTPS is configured at the application layer — TLS termination is expected to happen at an upstream load balancer or ingress. No CSRF protection is implemented on form POST endpoints (/login, /reports/generate). There is no RBAC or role-based authorization; any authenticated user can generate or download any report.

Core Business Workflows

ZavaComplianceReporter enables bank compliance officers to generate and submit regulatory reports — Currency Transaction Reports (CTR) and Suspicious Activity Reports (SAR) — to the financial regulator FinCEN, based on transaction and fraud alert data from the shared Zava Bank platform.

Primary Workflows

Workflow 1: User Authentication via Session Token

A compliance officer accesses the application by providing a session token obtained from the Zava Bank platform SSO. The token may be supplied as a form field, a query parameter, an Authorization: Bearer header, an X-Session-Token header, or resolved from a .ZAVAAUTH cookie via the Auth Gateway.

Steps:

  1. Officer navigates to /login or any protected page under /reports.
  2. AuthFilter checks the HTTP session for an existing authenticated SessionUser. If found, the request proceeds immediately.
  3. If no session user exists, SsoSessionService.resolveSessionUser() attempts to extract a token from the request in priority order: form parameter → X-Session-Token header → Authorization: Bearer header → .ZAVAAUTH cookie.
  4. If a .ZAVAAUTH cookie is found, the service calls the Auth Gateway's WhoAmI.ashx endpoint (3-second timeout) to exchange the cookie for a session token.
  5. The resolved token is validated against the SessionTokens and Users tables: token must be active, not expired, and linked to an active user.
  6. On success, a SessionUser (userId + username) is placed in the HTTP session and the request continues. On failure, the user is redirected to the login page.

Workflow 2: Generate Currency Transaction Report (CTR)

A compliance officer generates a CTR to report all transactions that exceed a regulatory monetary threshold (default: $10,000).

Steps:

  1. Officer submits the report generation form at POST /reports/generate with reportType=CTR and an optional thresholdAmount.
  2. The GenerateReportServlet verifies the session user; unauthenticated requests redirect to /login.
  3. The threshold is parsed as a BigDecimal; if invalid or absent, it defaults to 10,000.00.
  4. Up to 200 transactions where ABS(Amount) >= threshold are fetched from the Transactions table joined with Accounts, ordered by most recent first.
  5. A plain-text report body is assembled, listing each transaction's ID, account number, amount, date, and description.
  6. The report is inserted into ComplianceReports with type CTR, status Generated, filing date (database GETDATE()), the officer's username as FiledBy, and FinCEN as RegulatoryBody.
  7. The officer is redirected to the reports list at /reports.

Workflow 3: Generate Suspicious Activity Report (SAR)

A compliance officer generates a SAR for high-severity or unresolved fraud alerts.

Steps:

  1. Officer submits the form at POST /reports/generate with reportType=SAR.
  2. Authentication check is performed (same as CTR workflow).
  3. Up to 200 fraud alerts where Severity IN ('High', 'Critical') or Status IN ('New', 'InReview') are fetched, ordered by most recent first.
  4. A plain-text report body is assembled, listing each alert's ID, linked transaction ID, severity, status, creation date, and description.
  5. The report is inserted into ComplianceReports with type SAR, same persistence logic as CTR.
  6. The officer is redirected to /reports.

Workflow 4: View and Download Compliance Reports

An authenticated compliance officer views the list of filed reports and optionally downloads the full report text.

Steps:

  1. Officer navigates to GET /reports. AuthFilter enforces authentication.
  2. The 100 most recently filed reports are loaded from ComplianceReports (ReportID, ReportType, FilingDate, Status, FiledBy).
  3. The list is rendered in the JSP report view.
  4. To download a specific report, the officer navigates to GET /reports/download?reportId={id}.
  5. DownloadReportServlet fetches the full ReportData text for the requested ID and streams it as a text/plain attachment.

Cross-Service Data Flows

ZavaComplianceReporter has no peer service integrations or API gateway aggregation patterns. Its only cross-service data flows are:

  • Auth Gateway call: When a .ZAVAAUTH cookie is present, SsoSessionService makes a synchronous HTTP GET to WhoAmI.ashx, passing the cookie. The gateway responds with a JSON payload containing a sessionToken. If the gateway is unreachable or returns a non-200 status, the token is treated as absent and the user must supply an explicit token. This is a best-effort fallback: cookie-based login degrades gracefully.
  • Shared database reads: The service reads Transactions, Accounts, and FraudAlerts directly from the shared SQL Server database. These entities are owned by other Zava Bank services. There is no API contract; the service depends directly on the physical schema. Schema changes in the owning services can break report generation silently.

Business Rules & Decision Logic

Validation Rules

  • Report type: Only CTR and SAR are accepted as reportType. Any other value causes a redirect back to /reports without generating a report.
  • CTR threshold: The thresholdAmount parameter must be parseable as a BigDecimal. If missing or invalid, it defaults to 10,000.00. No upper bound is enforced.
  • Session token validity: A token is valid only if it exists in SessionTokens, is marked IsActive = 1, has ExpiresAt > GETDATE(), and is linked to a Users record with IsActive = 1. All conditions must hold simultaneously.

Decision Logic

  • Token resolution priority: Form parameter → X-Session-Token header → Authorization: Bearer header → .ZAVAAUTH cookie (via Auth Gateway). The first non-empty result wins.
  • Report generation switch: reportType determines which SQL query and report header are used — CTR queries Transactions/Accounts by amount threshold; SAR queries FraudAlerts by severity and status.
  • SAR alert filter: Alerts are included if Severity IN ('High', 'Critical') or Status IN ('New', 'InReview') — the two conditions are OR-combined, so a low-severity alert in InReview status is still included.

State Transitions

  • ComplianceReport lifecycle: All reports are created with Status = 'Generated'. There is no update, review, submission, or archival state managed by this service — the status field exists in the schema but is never updated after insertion.
  • FraudAlert status (read-only): Alerts exist in states New, InReview, and implicitly resolved states. This service only reads alerts; it does not update their status.

Business Constraints

  • Row limits: CTR and SAR queries are hard-capped at 200 rows (TOP 200). Reports are capped at 100 rows in the listing view (TOP 100). These are hardcoded limits with no configurable override.
  • Single-user authorization: Any authenticated user may generate or download any report. There is no role hierarchy, no approval workflow, and no per-report access control.

Cross-Cutting Concerns

  • Transactions: No @Transactional annotations or explicit transaction management exist. Each JDBC operation is its own auto-committed transaction. If the INSERT INTO ComplianceReports succeeds but a subsequent operation fails, there is no rollback.
  • Error handling: All SQLException and general Exception instances in data-access paths are silently swallowed (catch (Exception ignored)). Business errors surface as empty results or redirects, not as user-facing error messages. There is no audit trail of failures.
  • Audit trail: The FiledBy column records the username of the officer who generated each report, providing a minimal audit trail. There is no timestamp beyond FilingDate and CreatedDate.
  • Authorization: Authentication is enforced by AuthFilter on /reports and /reports/*. The /health and /login endpoints are unrestricted. There is no role-based authorization — all authenticated users have full access to all report operations.

Version & Compatibility Risks

javax.servlet-api 3.1.0 and javax.servlet jstl 1.2 are bound to the Java EE namespace (javax.), which was superseded by Jakarta EE (jakarta.) starting with Tomcat 10. Deploying this WAR on Tomcat 10+ will fail unless all javax.servlet references are migrated to jakarta.servlet. The mssql-jdbc 12.8.1.jre8 driver is the latest stable release (2024) and supports Azure SQL Database, so it presents no compatibility risk on its own. However, the jre8 classifier restricts the runtime to Java 8 — moving to Java 11+ would require switching to the jre11 or default (non-classified) artifact.

Notable Observations

  • Extremely lean dependency tree: Only 3 declared dependencies; no ORM, no DI framework, no logging library, and no HTTP client library. All database access relies on raw JDBC and all HTTP outbound calls use the JDK's built-in HttpURLConnection.
  • No logging framework declared: The application has no SLF4J, Log4j, or Logback dependency. Exceptions are silently swallowed (caught with ignored) throughout the codebase, making production diagnostics difficult.
  • No security library: Authentication is implemented manually via servlet filters and direct SQL queries against the SessionTokens table. There is no Spring Security, OWASP ESAPI, or equivalent framework to harden the implementation.
  • Hardcoded credentials in properties file: compliance.properties contains default database credentials (Zava123!) that will be packaged into the WAR unless overridden by environment variables. This is a security concern for cloud deployments.

Data Architecture & Persistence Layer

ZavaComplianceReporter accesses six SQL Server tables through raw JDBC (no ORM), with all schema management delegated to the shared ZavaBankDB database that is also owned and managed by the broader Zava Bank platform.

Caching Strategy

No caching layer is present. Every request to any endpoint that requires data performs a live JDBC query against SQL Server. There is no in-process cache (EhCache, Caffeine), no distributed cache (Redis, Memcached), no HTTP response caching (Cache-Control headers), and no second-level cache configuration. Session data is stored in the Tomcat HTTP session only (in-process memory), which means the application cannot be horizontally scaled without sticky sessions or an external session store.

Data Ownership Boundaries

ZavaComplianceReporter operates against a shared, platform-wide SQL Server database (ZavaBankDB). It does not own or manage the schema for Users, SessionTokens, Transactions, Accounts, or FraudAlerts — those tables are created and maintained by other Zava Bank services. The service has write access only to ComplianceReports.

This shared-database pattern creates tight coupling: any schema change to Transactions, Accounts, or FraudAlerts made by another team can silently break report generation. There is no API layer between this service and the data it reads — it issues direct SQL against tables it does not own, with no versioning or contract boundary.

Cross-service data access is entirely via direct JDBC rather than REST API calls to the owning service. There is no CQRS separation; reads and writes share the same connection pool (absent as they are) and the same database credentials.
