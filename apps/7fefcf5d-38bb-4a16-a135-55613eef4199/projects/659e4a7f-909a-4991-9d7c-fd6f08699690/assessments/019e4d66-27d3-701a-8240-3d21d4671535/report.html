Data Storage & External Services

The service stores and reads operational data from SQL Server tables such as Users, SessionTokens, Accounts, and Transactions. It integrates with ZavaAuthGateway to translate an SSO cookie into a session token and with ZavaLedger to post payment transactions over HTTP.

Key Architectural Decisions

  • Uses server-rendered JSP pages with Struts actions rather than a REST-first API.
  • Uses direct SQL with JDBC instead of a repository/ORM abstraction.
  • Delegates authentication token decoding to an external auth gateway because Java side cannot decode the .NET auth cookie directly.

DTOs & Contracts

Gateway contracts are form and view-model oriented rather than JSON REST DTOs. LoginForm and PaymentForm are request contracts for incoming browser posts; SessionUser, AccountOption, and PaymentHistoryRecord are response/view contracts used to render JSP pages. Outbound service contracts are custom payloads: the auth-gateway call expects JSON containing sessionToken, and the ledger call sends XML <transactionRequest> with debit/credit/amount/description/referenceNumber and expects <status>POSTED</status> in response.

Communication Patterns

Communication is fully synchronous HTTP and database I/O. User requests reach Struts actions, which query SQL Server directly via JDBC and call downstream HTTP services when needed. Resilience behavior is basic timeout and fallback: SsoSessionService uses 3s connect/read timeouts to auth-gateway cookie resolution and returns unauthenticated on failure; PaymentAction uses 8s connect/read timeouts to ledger and returns user-facing failure when posting fails. No service discovery, retry framework, circuit breaker, or message broker pattern is present. Startup availability of SQL Server and external auth/ledger services directly affects request success. No explicit TLS enforcement, authentication framework, or authorization annotations are present at the gateway API contract layer.

Primary Workflows

Workflow 1: Authenticate and establish user session

  1. User opens login entry and submits token (or provides existing cookie/header token).
  2. Gateway attempts to resolve session against active session records.
  3. If token is missing from direct request values, gateway calls auth gateway to resolve from SSO cookie.
  4. If a valid user is found, session is stored and user is forwarded to payment screen; otherwise login error is shown.

Workflow 2: Post payment through ledger

  1. Authenticated user submits payment form with account, amount, and payment type.
  2. Gateway validates required fields and amount > 0 rule.
  3. Gateway chooses settlement account and avoids using the same debit and settlement account.
  4. Gateway posts XML transaction request to ledger service.
  5. On posted status, user is redirected to history; on failure, gateway shows error message.

Workflow 3: View payment history

  1. User opens payment history view.
  2. Gateway validates active session.
  3. Gateway loads recent transactions tagged with gateway-originated description prefix.
  4. Results are transformed to UI view rows and rendered to the user.

Cross-Service Data Flows

The gateway composes user-facing flows by combining data from SQL Server and external service calls. Authentication flow may combine cookie-based identity resolution from ZavaAuthGateway with local session-table validation. Payment submission combines local form/session context with downstream ledger posting, then later reads resulting transaction history from SQL Server. If the ledger is unavailable, business outcome degrades to a failed payment notice; no asynchronous retry or queued compensation is implemented.

Business Workflow Sequence

sequenceDiagram participant User participant Gateway as "ZavaPayGateway" participant Auth as "ZavaAuthGateway" participant DB as "SQL Server" participant Ledger as "ZavaLedger" User->>Gateway: Submit payment request Gateway->>DB: Validate active session token alt Session not found locally Gateway->>Auth: Resolve token from auth cookie Auth-->>Gateway: Return session token Gateway->>DB: Re-check token and user DB-->>Gateway: Session user else Session found DB-->>Gateway: Session user end Gateway->>Gateway: Apply payment rules and settlement selection Gateway->>Ledger: Post transaction request alt Ledger accepts payment Ledger-->>Gateway: Transaction posted Gateway->>DB: Read latest gateway transactions for history DB-->>Gateway: History rows Gateway-->>User: Show updated payment history else Ledger rejects or times out Ledger-->>Gateway: Failure response Note over Gateway: Payment remains unposted Gateway-->>User: Show payment error message end

Business Rules & Decision Logic

  • Payment form requires account, amount, and payment type before processing.
  • Amount must be greater than zero; invalid number formats cause immediate rejection.
  • Settlement account is adjusted when it matches debit account to prevent same-account transfer in gateway flow.
  • Session is mandatory for payment and history operations; unauthenticated requests are redirected to login.
  • Gateway treats ledger non-OK response or timeout as failed payment and does not persist local success state.

Caching Strategy

No dedicated cache provider or cache annotations are present. All reads are performed directly against SQL Server for session validation, account lists, and transaction history.

Data Ownership Boundaries

The gateway relies on a shared banking database for authentication/session context and read-side account/transaction views. Write-side transaction posting is boundary-separated: the gateway sends a request to ZavaLedger, and the resulting transaction records are later read from the shared data store for history display.
