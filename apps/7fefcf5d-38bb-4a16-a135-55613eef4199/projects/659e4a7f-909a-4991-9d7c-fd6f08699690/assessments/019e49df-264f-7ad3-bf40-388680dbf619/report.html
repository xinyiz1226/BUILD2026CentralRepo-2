Data Storage & External Services

The application persists all data in a single Microsoft SQL Server database (ZavaBankDB) accessed via JDBC with direct SQL queries (no ORM). Tables used include Accounts, AccountTypes, Transactions, SessionTokens, and Users. Two external HTTP microservices are integrated: the ZavaAuth Gateway (WhoAmI.ashx) resolves SSO .ZAVAAUTH cookie values into session tokens, and the ZavaLedger API (POST /api/transactions) receives XML transaction payloads to book double-entry payments.

Key Architectural Decisions

  • Apache Struts 1.3 MVC: Routes all HTTP actions through a central ActionServlet dispatcher; each action class handles both GET (display) and POST (submit) by inspecting the request method directly.
  • SSO via cookie + auth gateway: Java cannot decrypt .NET FormsAuthentication cookies, so authentication delegates to the ZavaAuth Gateway (WhoAmI.ashx) to resolve the .ZAVAAUTH cookie into a session token, which is then validated against the database.
  • Configuration via environment variables with properties fallback: PayGatewayConfig reads from environment variables first and falls back to paygateway.properties, enabling container-friendly deployment without code changes.

DTOs & Contracts

Struts ActionForm beans (request binding):

  • LoginForm — binds the sessionToken form field from the login POST. Mutable JavaBean; not immutable.
  • PaymentForm — binds accountId, amount, paymentType, and memo from the payment POST. Mutable JavaBean; not immutable.

Internal view models (set as request attributes, passed to JSP):

  • SessionUser — immutable (final fields, constructor-only assignment); holds userId and username for the authenticated user. Stored in the HTTP session under zavaPayGateway.user.
  • AccountOption — mutable JavaBean; represents a selectable account in the payment form dropdown. See data-architecture.md for field details.
  • PaymentHistoryRecord — mutable JavaBean; represents one row in the payment history view. See data-architecture.md for field details.

No OpenAPI/Swagger, protobuf, or GraphQL schemas exist. Serialization is not applicable — all data is passed as JSP request attributes, not serialized to JSON/XML for browser responses. The one outbound XML payload (ZavaLedger transaction request) is constructed manually as a string in PaymentAction.

Communication Patterns

Synchronous (blocking HTTP — outbound only):

  • ZavaAuth Gateway (GET /WhoAmI.ashx): Called by SsoSessionService when a .ZAVAAUTH cookie is present but no session token is found via other means. Uses java.net.HttpURLConnection directly. Connect timeout: 3 000 ms; read timeout: 3 000 ms. No retry logic. No circuit breaker. Failure silently returns an empty token (user sees a login error).
  • ZavaLedger API (POST /api/transactions): Called by PaymentAction to book each payment. Uses java.net.HttpURLConnection. Connect timeout: 8 000 ms; read timeout: 8 000 ms. No retry logic. No circuit breaker. Failure results in a user-visible error message ("Unable to post payment to ZavaLedger").

Asynchronous patterns: None. All inter-service calls are synchronous and blocking.

Service discovery: None. Service URLs are hardcoded via configuration (AUTH_GATEWAY_WHOAMI_URL, LEDGER_URL environment variables or paygateway.properties). No Eureka, Consul, or Kubernetes DNS-based discovery is used.

API gateway: This application itself acts as a thin gateway between the browser and ZavaLedger: it authenticates the user, loads account data from SQL Server, and then forwards a payment instruction to ZavaLedger. There is no inbound API gateway in front of ZavaPayGateway.

Resilience patterns: No circuit breaker (Resilience4j or similar), no retry policies, no bulkhead patterns. The only resilience is fixed connect/read timeouts on HttpURLConnection. Failures are handled by swallowing exceptions silently and returning user-visible error messages.

Security posture: No TLS/HTTPS is configured at the application level (the servlet container or a reverse proxy must terminate TLS externally). There is no Spring Security, JWT, or OAuth2 framework. Authentication is implemented manually in each Action class by checking the HTTP session for a SessionUser attribute; unauthenticated requests are redirected to /login.do. All token-to-user resolution is delegated to the SQL Server SessionTokens table and the ZavaAuth Gateway. No CSRF protection is implemented. No authorization (role-based access control) beyond the session check is applied.

Primary Workflows

Workflow 1: User Authentication (SSO Login)

A user arrives at the payment gateway. If they do not have a valid server-side session, the system attempts to authenticate them via one of three token resolution strategies.

Steps:

  1. User navigates to /login.do (GET) — the login page is displayed.
  2. User submits the login form (POST) with a sessionToken field, OR the system inspects the incoming request for a session token in the following priority order:
    • Form field sessionToken (explicit POST)
    • Query parameter ?sessionToken=...
    • HTTP header X-Session-Token
    • HTTP header Authorization: Bearer <token>
    • .ZAVAAUTH SSO cookie → resolved via ZavaAuth Gateway (GET /WhoAmI.ashx)
  1. The resolved token is validated against SessionTokens in SQL Server: token must exist, be active (IsActive = 1), not expired (ExpiresAt > GETDATE()), and belong to an active user.
  2. On success: a SessionUser (userId, username) is stored in the HTTP session and the user is redirected to /makePayment.do.
  3. On failure: the login page is re-displayed with an error message ("Session token was not found or has expired.").

Business rules involved: Token expiry check, user active status check, multi-path token resolution.

Workflow 2: Payment Submission

An authenticated user selects a bank account, enters an amount, chooses a payment type, and optionally adds a memo. The payment is posted to ZavaLedger.

Steps:

  1. User navigates to /makePayment.do (GET) — the system verifies the HTTP session, loads up to 30 active accounts from SQL Server, and renders the payment form.
  2. User submits the payment form (POST) with accountId, amount, paymentType, and optional memo.
  3. Session is re-verified; if expired, user is redirected to /login.do.
  4. Input validation:
    • accountId must be non-empty
    • amount must be non-empty, parseable as a decimal, and greater than zero
    • paymentType must be non-empty
  1. Business rule — settlement account collision avoidance: if the selected debitAccountId equals the configured settlementAccountId, the settlement account is swapped to an alternate ID (account 2 if configured is 1, or account 1 otherwise).
  2. A unique reference number is generated: PGW-{System.currentTimeMillis()}.
  3. An XML transaction request is POSTed to the ZavaLedger API with: debit account ID, settlement credit account ID, amount, description (ZavaPayGateway:{paymentType}:{memo}), and reference number.
  4. ZavaLedger response is checked: HTTP 200 with body containing <status>POSTED</status> = success.
  5. On success: user is redirected to /paymentHistory.do.
  6. On failure (ledger error, timeout, or non-POSTED status): user sees "Unable to post payment to ZavaLedger."

Workflow 3: Payment History View

An authenticated user views the last 75 payments submitted through ZavaPayGateway.

Steps:

  1. User navigates to /paymentHistory.do (GET).
  2. Session is verified; if expired, user is redirected to /login.do.
  3. The system queries SQL Server for up to 75 transactions where Description LIKE 'ZavaPayGateway:%', ordered by TransactionDate DESC, joined with Accounts to retrieve account numbers.
  4. Payment type is extracted from the description field by splitting on : (format: ZavaPayGateway:{paymentType}:{memo}).
  5. Results are rendered in a history table showing: Transaction ID, Account Number, Amount, Payment Type, Reference Number, Date, and Status.

Workflow 4: Health Check

Steps:

  1. A monitoring system or load balancer calls GET /health.do.
  2. HealthAction forwards unconditionally to health.jsp, which returns an HTTP 200 response.
  3. No actual dependency probing (database, ZavaLedger, ZavaAuth) is performed.

Cross-Service Data Flows

ZavaPayGateway acts as an orchestration layer between the browser and two external services:

Authentication flow (ZavaAuth Gateway):

When a .ZAVAAUTH cookie is present but no session token is extractable from other sources, ZavaPayGateway forwards the cookie to GET /WhoAmI.ashx on the ZavaAuth Gateway. The gateway returns a JSON response containing a sessionToken field, which ZavaPayGateway then validates against SQL Server. If the ZavaAuth Gateway is unavailable or returns a non-200 response, the fallback is a failed login (no partial or degraded authentication is provided).

Payment flow (ZavaLedger API):

After all local validation passes, ZavaPayGateway constructs an XML payload and POSTs it to ZavaLedger. ZavaLedger is the sole writer of Transaction records. ZavaPayGateway reads those same records directly from SQL Server for the history view — there is no ZavaLedger read API used. If ZavaLedger is unavailable (timeout after 8 s or non-200 response), the payment is not retried; the user receives an error message and must resubmit manually.

Account data (SQL Server direct):

Account options for the payment form are read directly from the shared ZavaBankDB SQL Server database. There is no account-management microservice API; ZavaPayGateway queries the shared database directly.

Decision Logic

  • Settlement account collision avoidance: If the user-selected debit account ID equals the configured ledger.settlement.account.id, the credit (settlement) account is automatically swapped to account 2 (or 1 if the settlement account is 2). This prevents a self-transfer.
  • Token resolution priority: The system tries five distinct strategies to find an auth token (form field → query param → X-Session-Token header → Authorization: Bearer header → .ZAVAAUTH cookie via ZavaAuth Gateway), applying the first non-empty result.
  • Payment history filtering: Only transactions with Description LIKE 'ZavaPayGateway:%' are shown — transactions created by other systems sharing the same database are excluded.

State Transitions

ZavaPayGateway itself has no entity state machines. Transaction status values (Status column) are set and owned by ZavaLedger; ZavaPayGateway reads but does not mutate them.

Cross-Cutting Concerns

  • Transactions: No @Transactional boundaries exist. Each JDBC operation is auto-committed. The payment workflow is not atomic: if ZavaLedger accepts a payment but the redirect fails, the user may be confused about whether the payment was posted.
  • Error handling: All SQLException and IOException exceptions are silently swallowed with empty catch (Exception ignored) blocks throughout. Business errors are surfaced as JSP-rendered message strings, not structured exceptions.
  • Audit/logging: No structured logging or audit trail is produced by ZavaPayGateway. The only audit record is the Transactions entry created by ZavaLedger on success, with the ReferenceNumber (PGW-{timestamp}) as the correlation ID.
  • Authorization: Authorization is limited to session presence — any authenticated user can view all active accounts and all ZavaPayGateway transaction history. There is no per-user or role-based scoping of data.

Version & Compatibility Risks

Apache Struts 1.3.10 reached end-of-life in 2013 and has not received security patches since. It is incompatible with Jakarta EE (javax. vs jakarta. namespace split) and has no supported migration path to a modern framework without significant rewriting. The Servlet API 3.1 (javax.servlet) is similarly superseded by Servlet 5.0+ under the jakarta.servlet namespace in Jakarta EE 9+, which will be required for any modern application server. The Microsoft JDBC driver (12.8.1) is current and does not pose a compatibility risk.

Notable Observations

  • Struts 1.x is end-of-life and a known security risk: The framework has multiple unpatched CVEs and is no longer maintained by the Apache Software Foundation. Migration to Spring MVC, Jakarta Faces, or another modern framework is strongly recommended before any cloud migration.
  • No ORM or connection pooling: Database access is done via raw JDBC using DriverManager.getConnection(). There is no connection pooling (e.g., HikariCP) or ORM (e.g., Hibernate), which could lead to connection exhaustion under load.
  • No logging framework declared: The project declares no logging dependency (no SLF4J, Log4j, or Logback), relying on implicit Struts transitive dependencies or System.out. Structured logging is absent.
  • Minimal dependency footprint: Only 4 declared dependencies indicate a deliberately minimal application. Transitive dependencies from Struts 1.3.10 (e.g., Commons BeanUtils, Commons Chain, OGNL) are not declared explicitly and may introduce hidden vulnerabilities.

Caching Strategy

No caching layer is configured. There is no Redis, EhCache, Caffeine, or Spring Cache (@Cacheable) in the project.

Implicit session cache (HTTP session only): The authenticated SessionUser is stored in the HTTP session (HttpSession.setAttribute("zavaPayGateway.user", sessionUser)). Subsequent requests within the same session skip the database token lookup. This is the only caching mechanism present.

Account list: Loaded fresh from SQL Server on every GET /makePayment.do and POST /makePayment.do request — no caching applied.

Payment history: Loaded fresh from SQL Server on every GET /paymentHistory.do request — no caching applied.

Data Ownership Boundaries

ZavaPayGateway uses a shared external database (ZavaBankDB) that is also owned and written to by the ZavaAuth Gateway and ZavaLedger API. There is no database-per-service isolation.

Cross-service data access patterns:

  • ZavaPayGateway reads SessionTokens and Users directly from the shared SQL Server database (bypassing ZavaAuth's API for token validation). This creates a tight coupling to the ZavaAuth database schema.
  • ZavaPayGateway reads Accounts, AccountTypes, and Transactions directly from the shared database.
  • ZavaPayGateway writes transactions exclusively through the ZavaLedger HTTP API (never direct INSERT), which provides some logical separation for the write path.
  • Session authentication has a dual path: direct DB lookup first, then ZavaAuth Gateway as fallback — meaning ZavaPayGateway has implicit knowledge of the ZavaAuth internal schema.

Read/write observations: This is a predominantly read-heavy application (account list, history). The only write operation (payment posting) is externalized to ZavaLedger. No CQRS or event sourcing patterns are present.
