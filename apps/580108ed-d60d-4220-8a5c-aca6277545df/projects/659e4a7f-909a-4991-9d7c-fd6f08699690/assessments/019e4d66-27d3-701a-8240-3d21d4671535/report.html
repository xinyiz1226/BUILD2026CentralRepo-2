Data Storage & External Services

The application uses Microsoft SQL Server (ZavaBankDB) as its primary data store, accessed directly via JDBC without an ORM layer. Wire transfer records are stored in the WireTransfers table, which is created on startup by WireBootstrapServlet. Two external HTTP microservices are called at runtime: a Currency Service provides real-time FX exchange rates for international transfers, and a Ledger Service accepts XML-formatted debit transactions to post charges against the originating account. After persisting a transfer, the application publishes an event to a RabbitMQ topic exchange (wire.events) using a routing key based on the transfer status.

Key Architectural Decisions

  • Direct JDBC without ORM: All database interactions use raw JDBC PreparedStatement calls rather than JPA/Hibernate, keeping the dependency footprint small but requiring manual SQL management.
  • Synchronous HTTP integration: FX rate lookup and ledger debit are performed inline within the transfer request, with fixed 7-second connect/read timeouts; failures are handled gracefully (FX defaults to 1.0, ledger failure sets status to PENDING_LEDGER).
  • Properties + environment variable configuration: WireConfig supports both a wire-transfer.properties file and environment variable overrides, enabling container-based deployment with runtime configuration injection.

DTOs & Contracts

No formal DTO classes or OpenAPI/Swagger specifications exist in this application. All request and response serialization is performed manually:

  • Wire Transfer Request (inbound): JSON body parsed inline in WireTransferServlet.doPost() using org.json.JSONObject. Fields: fromAccountId, toAccountNumber, toRoutingNumber, beneficiaryName, currencyCode, amount. No dedicated request class.
  • Wire Transfer Response (outbound): JSON object built inline using org.json.JSONObject. Fields: wireTransferId, status, referenceNumber, fxRate, debitAmountUSD, ledgerPosted. No dedicated response class.
  • Wire Status Response (outbound): JSON object built inline in WireStatusServlet.doGet(). Contains full wire transfer record fields. No dedicated response class.
  • FX Rate Response (inbound from Currency Service): Parsed as {"rate": <double>} JSON.
  • Ledger Debit Request (outbound to Ledger Service): Hand-crafted XML string with fields: debitAccountId, creditAccountId, amount, description, referenceNumber. Content-Type: application/xml.
  • RabbitMQ Wire Event (outbound): JSON payload built with org.json.JSONObject. Fields: wireTransferId, status, referenceNumber, amount, currencyCode, eventDate.

There are no protobuf schemas, GraphQL schemas, or OpenAPI specs. All contracts are implicit and embedded in the servlet implementation code.

Communication Patterns

Synchronous (REST over HTTP):

  • WireTransferServlet calls the Currency Service (GET /api/currency/rate?from=USD&to={currency}) to retrieve FX rates for international transfers. Both connect and read timeouts are set to 7 seconds. On failure or non-2xx response, the FX rate silently defaults to 1.0 — there is no retry, circuit breaker, or fallback alert.
  • WireTransferServlet calls the Ledger Service (POST /api/transactions) with an XML-formatted debit request. Both connect and read timeouts are set to 7 seconds. On failure, the transfer is persisted with status PENDING_LEDGER and failureReason = "Ledger posting failed." — no retry is performed.

Asynchronous (AMQP):

  • After database insertion, WireTransferServlet publishes a JSON event to the RabbitMQ wire.events topic exchange. The routing key is {routingKeyPrefix}.{status.toLowerCase()} (e.g., wire.posted or wire.pending_ledger). AMQP connection and channel are opened fresh per request and closed in a finally block. Publish failures are silently swallowed.

Resilience Patterns: None formally implemented. There are no circuit breakers, retry policies, bulkhead patterns, or dead-letter queue configurations. All failures in external calls are absorbed silently with minimal fallback logic (FX defaults to 1.0, ledger failure changes status field only).

Service Discovery: Services are located by hardcoded base URLs read from WireConfig (environment variables or wire-transfer.properties). There is no service registry, DNS-based discovery, or client-side load balancing.

API Gateway: Not present. Clients call the servlet endpoints directly.

Startup Dependency Chain: WireBootstrapServlet is configured with <load-on-startup>1</load-on-startup> in web.xml. It opens a JDBC connection to SQL Server and creates the WireTransfers table on startup. If SQL Server is unavailable at startup, the application fails to deploy.

Security Posture: No authentication, no authorization, and no TLS are configured at the API level. All endpoints (/api/wire/transfer, /api/wire/{id}/status, /health) are publicly accessible with no authentication or authorization checks. Wire transfer amounts and account identifiers can be submitted by any caller without credentials.

ZavaWireTransferService enables clients to initiate domestic and international wire transfers and query their status, orchestrating FX conversion, ledger debiting, and event publication as part of the transfer processing workflow.

Primary Workflows

Workflow 1: Initiate Wire Transfer

A client submits a wire transfer request. The service validates the request, optionally fetches a live FX rate for international transfers, posts a debit to the originating account via the Ledger Service, persists the transfer record in SQL Server, and publishes a status event to RabbitMQ.

Steps:

  1. Receive and parse request: Client submits JSON payload via POST. The service reads the request body and parses JSON fields.
  2. Input validation: All required fields must be present and non-empty; amount must be greater than zero; toRoutingNumber must pass ABA routing number checksum validation (9 digits, weighted checksum mod 10 = 0).
  3. FX rate lookup (international only): If currencyCode is not USD, the service calls the Currency Service to retrieve the current USD exchange rate. On failure or timeout (7s), the FX rate silently defaults to 1.0 and processing continues.
  4. Debit amount calculation: debitAmountUSD = amount × fxRate, rounded to 2 decimal places.
  5. Ledger debit posting: The service calls the Ledger Service with an XML debit request (originating account → settlement account, amount in USD, reference number). On success, status is set to POSTED. On failure or timeout (7s), status is set to PENDING_LEDGER with a failure reason recorded.
  6. Reference number generation: A unique reference number is generated as WIRE-{8-char uppercase UUID fragment}.
  7. Persist wire transfer record: The transfer is inserted into the WireTransfers table with all fields including status, FX rate, and failure reason if applicable. The auto-generated WireTransferID is captured.
  8. Publish event to RabbitMQ: A JSON event is published to the wire.events topic exchange with a routing key of wire.{status} (e.g., wire.posted or wire.pending_ledger). Publish failures are silently ignored.
  9. Return response: The service returns the wireTransferId, status, referenceNumber, fxRate, debitAmountUSD, and ledgerPosted flag to the client.

Workflow 2: Query Wire Transfer Status

A client queries the processing status of a previously submitted wire transfer by its numeric ID.

Steps:

  1. Receive request: Client submits GET request with the wire transfer ID in the URL path (/api/wire/{id}/status).
  2. ID validation: The path segment is parsed as a long integer. Non-numeric values return 400.
  3. Database lookup: The service queries the WireTransfers table by WireTransferID.
  4. Return result: If found, returns the full wire transfer record as JSON (all fields including status, amounts, dates, and failure reason). If not found, returns 404.

Workflow 3: Application Bootstrap (Startup)

On application startup, WireBootstrapServlet (configured as load-on-startup=1) connects to SQL Server and creates the WireTransfers table using a conditional DDL statement (IF OBJECT_ID ... IS NULL CREATE TABLE ...). This ensures the schema exists before the first transfer request is received.

Cross-Service Data Flows

Wire Transfer → Currency Service (synchronous, optional):

The wire transfer workflow calls the Currency Service only for international (non-USD) transfers. The service provides a single FX rate (rate field as a double) from USD to the target currency. If the Currency Service is unavailable or returns an error, the FX rate defaults to 1.0 and processing proceeds — meaning the debit will be calculated in the original currency amount, potentially causing an incorrect USD settlement for the originating account.

Wire Transfer → Ledger Service (synchronous, required for POSTED status):

The service posts an XML debit request to the Ledger Service for every transfer attempt. The Ledger Service is responsible for deducting funds from the originating account and crediting a settlement/suspense account. If the Ledger Service is unavailable or returns a non-2xx response, the wire transfer is still persisted but with status PENDING_LEDGER. There is no retry mechanism, compensating transaction, or dead-letter handling.

Wire Transfer → RabbitMQ (asynchronous, fire-and-forget):

After database persistence, the service publishes a JSON event to RabbitMQ. This is a one-way notification; the wire transfer service does not wait for acknowledgement or confirmation. Downstream consumers (not part of this codebase) may use these events to trigger further processing, notifications, or reconciliation. Event failures have no effect on the transfer record.

Business Rules & Decision Logic

Validation Rules

  • Required fields: fromAccountId (must be > 0), toAccountNumber (non-empty), toRoutingNumber (non-empty, must pass checksum), beneficiaryName (non-empty), amount (must be > 0).
  • Routing number format: Must be exactly 9 digits. Must satisfy the ABA routing number checksum: (3×d1 + 7×d2 + d3 + 3×d4 + 7×d5 + d6 + 3×d7 + 7×d8 + d9) mod 10 == 0.
  • Currency code: Defaults to USD if not provided. Any non-USD value triggers an international transfer with FX rate lookup.
  • Amount precision: Amount is rounded to 2 decimal places using HALF_UP rounding at parse time.

State Transitions

The WireTransfer entity has two possible post-processing statuses:

  • POSTED: Ledger debit succeeded; funds have been debited from the originating account.
  • PENDING_LEDGER: Ledger debit failed; the transfer record exists but funds have not been debited. No automatic retry or reconciliation is implemented.

Decision Logic

  • International vs domestic: Determined solely by currencyCode. Any currency other than USD is treated as international and triggers FX rate lookup.
  • FX rate fallback: If the Currency Service fails, fxRate defaults to 1.0, meaning debitAmountUSD equals the requested amount regardless of the actual exchange rate. This is a silent business risk.
  • Ledger failure handling: A ledger failure does not abort the transfer — the record is persisted with PENDING_LEDGER status. There is no compensating action, dead-letter queue, or human escalation workflow.

Cross-Cutting Concerns

  • Transactions: No @Transactional or JDBC transaction management. The ledger debit call and database insert are not atomic — it is possible for the ledger to be debited while the database insert fails (orphaned debit), or for the database insert to succeed with POSTED status while the ledger was not actually debited.
  • Error handling: External service failures (Currency Service, Ledger Service, RabbitMQ) are all caught and silently swallowed. No structured error logging, alerting, or compensating transaction is implemented.
  • Audit / logging: No audit trail or structured logging is implemented. All catch blocks use ignored exception variables and discard error information.
  • Authorization: No authentication or authorization is enforced. Any caller can submit a wire transfer for any fromAccountId without identity verification, making the endpoint vulnerable to unauthorized fund transfers.
  • Idempotency: No idempotency key or duplicate-detection mechanism exists. Repeated identical requests will create multiple wire transfer records and may trigger multiple ledger debits.

Version & Compatibility Risks

The most notable risk is org.json:json:20140107, which is over a decade old (released in 2014) and predates many security and correctness fixes in the library; it should be updated to a recent release (e.g., 20240303 or later). The Servlet API 4.0.1 targets Jakarta EE 8 (javax.* namespace); migration to a cloud-native runtime (e.g., Azure App Service, Azure Container Apps, or AKS) would still work, but any future upgrade to Jakarta EE 9+ requires renaming all javax.servlet imports to jakarta.servlet. The RabbitMQ amqp-client 5.20.0 is a recent stable release and presents no immediate risk, though a migration to Azure Service Bus would replace this dependency.

Notable Observations

  • No ORM layer: The application uses raw JDBC PreparedStatement calls directly. While this keeps the dependency count low, it means all SQL is hand-written and there is no schema migration framework (e.g., Flyway/Liquibase).
  • Very outdated JSON library: org.json:json:20140107 is from January 2014. It is functional but has had multiple correctness and performance improvements since then. Modern alternatives such as jackson-databind or an updated org.json release would reduce risk.
  • No logging framework declared: There is no SLF4J, Log4j, or Logback dependency. The application uses no structured logging; exceptions are silently swallowed (catch (Exception ignored)), making production observability and debugging difficult.
  • No caching, security, or observability libraries: There are no libraries for authentication/authorization, response caching, metrics, or distributed tracing, all of which would be expected in a production financial service.

Data Architecture & Persistence Layer

ZavaWireTransferService uses a single Microsoft SQL Server database (ZavaBankDB) with one application-managed table (WireTransfers), accessed entirely through raw JDBC with no ORM layer.

Caching Strategy

No caching layer is configured. Every request to WireStatusServlet executes a direct SQL SELECT against SQL Server with a freshly opened JDBC connection. There is no second-level cache, query result cache, in-process cache (e.g., Caffeine), or distributed cache (e.g., Redis). For a status-check API on a financial service, this means every read incurs a full database round-trip with a new TCP connection overhead.

Data Ownership Boundaries

ZavaWireTransferService owns the WireTransfers table exclusively. There is no shared database schema with other services; the application operates as the sole writer and reader of this table. Cross-service data access works only through HTTP calls:

  • Currency Service: queried at runtime via HTTP GET for FX rates — no shared database.
  • Ledger Service: receives XML POST requests for debit posting — no shared database.
  • RabbitMQ consumers: downstream consumers receive events via the wire.events exchange but have no direct access to the WireTransfers table.

Read/Write pattern: The application is write-heavy at the transfer endpoint (INSERT + external service calls) and read-light at the status endpoint (single-row SELECT by PK). There is no CQRS separation, no read replica, and no eventual consistency mechanism.
