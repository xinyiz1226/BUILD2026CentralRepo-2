Data Storage & External Services

The application uses SQL Server (ZavaBankDB) as its primary data store, writing wire transfer records to the WireTransfers table via JDBC. Two downstream HTTP microservices are called synchronously: the Currency Service (REST JSON) provides FX exchange rates for international transfers, and the Ledger Service (REST XML) posts debit transactions against the originating account. After a transfer is persisted, an AMQP event is published to RabbitMQ on the wire.events topic exchange with a routing key such as wire.posted or wire.pending_ledger, enabling downstream consumers to react asynchronously.

Key Architectural Decisions

  • Vanilla Java Servlet architecture: No Spring or Jakarta EE CDI — all wiring is manual, with static utility classes (WireConfig, WireConnectionFactory) providing shared configuration and JDBC connectivity.
  • Per-request JDBC connections: Each servlet request opens and closes its own java.sql.Connection via DriverManager, with no connection pooling.
  • Configuration via properties file + environment variables: WireConfig reads wire-transfer.properties from the classpath and overrides any value with a corresponding environment variable, enabling Docker-friendly externalization.

DTOs & Contracts

The application uses no dedicated DTO or model classes. All request and response serialization is handled inline:

  • Inbound requests are parsed directly from the raw HTTP body using new JSONObject(readBody(request)) inside WireTransferServlet.doPost().
  • Outbound responses are assembled inline with new JSONObject() and written directly to the HttpServletResponse writer.
  • Ledger Service call uses hand-crafted XML string concatenation (no JAXB, no schema).
  • Currency Service call parses the response body as a raw JSONObject and extracts a rate field.

There are no OpenAPI/Swagger specifications, no protobuf schemas, no GraphQL schemas, and no Jackson/Gson configuration. Field-level details and the WireTransfers table schema are documented in data-architecture.md.

Communication Patterns

Synchronous (HTTP):

  • WireTransferServlet calls the Currency Service (GET /api/currency/rate?from=USD&to={currency}) via java.net.HttpURLConnection with a 7-second connect and read timeout when the requested currency is non-USD. On any error or non-2xx response the FX rate silently defaults to 1.0.
  • WireTransferServlet calls the Ledger Service (POST /api/transactions) with an XML body via HttpURLConnection with a 7-second connect and read timeout. On failure the transfer is persisted with status=PENDING_LEDGER.

Asynchronous (AMQP):

  • After the SQL INSERT, WireTransferServlet publishes a JSON event to RabbitMQ on the wire.events topic exchange. The routing key is wire.{status} (e.g., wire.posted or wire.pending_ledger). The exchange is declared as durable=true. Publish failures are silently swallowed.

Resilience patterns:

  • No circuit breaker, no retry, no bulkhead — only raw timeouts (7 s) on HTTP calls.
  • Fallback behaviour: FX rate defaults to 1.0; ledger failure sets status to PENDING_LEDGER; AMQP publish failure is ignored.

Service discovery:

  • Services are located by hardcoded base URLs configured in wire-transfer.properties (overridable via environment variables). No service registry (Eureka, Consul, Kubernetes DNS) is used.

Startup dependency chain:

  • WireBootstrapServlet is marked load-on-startup=1 and creates the WireTransfers table on first boot. If SQL Server is unreachable at startup, the application fails to start. No readiness probe or retry logic is present.

Security posture:

  • No authentication, no authorization, and no TLS are configured. All four endpoints are publicly accessible with no credential checks. The /internal/bootstrap endpoint is mapped at a URL pattern but is not protected by any filter or security mechanism.

Startup Dependency Chain

The application has implicit startup dependencies that are not guarded by any wait mechanism:

  3. RabbitMQ must be reachable at request time — the AMQP connection is opened per-publish; failures are silently swallowed.

There is no dockerize, Docker Compose depends_on with health checks, Kubernetes readiness probe, or Spring retry configuration to enforce this order.

Primary Workflows

Workflow 1: Submit Wire Transfer

Entry point: POST /api/wire/transfer → WireTransferServlet.doPost()

Steps:

  1. Parse request — read JSON body from HTTP request; on malformed JSON return 400 Bad Request.
  2. Field presence validation — verify fromAccountId > 0, toAccountNumber non-empty, toRoutingNumber non-empty, beneficiaryName non-empty, and amount > 0; return 400 if any check fails.
  3. ABA routing number validation — apply the 3-7-1 weighted checksum algorithm to the 9-digit routing number; return 400 if invalid.
  4. Currency determination — if currencyCode is not USD, the transfer is flagged as international.
  5. FX rate lookup (international only) — call Currency Service GET /api/currency/rate?from=USD&to={code} with a 7-second timeout. On success, use the returned rate; on any failure, silently fall back to rate = 1.0 (no error is surfaced to the caller).
  6. Compute debit amount — debitAmountUSD = amount × fxRate, rounded to 2 decimal places.
  7. Generate reference number — produce a unique WIRE-{8-char UUID} identifier.
  8. Post ledger debit — call Ledger Service POST /api/transactions with an XML body containing debit account, credit (settlement) account, amount, and reference number (7-second timeout). On success, status = POSTED; on failure, status = PENDING_LEDGER.
  9. Persist wire transfer record — insert the full transfer record into SQL Server and retrieve the generated WireTransferID.
  10. Publish wire event — publish a JSON event to RabbitMQ topic exchange wire.events with routing key wire.{status} (e.g., wire.posted). Failures are silently swallowed.
  11. Return response — 200 JSON with wireTransferId, status, referenceNumber, fxRate, debitAmountUSD, and ledgerPosted.

Workflow 2: Query Wire Transfer Status

Entry point: GET /api/wire/{id}/status → WireStatusServlet.doGet()

Steps:

  1. Parse path — extract path segments from the URL; validate the pattern is /{id}/status; return 400 on malformed path.
  2. Parse transfer ID — parse the {id} segment as a long; return 400 if non-numeric.
  3. Fetch record — query WireTransfers WHERE WireTransferID = ?; return 404 if not found.
  4. Return status — 200 JSON with all transfer fields including current Status, FailureReason, timestamps.

Workflow 3: Application Bootstrap (Startup)

Entry point: WireBootstrapServlet.init() (load-on-startup=1)

Steps:

  1. Open JDBC connection to SQL Server on application startup.
  2. Create table if absent — execute IF OBJECT_ID('WireTransfers', 'U') IS NULL CREATE TABLE WireTransfers (...). On any SQL error, throw ServletException and fail the application startup.

Cross-Service Data Flows

Currency rate composition (synchronous, conditional):

ZavaWireTransferService calls the Currency Service only when currencyCode ≠ USD. The rate returned is a point-in-time snapshot stored permanently on the transfer record. If the Currency Service is unavailable (timeout or non-2xx), the service silently uses rate 1.0 — the transfer proceeds but may be mispriced for international amounts. There is no retry, circuit breaker, or caller notification.

Ledger debit posting (synchronous, required):

The Ledger Service is called for every transfer to record the account debit. Its response determines the transfer's initial Status: success → POSTED; failure → PENDING_LEDGER. The PENDING_LEDGER status signals to downstream consumers (via RabbitMQ) that manual reconciliation may be required. No re-attempt or compensating transaction is implemented.

Wire event publishing (asynchronous, fire-and-forget):

After the SQL INSERT, a JSON event is published to RabbitMQ. Downstream services (e.g., notification, reconciliation) consume from the wire.events exchange using the wire.* routing key pattern. If the publish fails, the failure is swallowed and the event is lost — there is no outbox pattern, dead-letter queue, or acknowledgement mechanism.

Business Rules & Decision Logic

Validation rules (applied before any external calls):

  • fromAccountId must be a positive integer (> 0).
  • toAccountNumber must be a non-empty string.
  • toRoutingNumber must be exactly 9 digits and pass the ABA checksum: the weighted sum 3d₀ + 7d₁ + 1d₂ + 3d₃ + 7d₄ + 1d₅ + 3d₆ + 7d₇ + 1d₈ must be divisible by 10.
  • beneficiaryName must be a non-empty string.
  • amount must be greater than zero; it is rounded to 2 decimal places (HALF_UP).
  • currencyCode defaults to USD if omitted.

Decision logic:

  • International determination: any currencyCode other than USD triggers an FX rate lookup.
  • FX rate fallback: if the Currency Service is unreachable or returns an invalid rate (≤ 0), the system silently uses 1.0 without notifying the caller.
  • Status assignment: POSTED if the Ledger Service returned a 2xx response; PENDING_LEDGER otherwise.

State transitions:

  • POSTED — transfer created and ledger debit confirmed; event published as wire.posted.
  • PENDING_LEDGER — transfer created but ledger debit failed; event published as wire.pending_ledger; no automated retry.

Computed values:

  • debitAmountUSD = amount × fxRate (rounded to 2 decimal places, HALF_UP).
  • referenceNumber = "WIRE-" + first 8 characters of a random UUID (uppercase).

Transaction management:

No @Transactional annotation or explicit JDBC transaction boundary wraps the multi-step workflow. Each operation (ledger call, SQL insert, AMQP publish) is independent. A failure after the SQL insert but before the AMQP publish results in a persisted transfer with no event, with no compensating action.

Error handling:

  • All external call failures (Currency Service, Ledger Service, RabbitMQ) are caught with broad catch (Exception ignored) blocks — errors are silently swallowed rather than surfaced or logged.
  • SQL errors during insert result in a 500 Internal Server Error to the caller.
  • No dead-letter queue, retry queue, or saga/outbox pattern is implemented.

Authorization and audit:

No authentication, authorization, or audit logging is implemented. All endpoints are publicly accessible and no user identity is captured in the wire transfer record.

Version & Compatibility Risks

The most significant risk is org.json:json:20140107 — published in January 2014, this artifact is over a decade old and has known CVE exposure (notably CVE-2022-45688, a stack overflow vulnerability in its XML-to-JSON conversion path). Recommendation: upgrade to org.json:json:20240303 (or the latest release on Maven Central) to resolve known CVEs; alternatively, replace it with Jackson Databind (com.fasterxml.jackson.core:jackson-databind:2.17.x) which has a more active security response process. The javax.servlet:javax.servlet-api:4.0.1 dependency uses the legacy javax. namespace (Java EE), which was superseded by jakarta. in Jakarta EE 9+; migrating to a Jakarta EE container or a modern framework will require a namespace change. The RabbitMQ AMQP client (5.20.0) and MSSQL JDBC driver (12.6.3) are both recent releases and carry no immediate version risk.

Notable Observations

  • No logging framework declared: The application has no SLF4J, Logback, or Log4j dependency; all exceptions are silently swallowed with catch (Exception ignored) blocks, making production troubleshooting difficult.
  • No connection pooling library: JDBC connections are opened and closed per-request using DriverManager.getConnection() with no pooling (HikariCP, DBCP, etc.), which is a scalability and latency concern at production load.
  • Minimal dependency footprint: With only four declared dependencies the artifact tree is very small, but this is partly because the application manually re-implements patterns (HTTP clients via HttpURLConnection, XML generation via string concatenation) that dedicated libraries handle more safely.
  • org.json version is a date, not a semver: The 20140107 artifact is the only release of this vintage on Maven Central and may contain unpatched vulnerabilities; replacing it with Jackson or Gson is strongly recommended before cloud migration.

Data Architecture & Persistence Layer

ZavaWireTransferService owns a single table (WireTransfers) in a SQL Server database, accessed via raw JDBC with no ORM, no migration tool, and no caching layer.

The WireTransfers table is the only entity in the service. There are no JPA annotations, no ORM mappings, and no relationships to other locally-owned tables. FromAccountID is a logical foreign key referencing an account owned by an external service; no referential integrity constraint is defined at the database level.

There are no repository interfaces or ORM repositories. All data access is implemented inline inside servlet methods using PreparedStatement directly:

Caching Strategy

No caching layer is present. Every status lookup executes a live SQL query against SQL Server, and every FX rate call hits the upstream Currency Service over HTTP. There is no Redis, EhCache, Caffeine, or any JSR-107 provider configured.

Data Ownership Boundaries

ZavaWireTransferService uses a dedicated logical database (ZavaBankDB) that it fully owns. No other service is known to share this database. Cross-service data access is entirely via HTTP REST:

  • Account balances and account details are owned by an external account/ledger service; this service only stores the FromAccountID integer as a reference.
  • FX rates are retrieved on-demand from the Currency Service and stored as a snapshot (FxRate column) in the wire transfer record — there is no local cache or shared rate table.
  • The Status field acts as a lightweight event-sourcing flag (POSTED, PENDING_LEDGER) observable by downstream consumers via the RabbitMQ wire.events topic.

Isolation: The data store is isolated to this microservice. There are no direct cross-service SQL joins or shared schema objects.

Write pattern: The service is purely write-on-transfer and read-on-status-query — a simple request/response CRUD pattern with no CQRS or event sourcing at the persistence layer.

The WireTransfers table stores PII (beneficiary name) and sensitive financial data (account numbers, routing numbers, transaction amounts). No encryption-at-rest, column-level masking, field-level access controls, or data tokenization are configured. The JDBC connection string hardcodes encrypt=false and trustServerCertificate=true in WireConfig.getDbUrl(), which disables TLS encryption and bypasses certificate validation for all SQL Server traffic. Remediation required before cloud migration: set encrypt=true, remove trustServerCertificate=true, and configure a trusted CA certificate or use Azure SQL's built-in certificate trust. These security gaps expose PII and financial data to network interception and must be resolved prior to production cloud deployment.
