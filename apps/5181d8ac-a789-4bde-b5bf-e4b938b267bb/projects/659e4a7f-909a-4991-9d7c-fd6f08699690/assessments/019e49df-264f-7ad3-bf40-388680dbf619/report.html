Architecture Diagram

ZavaInterestCalculator is a single deployable web application that exposes servlet endpoints for interest calculations and persists results directly to a shared SQL Server database. Its architecture is intentionally small: JSP and servlet entry points on Tomcat, calculation logic in plain Java classes, and JDBC-based database access.

Data Storage & External Services

The service uses one external data store: a SQL Server database named ZavaBankDB. There is no cache, message broker, or downstream HTTP dependency; all business data is read and written through JDBC using connection details sourced from environment variables or interest.properties.

Key Architectural Decisions

  • Uses classic Java EE servlet deployment instead of Spring Boot or a DI container, keeping the runtime dependency surface small but placing wiring in web.xml and static helper classes.
  • Performs startup schema initialization with InterestBootstrapServlet, which creates the InterestAccruals table before transactional write endpoints are used.
  • Keeps both single-account and bulk accrual logic inside the web tier, with SQL statements embedded directly in servlets rather than abstracted repositories.

DTOs & Contracts

The service does not declare named DTO classes. Both business endpoints parse request bodies with org.json.JSONObject and emit ad hoc JSON objects directly from servlet methods. InterestCalculateServlet.AccountSnapshot is an internal helper used to carry account balance and account-type rate data between JDBC and calculation logic, but it is not exposed as an external contract.

No OpenAPI, Swagger, GraphQL, or protobuf schema was found. Serialization is handled manually with org.json for JSON endpoints and string literals for the HTML health response.

Communication Patterns

All request handling is synchronous: the client calls a servlet endpoint, the servlet opens a JDBC connection, queries SQL Server, optionally writes transactional updates, and returns a response immediately. There is no asynchronous messaging, no service discovery, no API gateway, and no retry or circuit-breaker library.

Security controls are also absent at the API-contract level. No authentication, authorization, or transport-security configuration is present in the repository, so the endpoints appear publicly callable within whatever network exposes the Tomcat instance. API availability depends on SQL Server being reachable; if the database is unavailable, the servlets return HTTP 500 responses or fail startup during bootstrap.

Core Business Workflows

ZavaInterestCalculator supports the banking workflow of calculating accrued interest for deposit or loan accounts and optionally applying that interest back to account balances. It also provides a startup bootstrap routine to provision the accrual-history table used for audit-style tracking of applied interest.

Primary Workflows

Workflow 1: Calculate interest for a single account

A client submits POST /api/interest/calculate with an accountId and optional calculation parameters. The servlet validates the JSON payload, loads the account balance plus account-type rate, determines effective defaults, computes simple, compound, or amortized interest, and returns the resulting amount and projected balance. If applyToBalance=true, the workflow also updates the account row and inserts an InterestAccrual record in the same transaction.

Workflow 2: Accrue interest for all active accounts

A caller submits POST /api/interest/accrue-all with optional calculation parameters. The servlet loads all accounts whose status is Active, computes interest for each account using the chosen method, updates balances in bulk, inserts one InterestAccrual record per account, commits once, and returns the number of accounts processed plus the total interest applied.

Workflow 3: Bootstrap accrual storage on startup

When the web application starts, InterestBootstrapServlet runs automatically and executes a CREATE TABLE statement if InterestAccruals does not yet exist. This ensures that later workflows that persist applied interest have a storage target before the first API call is served.

Cross-Service Data Flows

No network-based cross-service flow is implemented in this repository. Instead, the service composes the business view it needs directly in SQL by joining Accounts with AccountTypes to obtain both balance and default rate data. If the database is unavailable, workflows degrade by returning HTTP 500 responses or failing startup; there is no fallback data source, retry loop, or circuit breaker.

Business Rules & Decision Logic

  • Invalid JSON returns HTTP 400, and accountId must be greater than zero for the single-account workflow.
  • days defaults to 30 when absent or non-positive, termMonths defaults to 12, and compoundsPerYear defaults to 12 when invalid.
  • annualRate comes from the request when supplied, otherwise from the account type, and finally from the configured default when no positive rate is available.
  • Supported calculation modes are simple, compound, and amortized; any other value raises a business error for the single-account API.
  • Zero or negative principal results in zero interest rather than a negative accrual.
  • When persistence is requested, balance updates and accrual-history inserts occur inside a database transaction; failures roll back and surface as HTTP 500 responses.
  • No business-level authorization, audit trail beyond the accrual table, or role checks are implemented in the repository.

Version & Compatibility Risks

The dependency graph is small, but the runtime posture is dated. The project targets Java 8 and uses an old org.json release, so modernization work will likely focus more on platform upgrades than on a large library migration effort.

Notable Observations

  • No Spring, Jakarta EE, ORM, or caching framework is declared; core application behavior is implemented with plain servlets and handwritten SQL.
  • The SQL Server JDBC dependency is the only persistence library, which simplifies dependency migration but tightly couples code to SQL Server syntax such as ISNULL and GETDATE().
  • javax.servlet-api indicates a pre-Jakarta servlet namespace, which may matter if the service is later upgraded to newer servlet containers.
