Data Storage & External Services

The application relies exclusively on Microsoft SQL Server (database name ZavaBankDB) as its data store. Connection details (host, port, credentials) are read from environment variables (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) with fallback to interest.properties. The application manages two core tables: Accounts (with AccountTypes) for balance and account-type interest rate storage, and InterestAccruals for audit-trail records of each interest accrual event. No external messaging, caching, or third-party API services are used.

Key Architectural Decisions

  • Direct JDBC pattern: All database access is performed through raw JDBC using PreparedStatement and manual connection management (no ORM/repository layer), keeping the dependency footprint minimal.
  • Servlet-based REST API: The application exposes JSON REST endpoints using plain Java Servlets (Jakarta EE 3.1) rather than a full MVC framework (e.g., Spring MVC), deployed as a traditional WAR file.
  • Configuration via environment variables with property file fallback: InterestConfig reads connection settings and defaults from environment variables first, falling back to interest.properties, enabling Docker/cloud-native configuration without code changes.

Core Business Workflows

ZavaInterestCalculator is a banking microservice that calculates and applies interest accruals to customer accounts, supporting three interest calculation methods — simple, compound, and amortized — for individual accounts and in bulk across all active accounts.

This is a single-service application with one bounded context. The service reads Account and AccountType data as shared data and owns the InterestAccruals audit log. All operations are synchronous and backed by a single SQL Server database.

Primary Workflows

Workflow 1: Single Account Interest Calculation

A client submits a request to calculate (and optionally apply) interest for a specific bank account.

Steps:

  1. Client sends POST /api/interest/calculate with accountId (required) and optional parameters: interestType, days, annualRate, compoundsPerYear, termMonths, applyToBalance.
  2. The servlet validates accountId — if missing or ≤ 0, it returns HTTP 400.
  3. The account is loaded from the database (Balance and AccountType interest rate).
  4. If the account does not exist, HTTP 404 is returned.
  5. Interest calculation parameters are resolved with defaults (see Business Rules below).
  6. The interest amount is computed using the selected interest type (simple, compound, or amortized).
  7. If applyToBalance is true, a database transaction updates the account balance and inserts an InterestAccrual record.
  8. The response returns the interest type, annual rate, days, balance before, interest amount, balance after, and whether it was applied.

Key business rules involved: accountId presence check, interest type selection, rate and period defaulting, calculation method dispatch, optional balance mutation.

Workflow 2: Batch Interest Accrual for All Active Accounts

An operator or scheduler triggers bulk interest accrual across all active accounts in a single atomic transaction.

Steps:

  1. Client sends POST /api/interest/accrue-all with optional parameters: interestType, days, compoundsPerYear, termMonths.
  2. Parameters are resolved with defaults.
  3. A single database connection opens a transaction and selects all accounts with Status = 'Active', including their balance and account-type interest rate.
  4. For each active account:

a. If the account's interest rate is ≤ 0, the default annual rate from configuration is used.

b. Interest is calculated using the specified method.

c. The account balance is updated and an InterestAccrual record is inserted (batched within the same transaction).

  1. On success, the transaction is committed and the response returns the count of accrued accounts and total interest applied.
  2. On any SQL error, the entire transaction is rolled back and HTTP 500 is returned.

Key business rules involved: active account filtering, rate fallback to configuration default, all-or-nothing atomicity, total interest aggregation.

Workflow 3: Schema Bootstrap at Startup

The application ensures the InterestAccruals table exists before any business request is served.

Steps:

  1. Tomcat starts and triggers InterestBootstrapServlet (load-on-startup=1) before any HTTP traffic is accepted.
  2. The servlet connects to SQL Server and executes an idempotent DDL statement to create InterestAccruals if it does not already exist.
  3. If the database is unreachable, a ServletException is thrown and the application fails to start.

Cross-Service Data Flows

ZavaInterestCalculator is a single-service application with no inter-service communication. All data flows are between the service and SQL Server:

  • Read: Account balance and AccountType interest rate are read at the start of each calculation or batch operation.
  • Write: Account balance, AvailableBalance, and InterestAccrued are updated; InterestAccrual audit records are inserted.
  • Atomicity: Both the balance update and the accrual insert are wrapped in a single JDBC transaction for each accrual event, ensuring the audit trail is always consistent with the account balance.

There are no circuit breaker fallback paths, no downstream HTTP calls, and no message-driven integrations.

Rate Resolution Priority (both workflows)

  1. annualRate from request payload (if provided and > 0)
  2. Account's AccountType.InterestRate (if > 0)
  3. default.annual.rate from configuration (fallback, default: 1.5%)

Transaction Boundaries

  • Single account accrual (applyToBalance=true): Account UPDATE + InterestAccruals INSERT are wrapped in a single JDBC transaction with explicit commit/rollback.
  • Batch accrual: All account UPDATEs and InterestAccruals INSERTs for the entire batch are wrapped in a single JDBC transaction — the entire batch succeeds or entirely rolls back (all-or-nothing semantics).
  • Calculation without application (applyToBalance=false): Read-only; no transaction is opened.

Business Constraints

  • Only accounts with Status = 'Active' are included in batch accrual.
  • Interest accrual is idempotent in the sense that calling the API multiple times will accrue interest multiple times — there is no deduplication or idempotency key mechanism.
  • There is no authorization check; any caller with network access can trigger accrual operations.

Audit Trail

Every applied accrual (single or batch) records an InterestAccruals entry capturing: account ID, interest type, principal amount, annual rate, accrual days, interest amount, and creation timestamp. This provides a complete audit history of all balance changes due to interest.

The org.json:json:20140107 library is over a decade old and predates many fixes and improvements in the JSON ecosystem. While functional, this version may carry unfixed bugs and lacks features present in current releases. The mssql-jdbc 12.6.3.jre8 driver is recent and well-maintained but targets JRE 8 — upgrading the Java runtime to JDK 11 or later would allow moving to the jre11 classifier.

Caching Strategy

No caching layer is implemented. Every request opens a fresh JDBC connection, executes SQL queries against SQL Server, and closes the connection. There is no in-process cache (EhCache, Caffeine), distributed cache (Redis, Memcached), or second-level ORM cache (Hibernate L2). Each call to /api/interest/calculate re-reads the account balance from the database, and each call to /api/interest/accrue-all performs a full table scan of active accounts.

Data Ownership Boundaries

ZavaInterestCalculator uses a single shared SQL Server database (ZavaBankDB). There is only one deployable service, so there is no inter-service data sharing concern. The Accounts and AccountTypes tables are treated as pre-existing (not created by this application), while InterestAccruals is owned and created by this service.

All data access is co-located in a single WAR with no CQRS separation; every servlet performs both reads and writes against the same database instance.

Notable concern: Database credentials (db.user, db.password) are stored in plaintext in interest.properties bundled inside the WAR file. In the default configuration this includes the SQL Server sa account password. No secrets management (Vault, Azure Key Vault, Kubernetes Secrets) is configured; the application relies on environment variable overrides to avoid embedding credentials, but the fallback file contains real credentials.
