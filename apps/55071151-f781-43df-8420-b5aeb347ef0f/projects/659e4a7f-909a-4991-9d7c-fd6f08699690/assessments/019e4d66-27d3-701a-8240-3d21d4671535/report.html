Data Storage & External Services

The application relies exclusively on a single SQL Server database (ZavaBankDB) accessed via ADO.NET. It queries existing tables (Accounts, Customers, Transactions, TransactionTypes) and writes to application-managed tables (StatementRequests, GeneratedStatements, StatementArchive). The GeneratedStatements table is provisioned at runtime if absent. Connection string configuration is resolved from environment variables (with DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD) or falls back to the web.config connection string entry. No external APIs, caching layers, or message brokers are used.

Key Architectural Decisions

  • Direct ADO.NET data access: The service bypasses ORM frameworks and uses raw SqlConnection/SqlCommand calls for all database interactions, giving fine-grained control at the cost of portability and maintainability.
  • WCF BasicHttpBinding over SOAP: The service exposes operations via SOAP/BasicHttpBinding, making it compatible with legacy enterprise clients while limiting modern REST/JSON consumption.
  • Mono-based container deployment: The service uses Mono 6.12 and XSP4 to run a .NET Framework 4.8 WCF application on Linux, enabling Docker-based deployment without a Windows runtime.

Core Business Workflows

ZavaStatementService is a bank statement generation service that creates, stores, and retrieves monthly HTML statements for customer accounts, serving as a document-of-record system for Zava Bank's account history.

ZavaStatementService is a single-service deployment. It does not communicate with other microservices over REST or messaging — all data access is via direct SQL against a shared database instance. Primary Workflows

Workflow 1: Generate Monthly Statement

A client requests a monthly statement for a specific account and period. The service validates the request, assembles transaction data from the banking core, renders an HTML document, and persists the statement alongside an audit request and archive record.

Steps:

  1. Client invokes GenerateStatement(accountId, month, year) over SOAP.
  2. Validation — ValidatePeriod checks that accountId > 0, month is 1–12, and year is in range 2000–2100. Throws FaultException on violation.
  3. Account lookup — Queries Accounts joined with Customers to retrieve AccountNumber, Balance (used as opening balance), and customer full name. Throws FaultException("Account was not found.") if no row is returned.
  4. Transaction fetch — Queries Transactions joined with TransactionTypes for the account within the requested period, ordered chronologically. Accumulates totalCredits and totalDebits in-process: credits are transactions with TypeCode = 'DEP' or Amount >= 0; all others are treated as debits.
  5. HTML statement rendering — BuildHtmlStatement produces a self-contained HTML document including customer header, period, opening balance, credit/debit totals, and transaction table. Customer name and account number are XML-escaped before insertion.
  6. Checksum computation — ComputeChecksum computes a SHA-256 hash of the HTML content (hex-encoded) for archive integrity verification.
  7. StatementRequest insert — Inserts a row into StatementRequests recording the account, period, format (HTML), and status (Completed). Returns the generated RequestID.
  8. GeneratedStatement insert — Inserts the HTML content into GeneratedStatements with the RequestID. Returns the generated StatementID.
  9. StatementArchive insert — Inserts an archive record with a logical file path (db://generated-statements/<StatementID>), file size (HTML byte length), checksum, and a 7-year expiry.
  10. Returns GeneratedStatementResponse with StatementId, account details, period, and the full HtmlContent.

Workflow 2: Retrieve Statement History

A client requests a list of all previously generated statements for an account, ordered from most recent to oldest.

Steps:

  1. Client invokes GetStatementHistory(accountId) over SOAP.
  2. Validation — Checks that accountId > 0. Throws FaultException if invalid.
  3. History query — Queries GeneratedStatements for the account, returning up to 100 records ordered by GeneratedDate DESC.
  4. Returns StatementHistoryResponse containing the account ID and a list of StatementHistoryItem records (statement ID, account ID, month, year, generated date). HTML content is not included.

Workflow 3: Retrieve Statement Detail

A client fetches the full content of a specific previously generated statement by its ID.

Steps:

  1. Client invokes GetStatement(statementId) over SOAP.
  2. Validation — Checks that statementId > 0. Throws FaultException if invalid.
  3. Statement lookup — Queries GeneratedStatements by StatementID. Throws FaultException("Statement was not found.") if no row is returned.
  4. Returns StatementDetailResponse with full statement metadata and HtmlContent.

Cross-Service Data Flows

ZavaStatementService is a single-service application with no inter-service communication. All data originates from one SQL Server database (ZavaBankDB). The service reads from Banking Core tables (Accounts, Customers, Transactions, TransactionTypes) and writes to its own tables in the same database instance.

There are no gateway aggregation patterns, circuit breakers, event-driven flows, or fallback degradation modes between services. Failure of any SQL operation results in an unhandled exception or SOAP Fault propagated directly to the client.

Business Rules

  • Period boundary calculation: Statement period is computed as [first day of month, last tick of last day of month] using DateTime.AddMonths(1).AddTicks(-1) to ensure all transactions within the calendar month are captured.
  • Credit/debit classification: A transaction is classified as a credit if its TypeCode equals DEP (case-insensitive) or its Amount >= 0; otherwise it is a debit. Total debits are stored as the absolute value of negative amounts.
  • Archive retention: All statement archives are assigned a 7-year expiry (DATEADD(YEAR, 7, GETDATE())), consistent with typical financial document retention regulations.
  • Statement history cap: Only the 100 most recent statements per account are returned (SELECT TOP 100), regardless of total history volume.
  • Output-encoded customer data: Customer name and account number are XML-escaped (SecurityElement.Escape) before insertion into the HTML statement body, providing basic XSS protection in the rendered document.

Transaction Management

No explicit transaction management is implemented. The GenerateStatement workflow performs four sequential INSERT operations (StatementRequests, GeneratedStatements, StatementArchive) without wrapping them in a SQL transaction. If any INSERT fails after a prior one succeeds, the database is left in a partially written state (e.g., a StatementRequest record exists without a corresponding GeneratedStatement).

Error Handling

All business exceptions are surfaced as System.ServiceModel.FaultException (generic, without typed fault contracts). No compensating actions, retry logic, dead-letter handling, or structured error codes are implemented. Internal exception detail is exposed to clients (includeExceptionDetailInFaults="true"), which is a security risk in production.

Audit and Observability

The StatementRequests table serves as an implicit audit log, recording who requested what statement for which period and when it was completed. No structured application logging (e.g., to a log aggregator) or distributed tracing is present.

Authorization

No authentication or authorization is enforced at the business logic layer. Any SOAP client with network access can generate, list, or retrieve statements for any accountId without proving identity or ownership.

Version & Compatibility Risks

All dependencies are .NET Framework 4.8 BCL assemblies, which are in maintenance-only support. The most significant risk is System.ServiceModel (WCF): WCF was not ported to .NET 5/6/7/8/10 and requires migration to CoreWCF to run on a modern runtime. Similarly, System.Web (used by the HealthHandler) is not available on .NET Core/5+, requiring replacement with ASP.NET Core middleware or minimal APIs. The System.Data.SqlClient namespace is considered legacy; the community-maintained Microsoft.Data.SqlClient NuGet package is the recommended replacement for new and migrated apps.

Notable Observations

  • Zero third-party NuGet packages: The project has no external package dependencies (packages.config is empty). This simplifies dependency scanning and CVE auditing but means all functionality — SOAP hosting, SQL access, serialization — relies entirely on the .NET Framework 4.8 platform.
  • WCF is a hard migration blocker: System.ServiceModel (WCF) is the central framework of this service. It has no equivalent in the cross-platform .NET runtime without adopting CoreWCF, which provides partial WCF compatibility. Clients consuming the SOAP endpoint would also need to be evaluated.
  • Mono-based container runtime: The Dockerfile uses mono:6.12 and XSP4 to host the .NET Framework 4.8 application on Linux. Mono is no longer actively developed by Microsoft, making this a technical debt and security risk for production deployments.
  • No logging, observability, or security libraries: The application has no structured logging (e.g., Serilog, NLog), no distributed tracing (e.g., OpenTelemetry), and no authentication/authorization framework, representing gaps for production cloud readiness.

Caching Strategy

No caching layer is implemented. Every request results in direct SQL Server queries. Frequently read reference data (e.g., TransactionTypes, account metadata) is re-fetched on each invocation. There are no distributed cache (Redis), in-process cache (MemoryCache), or second-level cache configurations present.

Data Ownership Boundaries

ZavaStatementService operates against a shared SQL Server instance (ZavaBankDB) that it does not fully own. The service has read-only access to core banking tables (Accounts, Customers, Transactions, TransactionTypes) that are presumed to be owned and maintained by an upstream banking system. The service writes exclusively to its own statement lifecycle tables (StatementRequests, GeneratedStatements, StatementArchive).

There is no logical or physical schema isolation between the core banking tables and the statement service tables. Direct SQL coupling to shared tables creates a tight data dependency that complicates independent schema evolution and deployment.

No cross-service API calls, CQRS separation, or event-driven data sync patterns are present. The service reads upstream data synchronously at request time without any local caching or read-model projection.
