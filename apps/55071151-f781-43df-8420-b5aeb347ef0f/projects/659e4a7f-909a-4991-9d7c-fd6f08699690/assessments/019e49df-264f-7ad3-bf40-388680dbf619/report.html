Data Storage & External Services

The service uses a single SQL Server database (ZavaBankDb) for account reads and statement persistence (StatementRequests, GeneratedStatements, StatementArchive). No message broker, cache service, or third-party API dependency is declared inside this repository. Database credentials can be sourced from environment variables or web.config connection strings.

Key Architectural Decisions

  • Uses direct SQL statements in the service layer (ADO.NET) instead of repository/ORM abstraction.
  • Generates and stores statement content as HTML in the database (GeneratedStatements.StatementHtml).
  • Creates GeneratedStatements table at runtime if missing, enabling bootstrap in partially initialized environments.

No asynchronous messaging (Kafka/RabbitMQ/Azure Service Bus), no service discovery mechanism, and no gateway composition pattern are present in this repository. Resilience mechanisms such as retries/circuit breakers/timeouts are not explicitly configured in code. API-level security posture is minimal in this repo: no explicit authentication/authorization policy is configured for service operations, and transport-level HTTPS/TLS requirements are not enforced in the repository configuration.

Core Business Workflows

This service supports the banking statement lifecycle by generating monthly account statements and allowing downstream consumers to retrieve current and historical statement artifacts. The core workflows revolve around request validation, transaction summarization, statement persistence, and retrieval.

Primary Workflows

Workflow 1: Generate Monthly Statement

Entry point: SOAP operation GenerateStatement(accountId, month, year).

  1. Service validates input period and account identifier (ValidatePeriod).
  2. Service loads account/customer context and period transactions from SQL.
  3. Business logic computes credit/debit totals and transforms transactions into statement rows.
  4. Service renders statement HTML and computes SHA-256 checksum.
  5. Service persists request, generated statement, and archive rows.
  6. Service returns GeneratedStatementResponse with generated artifact metadata/content.

Workflow 2: Retrieve Statement History and Statement Detail

Entry points: SOAP operations GetStatementHistory(accountId) and GetStatement(statementId).

  1. Service validates requested identifier (accountId or statementId).
  2. Service queries generated statement persistence tables.
  3. Service maps database rows into response contracts.
  4. If a statement identifier is not found, service returns a business fault (FaultException).

Cross-Service Data Flows

No API gateway aggregation flow is present in this repository. Data composition occurs within a single service using shared SQL data: account/customer and transaction records are joined and transformed into statement outputs in-process. There is no circuit-breaker fallback logic across remote services; error behavior is fault-based when required data is missing or validation fails.

Business Rules & Decision Logic

  • Validation rules: accountId > 0; month must be 1-12; year constrained to 2000-2100; missing account/statement yields FaultException.
  • Decision logic: Credit/debit aggregation uses transaction type (DEP) and amount sign to separate credits from debits.
  • State transitions: Request lifecycle moves from generation request to completed persisted statement and archived checksum record in one transaction-like flow.
  • Business constraints: Statement is generated for a single account and calendar month; retrieval history is limited to most recent 100 generated statements.
  • Computed values: Total credits, total debits, and SHA-256 checksum are derived at generation time.
  • Cross-cutting concerns: Uses deterministic fault handling for invalid/missing data; no explicit authorization policy is defined in service code/config.

Caching Strategy

No explicit caching layer is implemented. Reads are executed directly against SQL Server for each request, and generated statement content is persisted for retrieval rather than cached in an in-memory/distributed cache.

Data Ownership Boundaries

This repository represents a single service boundary with a shared relational store pattern. It both consumes existing core banking tables and writes statement-specific records in the same database context. Cross-service access via APIs is not visible in this codebase; data exchange appears to occur through shared tables rather than inter-service contracts.
