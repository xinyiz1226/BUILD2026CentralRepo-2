Data Storage & External Services

The worker depends on RabbitMQ as its inbound message broker and SQL Server as its persistent audit store. Messages are read from the audit.events queue, optionally dead-lettered through zava.dlx, and then inserted into either a legacy or modern AuditLog schema in the database.

Key Architectural Decisions

  • Uses a single-process polling worker instead of a hosted service or push-based consumer.
  • Supports both JSON and XML message payloads by normalizing them into a common AuditEvent shape.
  • Preserves compatibility with two database schemas by checking for legacy AuditLog.ActionID columns before writing.

DTOs & Contracts

The primary contract is the internal AuditEvent class, which normalizes queue payloads into a common shape containing event type, service name, user identifier, timestamp, raw details, and correlation identifier. The worker accepts either JSON objects or XML documents carrying equivalent semantics, and it serializes the original payload into the database without exposing a separate HTTP DTO layer or OpenAPI specification.

Communication Patterns

Communication is asynchronous on ingress and synchronous on persistence. The worker polls RabbitMQ with BasicGet, parses the message body, writes the normalized event to SQL Server with parameterized ADO.NET commands, and then either acknowledges the delivery on success or issues BasicNack without requeue on failure. No retry, circuit breaker, service discovery, TLS configuration, or API-level authentication/authorization logic is present in the codebase; connectivity is configured through host, port, virtual host, and credentials supplied by configuration.

Core Business Workflows

The application’s business role is to preserve an audit trail for events emitted by other banking services. Its core workflow is straightforward: accept an audit message, normalize it, and store it durably so compliance and traceability data are retained.

Primary Workflows

Workflow 1: Consume and persist an audit event

  1. The worker wakes on its polling interval and attempts to retrieve a message from the configured RabbitMQ queue.
  2. If a message is present, the payload is parsed as JSON or XML and mapped into an AuditEvent.
  3. The worker opens a SQL connection and checks whether the database exposes the legacy AuditLog.ActionID column.
  4. For legacy schemas, the worker ensures an AuditActions row exists for the event type and then writes a legacy audit record.
  5. For modern schemas, the worker inserts a direct AuditLog row using the normalized event fields.
  6. On success the message is acknowledged; on failure the worker logs the error and dead-letters the message by negative acknowledgement without requeue.

Workflow 2: Graceful shutdown

  1. The process listens for Ctrl+C or process exit events.
  2. When shutdown is requested, the _keepRunning flag is cleared.
  3. The polling loop exits after the current cycle, allowing the worker to stop cleanly.

Cross-Service Data Flows

This repository participates in one cross-service data flow: upstream services publish audit events to RabbitMQ, and ZavaAuditWorker persists those events into the shared audit database. The worker does not call back into other services or aggregate responses; its only fallback behavior is business-relevant message rejection when parsing or persistence fails, ensuring bad messages are not silently acknowledged.

Business Rules & Decision Logic

  • Audit payloads must not be empty; empty bodies are rejected before any database work is attempted.
  • Message format selection is content-based: XML payloads are identified by a leading <, otherwise JSON parsing is attempted.
  • Schema compatibility is a key decision rule: the worker writes to a legacy audit model when AuditLog.ActionID exists, otherwise it writes to the newer column-based schema.
  • Legacy audit writes require a matching AuditActions row; if none exists, one is created automatically with a default Info severity.
  • Error handling is fail-safe for compliance: failed messages are not acknowledged and are instead routed toward dead-letter processing.

Caching Strategy

No application-level caching layer is present. Each processed message results in live database metadata lookup and insert operations, with no in-memory cache, distributed cache, or query result reuse.

Data Ownership Boundaries

The repository models a single service writing into a shared SQL Server database. It appears to own the audit trail write path, but the target tables may also be consumed by other services or reporting tools outside this repository. There is no cross-service data access logic in code; the worker receives all business context from the message payload and performs inserts directly.
