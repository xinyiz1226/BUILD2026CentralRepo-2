Data Storage & External Services

The application uses a single Microsoft SQL Server database (ZavaBankDB) accessed via JDBC. Two tables are created at startup by LoanBootstrapServlet: LoanApplications (stores application records with status, amounts, and decision metadata) and LoanDecisions (stores individual decision audit records). Three downstream microservices are called over HTTP during loan orchestration: the KYC Service (identity verification), the Risk Engine (credit scoring and approval decision), and the Ledger Service (loan account creation).

Key Architectural Decisions

  • Flat servlet architecture: No dependency-injection framework is used; configuration is resolved at runtime via environment variables with properties-file fallback, keeping the deployment footprint minimal.
  • Inline JDBC data access: SQL is written directly in servlet methods using PreparedStatement; there is no ORM or repository abstraction layer.
  • Synchronous HTTP orchestration: External microservice calls (KYC → Risk → Ledger) are executed sequentially in LoanApplyServlet.orchestrateDecision() using java.net.HttpURLConnection, making the apply endpoint a synchronous pipeline.

DTOs & Contracts

The API uses no typed DTO classes. All request and response bodies are processed as raw org.json.JSONObject instances parsed directly from the HTTP input stream in each servlet's doPost / doGet method. There is no schema validation, no OpenAPI / Swagger specification, and no protobuf or GraphQL contract.

One internal, private static inner class OrchestrationResult exists inside LoanApplyServlet to carry intermediate orchestration state between private methods (KYC status, risk score/level, approved amount, interest rate, ledger account ID, decision reason). It is never serialized as a standalone DTO and is invisible to callers.

All JSON serialization is handled inline using JSONObject.put() and JSONObject.optString() / optInt() / optDouble() methods — no Jackson or Gson configuration is involved.

For full field details of the LoanApplications and LoanDecisions tables that back these responses, refer to data-architecture.md.

Communication Patterns

Synchronous HTTP orchestration: LoanApplyServlet calls downstream microservices sequentially using java.net.HttpURLConnection. The call order is fixed: KYC → Risk Engine → Ledger. Each call is made only if the previous step passes; a KYC failure short-circuits the pipeline before the Risk Engine is called.

Timeouts: Each outbound HTTP call has a connect timeout of 7,000 ms and a read timeout of 7,000 ms. No global timeout spans the full orchestration pipeline.

Resilience: No circuit breaker, retry policy, bulkhead, or fallback mechanism is implemented. If any downstream service returns a non-2xx response or throws an exception, callJsonApi() silently returns an empty JSONObject. The application continues with default values (e.g., risk score of 0) rather than surfacing the failure to the caller, which can produce misleading loan decisions.

Service discovery: Service base URLs are resolved at runtime from environment variables (KYC_SERVICE_BASE_URL, RISK_SERVICE_BASE_URL, LEDGER_SERVICE_BASE_URL) or from loan-origination.properties. No service registry (Eureka, Consul, Kubernetes DNS alias) is used — URLs are hardcoded in configuration.

Startup dependency chain: LoanBootstrapServlet is loaded at startup (load-on-startup order 1) and creates database tables if they do not exist. If the SQL Server is unavailable at startup, the servlet throws a ServletException, preventing the application from starting. Downstream microservices are not validated at startup; failures occur only at request time.

Security posture: No authentication, no authorization, and no TLS configuration are present anywhere in the application. All four endpoints (/health, /api/loans/apply, /api/loans/{id}/status, /internal/bootstrap) are publicly accessible with no credential checks. Outbound calls to KYC, Risk Engine, and Ledger services are made over plain HTTP with no token or mutual TLS.

Primary Workflows

Workflow 1: Loan Application Submission

Entry point: POST /api/loans/apply — a client submits a loan request for a specific customer and loan product.

Steps:

  1. Input validation — the API validates that customerId, loanProductId, requestedAmount, and termMonths are all non-zero positive values. If any are missing or invalid, a 400 Bad Request is returned immediately. fullName defaults to "Customer {customerId}" if not provided.
  2. Application record creation — a new LoanApplications row is inserted with Status='Submitted' and the current timestamp.
  3. KYC verification — the customer's identity is verified with the KYC Service using customerId, fullName, and idNumber. If the KYC status is not PASS, the application status is set to KYC_REVIEW, a manual-review decision is recorded, and processing stops.
  4. Risk scoring — if KYC passes, the Risk Engine evaluates the application using applicationId, customerId, requestedAmount, and termMonths. The engine returns a risk score, risk level (Low/Medium/High), approved amount, and interest rate.
  5. Approval decision — if the risk engine declines (score < 620, or explicit approved=false), the application is marked Declined and a decline decision is recorded. Processing stops.
  6. Ledger account creation — for approved applications, a loan account is created in the Ledger Service for the customer. If creation succeeds, the application status becomes Approved. If the ledger call fails or returns no account ID, the status is set to APPROVED_PENDING_LEDGER.
  7. Application update and decision audit — the application record is updated with the final status, approved amount, interest rate, decision date, and notes. A LoanDecisions record is inserted. Both writes are committed in a single database transaction.

Workflow 2: Loan Status Retrieval

Entry point: GET /api/loans/{id}/status — a client queries the current state of a loan application.

Steps:

  1. The path parameter {id} is validated to be a numeric value.
  2. A single query fetches the full LoanApplications row for the given ApplicationID.
  3. If no record is found, a 404 Not Found is returned.
  4. The current application state (status, amounts, interest rate, term, decision date, notes, and ledger account ID) is returned as JSON.

Workflow 3: Database Bootstrap

Entry point: Servlet container startup — LoanBootstrapServlet executes with load-on-startup=1 before the application accepts traffic.

Steps:

  1. A JDBC connection is opened to ZavaBankDB.
  2. LoanApplications and LoanDecisions tables are created if they do not already exist using IF OBJECT_ID(...) IS NULL CREATE TABLE ... DDL.
  3. If the database is unreachable, a ServletException is thrown, preventing the application from starting.

Cross-Service Data Flows

Loan application orchestration is a purely synchronous, sequential pipeline within a single HTTP request. No gateway aggregation or response composition is involved; the LoanApplyServlet itself acts as the orchestrator.

The data flow for an approved loan:

  • customerId, fullName, idNumber flow from the inbound request → KYC Service → result (kycStatus) stored in the orchestration result only.
  • applicationId, customerId, requestedAmount, termMonths flow to the Risk Engine → result (riskScore, riskLevel, approvedAmount, interestRate) are denormalised into LoanDecisions at commit time.
  • customerId, loanAccountTypeId flow to the Ledger Service → the returned accountId is persisted in LoanApplications.LoanAccountID.

Fallback behaviour: If any downstream service is unreachable or returns an error, callJsonApi() silently returns an empty JSONObject. The caller then uses default/zero values, which can produce a misleading business outcome — for example, a KYC or Risk service timeout results in a default kycStatus="REVIEW" or riskScore=0, causing an implicit KYC_REVIEW or Declined decision without any error surfaced to the caller.

Business Rules & Decision Logic

Validation Rules

  • customerId must be a positive integer (> 0); otherwise request rejected with 400.
  • loanProductId must be a positive integer (> 0); otherwise request rejected with 400.
  • requestedAmount must be greater than zero; otherwise request rejected with 400.
  • termMonths must be a positive integer (> 0); otherwise request rejected with 400.
  • fullName is optional; if blank or absent, it defaults to "Customer {customerId}" for the KYC call.
  • idNumber is optional; passed through to KYC as supplied.
  • Loan application ID path parameter in the status endpoint must be a valid non-negative long integer.

Decision Logic — KYC Gate

  • KYC status must equal "PASS" (case-insensitive) to proceed.
  • Any other value ("REVIEW", "FAIL", or empty) results in Status=KYC_REVIEW.
  • If the KYC Service is unreachable, callJsonApi() returns an empty object; optString("status", "REVIEW") defaults to "REVIEW", implicitly triggering KYC_REVIEW — no error is surfaced.

Decision Logic — Risk Approval

  • If the Risk Engine response contains an explicit approved field, that boolean value governs the decision.
  • If approved is absent, approval is determined by riskScore >= 620.
  • Risk level classification: score ≥ 700 → Low; 620–699 → Medium; < 620 → High.
  • If the Risk Engine is unreachable, score defaults to 0, producing an implicit Declined decision.
  • Default interest rate of 0.0725 (7.25%) is applied when the Risk Engine does not return an interest rate.

State Transitions


Submitted → KYC_REVIEW          (KYC not PASS)
Submitted → Declined             (Risk engine declines)
Submitted → Approved             (KYC pass + Risk approve + Ledger account created)
Submitted → APPROVED_PENDING_LEDGER  (KYC pass + Risk approve + Ledger fails)

Transaction Boundaries

The database writes at the end of LoanApplyServlet (insert application, update application, insert decision) are wrapped in a manual JDBC transaction (setAutoCommit(false) / commit() / rollback()). If any write fails, all three are rolled back. The initial insertApplication write is performed before calling the external services and is part of the same transaction; however, if rollback occurs, the application ID is discarded.

Audit

Every completed loan origination flow produces a LoanDecisions record with DecisionBy='LoanOriginationAPI', capturing the decision type (Approve, Decline, ManualReview), credit score at decision time, risk level, and conditions (including KYC status and term months concatenated into the DecisionNotes field of LoanApplications).

Authorization

No authentication or authorization is enforced at any level. All endpoints are publicly accessible with no role checks, token validation, or access control. Any caller can submit a loan application for any customer ID or query the status of any application ID.

Version & Compatibility Risks

The org.json library is pinned to version 20140107, released in early 2014 — over a decade old. While the API is stable, this version predates many security-related fixes and improvements introduced in later releases. The javax.servlet-api 4.0.1 dependency ties the application to the older javax.* namespace (Java EE 8 / Tomcat 9); migration to a Jakarta EE 9+ container would require renaming all javax.servlet imports to jakarta.servlet. The mssql-jdbc 12.6.3 driver is a recent release and carries no immediate compatibility concerns for Java 11 targets.

Notable Observations

  • No logging framework declared: The application uses no SLF4J, Log4j 2, or Logback dependency; all errors are swallowed silently (catch (Exception ignored)), making production diagnosis difficult and representing a significant observability gap.
  • Severely outdated JSON library: org.json:json:20140107 is pinned to a 2014 build. Upgrading to a current release (e.g., 20240303) or switching to the widely-used jackson-databind would improve safety and maintainability.
  • No caching, messaging, or security libraries: The dependency footprint is intentionally minimal; all cross-cutting concerns (auth, rate limiting, retries) are expected to be handled externally (e.g., by an API gateway or service mesh).
  • javax.* namespace lock-in: Use of javax.servlet-api 4.0.1 (Java EE 8) prevents deployment on Jakarta EE 10 containers (Tomcat 10+, WildFly 27+) without a namespace migration.

Data Architecture & Persistence Layer

ZavaLoanOriginationAPI persists data in a single Microsoft SQL Server database (ZavaBankDB) using two tables managed entirely through plain JDBC — there is no ORM framework in use.

Key Repository Methods

There is no repository interface or data access object (DAO) layer. SQL is written inline using PreparedStatement directly inside the servlet classes. The following table documents the key data operations:

All operations use PreparedStatement with bound parameters. LoanApplyServlet wraps its insert + update + decision insert in a manual transaction (setAutoCommit(false) / commit() / rollback()). The status query is read-only with no explicit transaction.

Caching Strategy

No caching layer is implemented. There is no Redis, EhCache, Caffeine, Spring Cache, or in-memory caching of any kind. Every request to /api/loans/{id}/status executes a live SQL query against the database, and every loan application triggers multiple synchronous database writes without any write buffering or outbox pattern.

Data Ownership Boundaries

Shared vs isolated stores: The application owns a single ZavaBankDB SQL Server database containing both LoanApplications and LoanDecisions. The three external microservices (KYC, Risk Engine, Ledger) maintain their own data stores, which are opaque to this service; data is exchanged exclusively via HTTP REST calls and is not shared at the database level.

Cross-service data access: Customer identity data (full name, ID number) is passed through from the inbound API request directly to the KYC Service and is not persisted locally. The Ledger account ID returned by the Ledger Service is stored in LoanApplications.LoanAccountID as a foreign reference but is never used to query the Ledger Service's internal data. Risk scores and levels are denormalised into LoanDecisions.CreditScoreAtTime and LoanDecisions.RiskLevel at decision time.

Read/write patterns: All writes are synchronous and transactional within a single database request. There is no read/write split, no CQRS pattern, and no event sourcing.

Customer full names and national ID numbers (fullName, idNumber) are received in the POST request body and forwarded to the KYC Service but are not persisted in the local database. The loan-origination.properties file contains a database password in plaintext (db.password=YourStrong!Passw0rd), which is a credentials-at-rest risk if the file is included in source control or container images. The JDBC connection URL uses encrypt=false, meaning data-in-transit between the application and SQL Server is unencrypted.
