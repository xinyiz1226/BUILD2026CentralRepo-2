Data Storage & External Services

The application persists all loan application and decision data in a single SQL Server instance (ZavaBankDB) via direct JDBC. Schema initialization (DDL for LoanApplications and LoanDecisions tables) is performed at startup by LoanBootstrapServlet. Three external microservices are consumed synchronously over HTTP: a KYC Service for identity verification, a Risk Engine for credit scoring, and a Ledger Service for account creation. Configuration is sourced from environment variables with fallback to loan-origination.properties.

Key Architectural Decisions

  • Servlet-first, no framework: The application uses raw javax.servlet.HttpServlet subclasses rather than a higher-level framework (Spring MVC, Jakarta RESTful Services), placing all request parsing, JSON handling, and response writing in servlet doGet/doPost methods.
  • Inline orchestration: The entire loan decision pipeline — KYC verification → risk scoring → ledger provisioning — is implemented as sequential HTTP calls within a single servlet method with transaction management via JDBC setAutoCommit(false).
  • Startup DDL bootstrap: Table creation is handled programmatically at servlet container startup rather than by a migration tool (Flyway/Liquibase), making schema evolution manual.

DTOs & Contracts

No formal DTO classes are defined. All request and response contracts are handled via inline JSON construction using org.json.JSONObject:

  • LoanApplicationRequest (implicit, POST /api/loans/apply): JSON body with fields customerId (int), loanProductId (int), requestedAmount (double), termMonths (int), purpose (string), fullName (string), idNumber (string), loanAccountTypeId (int). Parsed directly in LoanApplyServlet.doPost().
  • LoanApplicationResponse (implicit): JSON object with fields status, applicationId, kycStatus, riskScore, riskLevel, approvedAmount, ledgerAccountCreated, ledgerAccountId. Constructed inline.
  • LoanStatusResponse (implicit): JSON object mirroring LoanApplications table columns.
  • OrchestrationResult (internal): private static inner class in LoanApplyServlet used to pass orchestration state between methods — not serialized directly.

No OpenAPI/Swagger spec, protobuf schemas, or GraphQL schema files are present. No Jackson or custom serializer is configured — raw JSONObject.toString() is used.

Communication Patterns

Synchronous REST (outbound): LoanApplyServlet makes three sequential synchronous HTTP POST calls using java.net.HttpURLConnection with 7-second connect and read timeouts:

  1. POST {KYC_SERVICE_BASE_URL}/api/kyc/verify — KYC identity check
  2. POST {RISK_SERVICE_BASE_URL}/api/risk/score — credit risk scoring (only if KYC passes)
  3. POST {LEDGER_SERVICE_BASE_URL}/api/accounts/create — ledger account creation (only if risk approves)

No resilience patterns: There is no circuit breaker (Resilience4j, Hystrix), retry policy, or bulkhead pattern. If a downstream service is unreachable or returns a non-2xx response, the callJsonApi helper silently swallows the exception and returns an empty JSONObject. Upstream callers then interpret default/absent fields as failure signals.

Service discovery: Services are discovered via hardcoded hostnames configured in loan-origination.properties (e.g., zava-kyc-service, zava-risk-engine, zava-ledger). No Eureka, Consul, or Kubernetes DNS abstraction is used.

No API gateway: The application receives client requests directly. There is no gateway layer for routing, load balancing, or request aggregation.

No asynchronous messaging: Kafka, RabbitMQ, Azure Service Bus, or equivalent are not present. The entire loan origination flow is synchronous and blocking.

Security posture: No authentication, authorization, or TLS is configured at the API level. All three endpoints (/health, /api/loans/apply, /api/loans/{id}/status) are publicly accessible with no authorization checks. No Spring Security, JWT, OAuth2, or Basic Auth is present. Communication with downstream services also has no authentication headers.

Primary Workflows

Workflow 1: Loan Application Submission

A client submits a POST to /api/loans/apply. The application validates required fields, persists an initial application record, then orchestrates a sequential three-step decision pipeline:

  1. Input validation — checks customerId, loanProductId, requestedAmount, and termMonths are present and positive; returns 400 if not.
  2. Application persistence — inserts a LoanApplications record with Status = 'Submitted' and captures the generated applicationId. A database transaction is opened.
  3. KYC verification — POSTs customer identity data (customerId, fullName, idNumber) to the KYC Service. If kycStatus != 'PASS', the application is updated to Status = KYC_REVIEW and a ManualReview decision is recorded; processing stops.
  4. Risk scoring — POSTs applicationId, customerId, requestedAmount, and termMonths to the Risk Engine. Receives riskScore, riskLevel, and an approved flag. If not approved (score < 620 threshold), the application is updated to Status = Declined and a Decline decision is recorded; processing stops.
  5. Ledger account creation — POSTs to the Ledger Service to open a loan account. Receives accountId and created flag.
  6. Final persistence — updates LoanApplications with final status (Approved or APPROVED_PENDING_LEDGER), approved amount, interest rate, and ledger account ID; inserts an Approve decision record. Transaction is committed.
  7. Response — returns JSON with status, applicationId, kycStatus, riskScore, riskLevel, approvedAmount, ledgerAccountCreated, and ledgerAccountId.

Workflow 2: Loan Status Inquiry

A client sends GET /api/loans/{id}/status. The application:

  1. Parses and validates the {id} path segment as a non-negative long.
  2. Queries LoanApplications by ApplicationID.
  3. Returns a JSON response with all key fields (status, amounts, decision date, notes, ledger account ID), or a 404 if the application is not found.

Workflow 3: Application Bootstrap (Startup)

On application startup (servlet container initialization), LoanBootstrapServlet (load-on-startup=1):

  1. Opens a JDBC connection to SQL Server.
  2. Executes idempotent DDL to create LoanApplications and LoanDecisions tables if they do not already exist.
  3. Logs or throws a ServletException on failure, preventing the application from starting.

Cross-Service Data Flows

The loan origination flow involves sequential synchronous HTTP calls to three external services. Data passes forward through the pipeline:

  • KYC Service: Receives customerId, fullName, idNumber; returns kycStatus. The origination service does not persist fullName or idNumber.
  • Risk Engine: Receives applicationId, customerId, requestedAmount, termMonths; returns riskScore, riskLevel, approved, approvedAmount, interestRate. The origination service persists approvedAmount, interestRate, riskScore, and riskLevel in its own tables.
  • Ledger Service: Receives customerId, accountTypeId, openingBalance, description; returns accountId and created. The accountId is persisted as LoanAccountID in LoanApplications.

Fallback behavior: If any downstream service is unreachable or returns an error, the callJsonApi helper silently returns an empty JSONObject. The calling code then interprets missing fields using optString/optInt/optBoolean defaults:

  • KYC failure → kycStatus defaults to "REVIEW" → application is set to KYC_REVIEW (conservative — treated as manual review required)
  • Risk failure → approved defaults to false → application is declined
  • Ledger failure → created defaults to false, accountId defaults to 0 → application is set to APPROVED_PENDING_LEDGER

Business Rules & Decision Logic

Validation Rules:

  • customerId, loanProductId, termMonths must be positive integers (> 0)
  • requestedAmount must be greater than zero
  • fullName defaults to "Customer {customerId}" if not provided (no hard validation)
  • loanAccountTypeId defaults to 2 if not provided

Decision Logic:

  • KYC must return kycStatus == 'PASS' (case-insensitive) for the application to proceed beyond KYC review
  • Risk engine riskScore >= 620 is used as the implicit approval threshold when the engine does not return an explicit approved field; scores < 620 are declined
  • Risk levels: score >= 700 → Low; score >= 620 → Medium; score < 620 → High
  • Default interest rate is 7.25% (loan.default.interestRate), used when the risk engine does not return an explicit rate

State Transitions (LoanApplication.Status):

  • Submitted → (KYC fails) → KYC_REVIEW
  • Submitted → (KYC passes, Risk declines) → Declined
  • Submitted → (KYC passes, Risk approves, Ledger succeeds) → Approved
  • Submitted → (KYC passes, Risk approves, Ledger fails) → APPROVED_PENDING_LEDGER

Decision Types (LoanDecisions.DecisionType):

  • Approve — application is approved (status Approved or APPROVED_PENDING_LEDGER)
  • Decline — risk engine rejected the application
  • ManualReview — KYC did not pass, human review required

Transactions:

  • The entire apply flow (INSERT application + UPDATE application + INSERT decision) runs in a single JDBC transaction with autoCommit=false
  • On SQLException, the transaction is rolled back; a 500 response is returned
  • KYC/Risk/Ledger HTTP calls are outside the transaction boundary — they are made after the initial INSERT commits implicitly (the transaction covers the UPDATE and decision INSERT after the orchestration result is known)

Error Handling:

  • All downstream HTTP call failures are silently swallowed (exceptions caught and empty JSONObject returned)
  • All SQLException exceptions during reads result in a 500 response
  • No dead-letter queue, compensating transaction, or saga pattern is implemented for partially completed flows

Authorization:

  • No authorization checks — all endpoints are publicly accessible
  • The DecisionBy field in LoanDecisions is hardcoded to 'LoanOriginationAPI' (no user identity context)

Version & Compatibility Risks

javax.servlet-api 4.0.1 targets the Jakarta EE 8 namespace (javax.), which was superseded by Jakarta EE 9+ (renamed to jakarta. namespace). Migrating to a modern container (e.g., Tomcat 10+, WildFly 27+) requires a namespace migration. org.json 20140107 is over a decade old (released 2014) and carries known CVEs in older versions; upgrading to a recent release (≥ 20231013) is recommended. mssql-jdbc 12.6.3.jre11 is a recent release and is compatible with Azure SQL Database and Azure SQL Managed Instance.

Notable Observations

  • Extremely lean dependency footprint: Only 3 declared dependencies — no framework (Spring, Jakarta EE CDI), no ORM (Hibernate, JPA), no logging facade (SLF4J), and no security library. All functionality is implemented by hand using raw JDBC and the Servlet API.
  • Ancient JSON library: org.json 20140107 is from January 2014 and is not maintained through a standard release cadence. Modern alternatives (Jackson, Gson, JSON-B) offer better performance, type safety, and active CVE patching.
  • No logging framework: Neither SLF4J, Log4j2, nor Logback is declared. All error handling silently swallows exceptions (catch (Exception ignored)), making production observability and troubleshooting very difficult.
  • No test dependencies: No testing framework (JUnit, TestNG, Mockito) is declared. The project has no automated tests.

All database operations use PreparedStatement and manual resource management (no try-with-resources). Transaction management is performed manually via connection.setAutoCommit(false) / connection.commit() / connection.rollback().

Caching Strategy

No caching layer is present. Every request results in a direct JDBC call to SQL Server. No Redis, EhCache, Caffeine, or in-memory cache is configured. No @Cacheable or equivalent annotation is used.

Data Ownership Boundaries

Isolated data store: The application owns a single SQL Server database (ZavaBankDB) exclusively. There is no shared database with other services; external services (KYC, Risk Engine, Ledger) are called via HTTP REST and own their own data stores.

Cross-service data access: All cross-service data exchange is via synchronous REST (HTTP POST). The application does not access any external service's database directly. The only persistent identifier passed between services is customerId (passed to KYC and Ledger) and applicationId (passed to Risk Engine). The ledgerAccountId returned by the Ledger Service is stored in LoanApplications.LoanAccountID as a foreign reference.

Read/write patterns: Write-heavy for LoanApplyServlet (one INSERT + one UPDATE + one INSERT per request); read-only for LoanStatusServlet (one SELECT per request). No CQRS, event sourcing, or read replica patterns are used.
